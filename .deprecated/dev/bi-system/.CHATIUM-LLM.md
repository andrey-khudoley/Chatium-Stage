# Система бизнес-аналитики

Система бизнес-аналитики для управления событиями и отслеживания активности.

## Описание проекта

Система бизнес-аналитики предназначена для отслеживания и анализа событий, связанных с трафиком, поведением пользователей и внешними сервисами. 

Реализована полная интеграция с ClickHouse для мониторинга событий.

## Основные таблицы

### AnalyticsSettings (analytics_settings_a3f8b2c1)
Хранение настроек системы в формате key-value.

**Поля:**
- `key` (String) - Ключ настройки
- `value` (String) - Значение настройки
- `description` (String) - Описание настройки

**Примечание:** Активные мониторинги хранятся в `AnalyticsSettings` с ключом `active_monitoring`. При запуске мониторинга создаётся настройка с JSON-данными (userId, socketId, taskId, startedAt). При остановке настройка удаляется.

### ActiveJobs (active_jobs_b5c8d9e1) ✨
Таблица для хранения активных taskId джобов. Исключает race condition при одновременном добавлении/удалении джобов.

**Поля:**
- `taskId` (String) - Уникальный ID задачи (taskId)
- `jobType` (String, optional) - Тип джоба (например, monitor-events, load-cache-batch)
- `parentTaskId` (String, optional) - ID родительского джоба (для цепочек джобов)
- `metadata` (Any, optional) - Дополнительная информация

**Примечание:** Используется для отслеживания активных джобов и предотвращения конфликтов при параллельном выполнении.

### AnalyticsDatasets (analytics_datasets_d8a4e7f2) ✨
Хранение конфигурации датасетов - наборов событий с параметрами для работы с данными.

**Поля:**
- `name` (String) - Название датасета
- `description` (String) - Описание датасета
- `config` (String) - JSON конфигурация компонентов датасета
- `createdAt` (DateTime) - Дата создания (автоматически)
- `updatedAt` (DateTime) - Дата последнего обновления (автоматически)

### AnalyticsDashboards (analytics_dashboards_c9d3e4f5) ✨
Хранение конфигурации дашбордов - наборов визуальных компонентов поверх датасетов.

**Поля:**
- `name` (String) - Название дашборда
- `description` (String) - Описание дашборда
- `config` (String) - JSON конфигурация компонентов дашборда
- `createdAt` (DateTime) - Дата создания (автоматически)
- `updatedAt` (DateTime) - Дата последнего обновления (автоматически)

### AnalyticsDatasetCache (analytics_dataset_cache_f9e5d2a1) ✨
Кэш событий для компонентов датасетов. Хранит все события, относящиеся к компонентам датасетов, для быстрого доступа без постоянных запросов к ClickHouse.

**Поля:**
- `dataset_id` (RefLink) - Ссылка на датасет (обязательное)
- `component_id` (String) - ID компонента датасета (обязательное)
- Все поля событий из ClickHouse (ts, dt, url, urlPath, action, user_id, uid, и т.д.)

**Примечание:** Кэш автоматически обновляется при создании/редактировании датасета через асинхронный job `loadDatasetCacheJob`. При удалении компонента или датасета соответствующие записи кэша автоматически удаляются через job `deleteDatasetJob`.

**Система кэширования:**
- **Автоматическая загрузка**: при создании/обновлении датасета запускается job `loadDatasetCacheJob`, который загружает события из ClickHouse в кэш
- **WebSocket прогресс**: job отправляет прогресс загрузки через WebSocket (`dataset-cache-load-start`, `dataset-cache-load-progress`, `dataset-cache-load-complete`)
- **Пакетная обработка**: события загружаются пакетами для оптимизации производительности
- **Автоматическая очистка**: при удалении компонента или датасета кэш удаляется автоматически
- **Поле params**: все параметры из URL сохраняются в поле `params` для быстрого доступа к UTM-меткам и другим параметрам

**Структура config (упрощённо):**
- `timePeriod?: string` — опциональный общий период анализа
- `components: Array<{ id, title, viewType: 'counter'|'simple-table'|'pivot-table', datasetId, metric: 'UNIQ', columns?: string[], pivotConfig?: { attributions: string[] }, layout: { x, y, w, h } }>`

**Примечание:** Логика обработки данных для дашбордов удалена и будет переписана с нуля. Визуальные компоненты сохранены, но отображают заглушки вместо реальных данных.

## API Endpoints

### Settings API (`/api/settings`)

- **GET `/list`** - Получить список всех настроек
- **POST `/update`** - Создать или обновить настройку (по ключу)
- **POST `/delete`** - Удалить настройку
- **GET `/event-filter`** - Получить текущий фильтр событий
- **POST `/event-filter`** - Сохранить фильтр событий (массив типов событий)

### Events API (`/api/events`)

- **GET `/list`** - Получить события из ClickHouse
  - Query: `limit`, `offset`, `eventType`, `eventName`
  - **Автоматическая дедупликация** по URL + секунда
- **POST `/details`** - Получить детальную информацию о событии
  - Body: `urlPath`, `timestamp`
  - Возвращает: полные данные события со всеми параметрами
- **POST `/search`** - Поиск событий по всем полям ⭐
  - Body: `query` (строка поиска), `limit` (default: 25), `offset` (default: 0), `getTotalOnly` (optional)
  - **Двухэтапный поиск**:
    1. Приоритетный поиск по основным полям (user_email, user_id, uid, user_first_name, user_last_name, ip)
    2. Расширенный поиск по всем полям (если приоритетный не дал результатов)
  - **Автоматическая связка визитов**: если найдены события с user_id, добавляются анонимные визиты через цепочку user_id → form/sent → uid → HTTP события
  - **Период поиска**: 90 дней
  - **Дедупликация результатов**
  - **Поддержка пагинации**: offset и limit для больших результатов
  - **Опция getTotalOnly**: для получения только общего количества без событий
  - Возвращает: `{ success: boolean, events: [], total: number }`
- **POST `/start-monitoring`** - Запустить **общий** мониторинг событий (job каждые 15 сек)
  - **⚠️ ОБЩИЙ МОНИТОРИНГ**: В системе может быть только один активный мониторинг для всех пользователей
  - Возвращает: `socketId` для WebSocket подключения
  - **Автоматическая дедупликация** перед отправкой через WebSocket
  - **Хранение**: создаёт настройку `active_monitoring` в `AnalyticsSettings` с JSON-данными (userId, socketId, taskId, startedAt)
  - **Идемпотентность**: повторный вызов возвращает существующий общий socketId (любой админ может подключиться к активному мониторингу)
  - **Множественные подключения**: несколько пользователей могут подключаться к одному WebSocket и получать одни и те же события
- **POST `/stop-monitoring`** - Остановить **общий** мониторинг (отменяет job)
  - **⚠️ ВЛИЯЕТ НА ВСЕХ**: остановка отключает мониторинг для всех подключённых пользователей
  - Отменяет запланированную задачу через `cancelScheduledJob`
  - **Удаляет** настройку `active_monitoring` из базы (не помечает неактивной!)
  - Возвращает: `{ success: boolean, message: string }`
  - **Не требует владения**: любой админ может остановить общий мониторинг
- **GET `/monitoring-status`** - Проверить статус **общего** мониторинга
  - Проверяет наличие настройки `active_monitoring`
  - Возвращает: `{ success: true, isActive: boolean, monitoring: {...} | null }`
  - **Видят все**: все админы видят одинаковый статус мониторинга

## Архитектура общего мониторинга событий

### 🔄 Как работает общий мониторинг

Мониторинг событий в real-time - это **общий функционал для всех администраторов**. В системе может быть только один активный мониторинг одновременно.

**Принципы работы:**
- 📡 **Один WebSocket на всех** - все подключённые админы получают одни и те же события
- 👥 **Множественные подключения** - несколько админов могут одновременно слушать поток событий
- 🔓 **Общий доступ** - любой админ может запустить/остановить/подключиться к мониторингу
- 💾 **Одна настройка** - `active_monitoring` в `AnalyticsSettings` (не персональная для каждого пользователя)

**Сценарий использования:**

1. **Админ 1** нажимает "Запустить мониторинг" на странице `/events`
   - Создаётся настройка `active_monitoring` с общим `socketId`
   - Запускается `monitorEventsJob` (повторяется каждые 15 секунд)
   - Админ 1 получает `socketId` и подключается к WebSocket

2. **Админ 2** открывает страницу `/events` и нажимает "Запустить мониторинг"
   - API возвращает тот же `socketId` (с флагом `alreadyActive: true`)
   - Админ 2 подключается к **тому же WebSocket**
   - Оба админа получают одни и те же события в реальном времени

## Дедупликация HTTP-событий

### Проблема
При размещении ифреймов на странице счётчики могут срабатывать одновременно или с разницей в несколько секунд, создавая дублирующиеся события с одинаковым URL от одного пользователя.

### Решение
Реализована **последовательная дедупликация** событий: событие считается дубликатом, если оно имеет тот же `URL + UID` что и **предыдущее добавленное** событие с разницей во времени **≤ 5 секунд**.

**Расположение**: `lib/events/deduplication.ts`

### Алгоритм работы

```typescript
/**
 * Дедупликация событий - убирает последовательные дубликаты
 * Событие = дубликат если оно идет СРАЗУ после события с тем же URL+UID и разница <= 5 сек
 * @shared
 * Файл: lib/events/deduplication.ts
 */
export function deduplicateEvents(events: any[]): any[] {
  if (!events || events.length === 0) {
    return events
  }
  
  const result: any[] = []
  let lastAddedKey: string | null = null
  let lastAddedTs: number | null = null
  
  for (const event of events) {
    const urlPath = event.urlPath || ''
    const uid = event.uid || event.user_id || ''
    const key = `${urlPath}:${uid}`
    const eventTs = new Date(event.ts || '').getTime()
    
    let isDuplicate = false
    
    // Проверяем только с ПОСЛЕДНИМ ДОБАВЛЕННЫМ событием
    if (lastAddedKey && lastAddedTs) {
      if (key === lastAddedKey) {
        const diffSeconds = Math.abs(eventTs - lastAddedTs) / 1000
        
        // Если разница <= 5 секунд - это последовательный дубликат
        if (diffSeconds <= 5) {
          isDuplicate = true
        }
      }
    }
    
    if (!isDuplicate) {
      // Добавляем событие и обновляем "последнее добавленное"
      result.push(event)
      lastAddedKey = key
      lastAddedTs = eventTs
    }
    // Если дубликат - НЕ обновляем "последнее добавленное"
  }
  
  return result
}
```

## Фильтрация событий

### Описание

Система фильтрации позволяет пользователям выбирать, какие типы событий они хотят отслеживать. Фильтр применяется **повсеместно** - как к real-time мониторингу через WebSocket, так и к списку событий на странице.

### Как это работает

#### 1. Выбор событий в настройках

На странице настроек (`/dev/bi-system/settings`) пользователь может:
- Выбрать конкретные типы событий (HTTP-события, события GetCourse, категории)
- Сохранить фильтр в базу данных (настройка `events_filter`)
- Система автоматически перезапускает мониторинг, если он был активен

#### 2. Сохранение фильтра

```typescript
// Фильтр сохраняется в таблице AnalyticsSettings
{
  key: 'events_filter',
  value: JSON.stringify(['pageview', 'button_click', 'dealCreated']),
  description: 'Фильтр типов событий для мониторинга'
}
```

#### 3. Применение фильтра

Фильтр применяется в двух местах:

**A. Список событий** (`apiEventsListRoute`):
- При загрузке событий из ClickHouse
- Фильтр читается из настроек
- SQL условие строится через `buildEventFilterConditions()`
- События отфильтровываются на уровне базы данных

**B. Real-time мониторинг** (`monitorEventsJob`):
- Job выполняется каждые 15 секунд
- Фильтр читается из настроек
- SQL условие строится через ту же функцию `buildEventFilterConditions()`
- Отфильтрованные события отправляются через WebSocket

### Функция buildEventFilterConditions()

Централизованная функция для построения SQL условий фильтрации.

**Расположение**: `lib/events/filters.ts`

```typescript
/**
 * Построение SQL условий для фильтрации событий
 * @shared
 * Файл: lib/events/filters.ts
 */
export function buildEventFilterConditions(eventTypesFilter: string[]): string {
  // Если фильтр пустой - получаем все типы событий
  if (eventTypesFilter.length === 0) {
    eventTypesFilter = getAllEvents().map(e => e.name)
  }
  
  const allEventDefinitions = getAllEvents()
  const actionConditions: string[] = []
  
  for (const eventTypeName of eventTypesFilter) {
    const eventDef = allEventDefinitions.find(e => e.name === eventTypeName)
    
    if (!eventDef) continue
    
    if (eventDef.urlPattern) {
      // Для паттернов (например, event://getcourse/%)
      actionConditions.push(`urlPath LIKE '${eventDef.urlPattern}'`)
    } else if (eventDef.urlPath) {
      // Для точных путей событий GetCourse
      actionConditions.push(`urlPath = '${eventDef.urlPath}'`)
    } else if (eventDef.type === 'traffic') {
      // Для HTTP событий
      if (eventDef.name === 'pageview') {
        actionConditions.push(`startsWith(urlPath, 'http')`)
      } else {
        actionConditions.push(`action = '${eventDef.name}'`)
      }
    }
  }
  
  // Если нет условий - возвращаем условие, которое ничего не найдет
  if (actionConditions.length === 0) {
    return '1 = 0'
  }
  
  // Объединяем условия через OR
  return actionConditions.join(' OR ')
}
```

### Пример использования

**Шаг 1:** Пользователь выбирает только `pageview` в настройках

**Шаг 2:** Фильтр сохраняется:
```json

```

**Шаг 3:** SQL запрос в ClickHouse:
```sql

  AND dt >= today() - 7
ORDER BY ts DESC
LIMIT 100
```

**Шаг 4:** На странице событий отображаются **только** события pageview

### Преимущества подхода

1. **Единая логика**: Одна функция `buildEventFilterConditions()` используется везде
2. **Эффективность**: Фильтрация на уровне SQL, а не в памяти
3. **Гибкость**: Поддержка точных событий, паттернов и HTTP-событий
4. **Консистентность**: Список событий и мониторинг показывают одни и те же данные

### Алгоритм работы API атрибуции

API `/api/attribution` работает напрямую с ClickHouse, выполняя следующие шаги:

1. **user_id → uid**: Находит все `uid`, связанные с `user_id` через события `form/sent`
2. **uid → сессии**: Для каждого `uid` находит все уникальные сессии
3. **Сессии → первое HTTP-событие**: Для каждой сессии находит первое HTTP-событие с параметрами URL
4. **Парсинг параметров**: Извлекает параметры из URL первой/последней сессии

**Особенности:**
- Работает напрямую с ClickHouse без предварительной обработки
- Данные всегда актуальные (read-on-demand)
- Для `attribution='last'` пропускает сессии без параметров
- Для `attribution='first'` возвращает первую сессию, даже если нет параметров

**Подробная документация:** См. `016-analytics-attribution.md` для детального описания алгоритма и SQL-запросов.

### Функция парсинга параметров URL

**Расположение**: `lib/events/urlParser.ts`

```typescript
/**
 * Парсинг параметров из URL (между ? и #)
 * @shared
 * Файл: lib/events/urlParser.ts
 */
export function parseUrlParams(url: string): Record<string, string> {
  if (!url) return {}
  
  try {
    // Извлекаем часть между ? и #
    const queryStart = url.indexOf('?')
    if (queryStart === -1) return {}
    
    const hashStart = url.indexOf('#', queryStart)
    const queryString = hashStart === -1 
      ? url.substring(queryStart + 1)
      : url.substring(queryStart + 1, hashStart)
    
    const params: Record<string, string> = {}
    const pairs = queryString.split('&')
    
    for (const pair of pairs) {
      if (!pair) continue
      const equalIndex = pair.indexOf('=')
      if (equalIndex === -1) {
        // Параметр без значения (?debug)
        const key = decodeURIComponent(pair)
        if (key) params[key] = ''
      } else {
        // Параметр со значением (?utm_source=google)
        const key = decodeURIComponent(pair.substring(0, equalIndex))
        const value = decodeURIComponent(pair.substring(equalIndex + 1))
        if (key) params[key] = value || ''
      }
    }
    
    return params
  } catch (error) {
    return {}
  }
}
```

**Примеры работы:**

| Входной URL | Результат |
|-------------|-----------|
| `https://site.com/?p=test123&ref=google` | `{p: "test123", ref: "google"}` |
| `https://site.com/?p=test456#anchor` | `{p: "test456"}` |
| `https://site.com/page` | `{}` |

## Система поиска событий

### Описание

Функционал поиска позволяет находить события по любым доступным полям с автоматической связкой анонимных визитов пользователя.

### Архитектура поиска

#### 1. Двухэтапная стратегия

**Этап 1: Приоритетный поиск** (быстрый)
```sql
SELECT * FROM chatium_ai.access_log
WHERE (
  user_email LIKE '%query%'
  OR user_id LIKE '%query%'
  OR uid LIKE '%query%'
  OR user_first_name LIKE '%query%'
  OR user_last_name LIKE '%query%'
  OR ip = 'query'
)
AND dt >= today() - 90
ORDER BY ts DESC
LIMIT 50
```

**Этап 2: Расширенный поиск** (если приоритетный не дал результатов)
- Поиск по 30+ полям через функцию `buildSearchConditions()` (`lib/events/search.ts`)
- Включает все доступные поля: URL, параметры, UTM-метки, технические данные

#### 2. Автоматическая связка визитов

Если в результатах поиска найдены события с `user_id`, автоматически добавляются анонимные визиты:

**Цепочка связи:**
```
1. Извлекаем user_id из найденных событий
   └→ user_id = "219894:477261510"

2. Находим uid через события form/sent
   └→ SELECT uid FROM access_log 
      WHERE urlPath = 'event://getcourse/form/sent'
      AND user_id = "219894:477261510"
   └→ uid = "8DueRPvg2vGoKYAhKTtdMjZBwHM12krC"

3. Находим визиты по uid
   └→ SELECT * FROM access_log
      WHERE uid = "8DueRPvg2vGoKYAhKTtdMjZBwHM12krC"
      AND (startsWith(urlPath, 'http') OR action != '')
   └→ Все анонимные визиты пользователя до регистрации

4. Объединяем результаты
   └→ События + Визиты → сортировка по времени → дедупликация
```

### Приоритеты полей поиска

Поля разделены на 8 приоритетов для оптимальной производительности:

1. **Наиболее вероятные**: email, user_id, uid
2. **Идентификаторы**: IP, session_id, gc_visitor_id, gc_session_id
3. **URL и контент**: url, urlPath, title, referer
4. **Локация**: country, city, region
5. **Параметры действий**: action, action_param1/2/3
6. **UTM метки**: utm_source, utm_medium, utm_campaign, utm_term, utm_content
7. **Информация о пользователе**: first_name, last_name, phone, account_role
8. **Технические данные**: user_agent, os_name, client_name, device_type

### Особенности

- ✅ **Период поиска**: 90 дней
- ✅ **Лимит результатов**: 50 событий (настраиваемый)
- ✅ **Дедупликация**: автоматическое удаление дубликатов
- ✅ **Частичное совпадение**: LIKE поиск по большинству полей
- ✅ **Точное совпадение**: для IP-адресов
- ✅ **Автоматическая остановка мониторинга** при активации поиска

## Система датасетов ✨

### Описание

Модульная система конфигурации датасетов - наборов событий с параметрами для работы с данными из ClickHouse.

**Датасет** - это промежуточный слой между сырыми данными ClickHouse и визуализацией. Датасет определяет:
- Какие события анализируются
- За какой период
- Какие параметры извлекаются
- В будущем: SQL-запросы для получения данных

Поверх датасетов в будущем будут создаваться **дашборды** с визуализацией (графики, таблицы, метрики).

**Ключевые особенности:**
- ✅ **Автоматическое кэширование** - события загружаются в таблицу `AnalyticsDatasetCache` для быстрого доступа
- ✅ **WebSocket прогресс** - отслеживание прогресса загрузки и удаления кэша в реальном времени
- ✅ **Асинхронная обработка** - все операции с кэшем выполняются через background jobs
- ✅ **Поддержка UTM-фильтров** - фильтрация событий по UTM-параметрам в компонентах датасета
- ✅ **Сохранение параметров** - все параметры из URL сохраняются в поле `params` для быстрого доступа

### Архитектура

**Базовая логика:**
1. Heap-таблица `AnalyticsDatasets` хранит конфигурацию датасетов
2. Страница `/dev/bi-system/dataset-config` для создания и редактирования

- **Датасет** = набор компонентов с общим периодом анализа

### Система кэширования датасетов ✨

**Назначение:**
Кэш событий (`AnalyticsDatasetCache`) хранит все события, относящиеся к компонентам датасетов, для быстрого доступа без постоянных запросов к ClickHouse.

**Механизм работы:**

1. **Автоматическая загрузка кэша:**
   - При создании или обновлении датасета запускается job `loadDatasetCacheJob`
   - Job загружает события из ClickHouse для каждого компонента датасета
   - События сохраняются в таблицу `AnalyticsDatasetCache` с привязкой к `dataset_id` и `component_id`
   - Все параметры из URL сохраняются в поле `params` для быстрого доступа к UTM-меткам

2. **WebSocket прогресс загрузки:**
   - Job отправляет сообщения о прогрессе через WebSocket:
     - `dataset-cache-load-start` - начало загрузки (с `datasetId`, `componentId`, `total`)
     - `dataset-cache-load-progress` - прогресс загрузки (с `loaded`, `total`, `progress`)
     - `dataset-cache-load-complete` - завершение загрузки
     - `dataset-cache-load-error` - ошибка при загрузке

3. **Пакетная обработка:**
   - События загружаются пакетами для оптимизации производительности
   - Поддержка фильтров по UTM-параметрам в компонентах датасета
   - Автоматическая дедупликация событий при загрузке

4. **Автоматическая очистка:**
   - При удалении компонента удаляются все записи кэша для этого компонента
   - При удалении датасета удаляется весь кэш датасета через job `deleteDatasetJob`
   - Кэш удаляется ПЕРЕД удалением датасета, чтобы RefLink оставался валидным

5. **Функции кэширования:**
   - `loadDatasetCache()` - загрузка событий в кэш для компонента
   - `deleteComponentCache()` - удаление кэша компонента
   - `deleteDatasetCache()` - удаление всего кэша датасета
   - Все функции находятся в модуле `lib/datasets/cache-loader.ts`

**Преимущества:**
- ✅ Быстрый доступ к данным без постоянных запросов к ClickHouse

## 🚨 Важные замечания и частые ошибки

### ❌ КРИТИЧЕСКАЯ ОШИБКА: props.apiUrls is undefined

**Проблема:**
```javascript
// Vue компонент
const props = defineProps({
  apiUrls: { type: Object, required: true }
})

// ❌ ОШИБКА: Uncaught TypeError: can't access property "indexPage", props.apiUrls is undefined
const indexUrl = computed(() => props.apiUrls.indexPage)
```

**Причина:**
Vue компоненты выполняются **на клиенте** (в браузере), где:
- ❌ Нет доступа к Heap таблицам
- ❌ Нет контекста `ctx`
- ❌ Нельзя вызывать `route.url()` напрямую

**Решение:**
Генерировать все URL **на сервере** (в `.tsx` файлах) и передавать как props:

```typescript
// index.tsx (СЕРВЕР)
// @shared
import { jsx } from "@app/html-jsx"
import IndexPage from './pages/IndexPage.vue'
import { apiDatasetDeleteByIdRoute } from './api/datasets'
import { datasetConfigRoute } from './dataset-config'

export const indexPageRoute = app.html('/', async (ctx, req) => {
  const datasets = await AnalyticsDatasets.findAll(ctx, {})
  
  // ✅ Генерируем ВСЕ URL на сервере
  const apiUrls = {
    deleteDataset: apiDatasetDeleteByIdRoute.url(),
    indexPage: indexPageRoute.url(),
    datasetConfig: datasetConfigRoute.url()
  }
  
  return (
    <html>
      <body>
        {/* ✅ Передаём apiUrls как prop */}
        <IndexPage initialDatasets={datasets} apiUrls={apiUrls} />
      </body>
    </html>
  )
})
```

```vue
<!-- IndexPage.vue (КЛИЕНТ) -->
<script setup>
const props = defineProps({
  initialDatasets: { type: Array, default: () => [] },
  apiUrls: { type: Object, required: true }  // ✅ Без fallback!
})

// ✅ Используем URL из props
function deleteDataset(id) {
  const form = document.createElement('form')
  form.action = props.apiUrls.deleteDataset  // ✅ URL с сервера
  form.submit()
}
</script>
```

**Правило:**
🚨 **НИКОГДА не используйте хардкод URL!** Ни на сервере, ни на клиенте. Всегда:
1. Генерируйте URL на **сервере** через `route.url()`
2. Передавайте в Vue через **props** (`apiUrls`)
3. Используйте в `fetch()` и `form.action` только из **props**

### ❌ КРИТИЧЕСКАЯ ОШИБКА: Импорт Heap таблиц в Vue

**Проблема:**
```vue
<!-- ❌ ОШИБКА: NS_ERROR_CORRUPTED_CONTENT -->
<script setup>
import { AnalyticsDatasets } from '../tables/datasets.table'

// ❌ Heap таблицы НЕ работают на клиенте!
const datasets = await AnalyticsDatasets.findAll(ctx, {})
</script>
```

**Причина:**
Heap таблицы — это серверная логика Node.js, которая недоступна в браузере.

**Решение:**
Используйте SSR - загружайте данные на **сервере** и передавайте через **props**:

```typescript
// index.tsx (СЕРВЕР)
import { AnalyticsDatasets } from './tables/datasets.table'

export const indexPageRoute = app.html('/', async (ctx, req) => {
  // ✅ На сервере можем использовать Heap
  const datasets = await AnalyticsDatasets.findAll(ctx, {})
  
  return (
    <html>
      <body>
        {/* ✅ Передаём как props */}
        <IndexPage initialDatasets={datasets} />
      </body>
    </html>
  )
})
```

```vue
<!-- IndexPage.vue (КЛИЕНТ) -->
<script setup>
// ✅ Получаем готовые данные с сервера
const props = defineProps({
  initialDatasets: { type: Array, default: () => [] }
})

const datasets = ref(props.initialDatasets)
</script>
```

### ❌ КРИТИЧЕСКАЯ ОШИБКА: Использование .run() на клиенте

**Проблема:**
```vue
<!-- ❌ ОШИБКА: ctx is not an object or null -->
<script setup>
import { apiGetDataRoute } from '../api/data'

onMounted(async () => {
  // ❌ ctx недоступен на клиенте!
  const result = await apiGetDataRoute.run(ctx, {})
})
</script>
```

**Причина:**
Метод `.run()` требует серверный контекст `ctx`, который существует **только на сервере**.

**Решение:**
Используйте `fetch()` с URL из props:

```typescript
// data-page.tsx (СЕРВЕР)
import { apiGetDataRoute } from './api/data'

export const dataPageRoute = app.html('/', async (ctx, req) => {
  const apiUrls = {
    getData: apiGetDataRoute.url()  // ✅ Генерируем URL
  }
  
  return (
    <html>
      <body>
        <DataPage apiUrls={apiUrls} />
      </body>
    </html>
  )
})
```

```vue
<!-- DataPage.vue (КЛИЕНТ) -->
<script setup>
const props = defineProps({
  apiUrls: { type: Object, required: true }
})

onMounted(async () => {
  // ✅ Используем fetch() с URL из props
  const response = await fetch(props.apiUrls.getData)
  const result = await response.json()
})
</script>
```

| **ctx контекст** | ✅ Доступен | ❌ Нет |

| **.run()** | ✅ Доступно | ❌ ЗАПРЕЩЕНО |
