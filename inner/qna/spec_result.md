## ❌ Ошибки (требуют исправления)

### 1. §3.2 — статический `import` JSON в бандл UGC

> «подключается **статическим импортом** в серверном коде (например `import operationsCatalog from '../gateway/operationsCatalog.json'`)»

**Проблема:** среда исполнения Chatium — не Node.js; импорт `*.json` через ES-модули **не подтверждается** в типизации платформы. Это ставит под удар основной механизм поставки каталога.

**Исправление:** заменить `*.json` на TypeScript-модуль:

```ts
// lib/gateway/operationsCatalog.ts
export const operationsCatalog = { ... } as const
```

Импорт `import { operationsCatalog } from '../gateway/operationsCatalog'` — стандартный TS, работает в любом UGC-бандле. Альтернатива — runtime через `readWorkspaceFile` (с оговоркой о возможной перечитке на каждый запрос, что спека сама отмечает).

---

### 2. §3 / §3.5 — «JSON Schema (Draft 2020-12 или Draft-07)» для валидации `args`

> «JSON Schema (подмножество Draft 2020-12 или Draft-07)…», «валидатор JSON Schema из поля `argsSchema`»

**Проблема:** в Chatium **нет встроенного JSON Schema валидатора** (нет AJV в `/node_modules`). Платформа использует свой `@app/schema` (билдер `s`).

**Исправление варианты:**

- **(a)** Хранить схемы в формате собственного билдера: фабрики `(s) => ({...})` в каталоге, использовать `s.parse` / `s.safeParse` для валидации (поддерживается платформой).
- **(b)** Если нужен именно JSON Schema по требованиям SDK-клиентов — придётся принести AJV в бандл UGC и принять риски совместимости с движком; это надо явно зафиксировать в спеке как «AJV в бандле».

Текущий текст оставляет ложное впечатление, что валидатор JSON Schema даёт платформа.

---

### 3. §0.3, §3.3 — `withProjectRoot(route.url())` и `PROJECT_ROOT`

> «для ссылок с клиента — `withProjectRoot(route.url())`», «`PROJECT_ROOT`»

**Проблема:** в типизации платформы `withProjectRoot` и `PROJECT_ROOT` **не найдены**. Это, видимо, утилиты шаблона `p/template_project`, не платформы.

**Исправление:** либо явно отметить «утилита шаблона `template_project`, не платформы», либо использовать прямой `route.url()` (`RouteRef.url()` — стандарт Chatium и достаточен в большинстве случаев).

---

### 4. §7.3 — хук `@start/after-event-write`

> «хук `@start/after-event-write`»

**Проблема:** в `/node_modules/@start/*` такой хук **не найден**.

**Исправление:** удалить упоминание или заменить на проверенный хук (если есть аналог из `@start/sdk` или другой подсистемы — нужно подтвердить отдельно). В текущем виде — битая ссылка на несуществующий API.

---

### 5. §7-вступление — `queryAi`, `gcQueryAi`

> «трафик и `access_log` — `queryAi` к `chatium_ai.access_log`», «`gcQueryAi`»

**Проблема:** обе функции **не найдены** в `/node_modules`.

**Исправление:** удалить ссылки или подтвердить отдельной проверкой их источник (возможно, это часть закрытой/непубличной части платформы). В SSOT нельзя оставлять имена API без подтверждённого модуля.

## ⚠️ Потенциальные проблемы (опциональные исправления)

### 6. §2.1 / §3.5 — «общий обработчик слоя `/v1/`, если платформа это допускает»

**Статус:** в платформе **есть** `.pathParams()` с поддержкой wildcard `*` (например `app.get('/').pathParams(s => ({ '*': s.string() }))`), но мои базовые инструкции (по которым обычно генерируется код в этом workspace) запрещают переменные сегменты пути и предписывают «один файл — один путь `/`».

**Опциональное исправление:** **зафиксировать** в спеке выбор между двумя стратегиями:

- **(A) Строго по файлу на каждый `op`** (рекомендуемый путь по моим coding guidelines) — без fallback-роута;
- **(B) Один общий роут с `.pathParams({ op: s.string() })` и явной валидацией по каталогу** (поддерживается типами платформы).

Сейчас формулировка «если платформа это допускает» оставляет неопределённость, а CI-проверка «каждому `op` соответствует файл роута» (§3.2) намекает на стратегию A — но это стоит сделать явно.

---

### 7. §9.0 (по ссылкам) — формат ответа `{ statusCode, rawHttpBody, headers }`

**Статус:** платформа поддерживает это через интерфейс `TuneHttpHeadersResponse` (`rawHttpBody`, `headers?`, `statusCode?`) — `ctx.resp` НЕ существует (вопреки моим прежним советам в этой беседе — я был неточен).

**Опциональное исправление:** в §9.0 явно сослаться на `TuneHttpHeadersResponse` как на платформенный тип, а не на абстрактный «низкоуровневый ответ». Это уберёт двусмысленность для разработчика.

---

### 8. §2.2 — `Content-Type: application/json` для POST

**Статус:** требование жёсткое — это нормально, но обработка `application/x-www-form-urlencoded` и `multipart/form-data` платформа делает автоматически. Для будущих расширений (загрузка файлов) такая жёсткость может быть тормозом.

**Опциональное исправление:** добавить пометку «расширение Content-Type требует отдельной правки спеки», чтобы зафиксировать решение, а не закрыть путь.

---

### 9. §3.2 — «module-level singleton кэш не гарантирован»

**Статус:** утверждение в целом верное (V8-изоляты могут создаваться по запросу), но формулировка «**запрещено** считать нормативом» — слишком сильная: на практике module-level переменные часто переживают много запросов в рамках одного воркера, просто **без гарантий**.

**Опциональное исправление:** мягче — «недопустимо **полагаться** на межзапросный кэш как на гарантию», но допустимо использовать как **оптимизацию без обещаний**. Это сэкономит реализаторам лишний цикл чтения, не нарушая корректности.

---

### 10. §7.2 — `writeServerLog` из `lib/logger.lib`

**Статус:** это утилита шаблона `p/template_project`, не платформы. Это нормально, но в SSOT-доке стоит явно отделять платформенное (`@start/sdk`, `@app/*`) от шаблонного (`lib/*`).

**Опциональное исправление:** в §7.2 явно отметить «шаблонная обёртка над `ctx.account.log` / `writeWorkspaceEvent`», чтобы при ревью не возникала иллюзия платформенного API.
