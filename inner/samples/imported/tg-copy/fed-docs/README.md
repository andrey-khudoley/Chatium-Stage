# Федеративные чаты - Документация

## Введение

Федеративные чаты — это функциональность, позволяющая чатам из одного Chatium-аккаунта быть доступными для подключения из других аккаунтов. Участники из разных инстансов могут общаться в едином пространстве, при этом каждый остаётся в своём аккаунте.

**Основные принципы:**
- **Один хост** — чат "живёт" в одном аккаунте (хост-инстанс)
- **Множество клиентов** — другие аккаунты подключаются как клиенты
- **Единая история** — все видят одни и те же сообщения
- **Видимость инстанса** — каждый участник помечен доменом своего аккаунта
- **Безопасность** — полная изоляция через токены и WebSocket каналы

## Быстрый старт

### Для владельца хост-инстанса

1. Создай обычный чат (группу или канал)
2. В настройках чата поставь галочку "Федеративный чат" (только для Admin)
3. Скопируй federation URL
4. Отправь ссылку администраторам других инстансов

### Для владельца клиентского инстанса

1. Получи federation URL от хост-инстанса
2. Нажми "Подключиться к федеративному чату"
3. Вставь ссылку
4. Готово! Чат появится в списке

### Для участников

Просто присоединяйся к чату и общайся. Видно, кто из какого аккаунта.

## Технические требования

### Хост-инстанс
- ✅ Публично доступен по HTTPS
- ✅ Стабильное соединение для WebSocket
- ✅ Роль Admin в Chatium (для создания федеративных чатов)

### Клиентский инстанс
- ✅ Может делать исходящие HTTPS запросы
- ✅ Поддерживает WebSocket клиент
- ✅ Роль Admin в Chatium (для подключения к федеративным чатам)

### Ограничения
- Максимум 100 подключенных инстансов на чат
- Максимум 1000 участников во всех инстансах суммарно
- История: клиент получает последние 1000 сообщений при подключении

## Безопасность — основные моменты

✅ **Что обеспечено:**
- Токены 256-bit, криптографически стойкие
- Constant-time comparison против timing attacks
- Изоляция WebSocket по токенам
- Rate limiting на всех уровнях
- Валидация всех входящих данных
- Автоматическая блокировка при подозрительной активности

❌ **Чего нельзя:**
- Подключиться без правильного токена
- Получить события из других чатов
- Отправить сообщение от имени другого пользователя
- Перебрать токены (защита rate limiting)

## Сценарии использования

### 1. Компания с несколькими офисами
```
Офис Москва (host)
  └─ Создаёт "Общий проект"
  
Офис СПб (client)
  └─ Подключается
  
Офис Новосибирск (client)
  └─ Подключается
```
Все офисы общаются в едином чате.

### 2. Партнёрские проекты
```
Компания A (host)
  └─ Создаёт "Партнерский проект"
  
Компания B (client)
  └─ Подключается
```
Партнёры видят, кто из какой компании.

### 3. Образовательные платформы
```
Школа (host)
  └─ Создаёт "Урок математики"
  
Ученики из разных аккаунтов
  └─ Подключаются
```
Преподаватель ведёт урок, ученики участвуют.

## Важные замечания

### Обязательно:
- В каждом сообщении от федеративного участника должно быть видно, из какого аккаунта он пришёл
- Формат: "Имя из domain.com"
- WebSocket каналы должны быть изолированы по токенам
- Все ошибки должны логироваться

### Опционально, но рекомендуется:
- Certificate pinning для MITM защиты
- Токены с expiration
- Верификация пользователей через обратный запрос
- E2E тесты с несколькими браузерами

### Не делать:
- Не хранить историю сообщений на клиенте (только на хосте)
- Не доверять данным от клиента без валидации
- Не использовать обычные WebSocket каналы для федерации
- Не игнорировать rate limiting
