## Что описывает fed-docs (план)

Возможность подключать чаты одного Chatium-аккаунта (хост) из других аккаунтов (клиенты) с единой историей и изоляцией по токенам.

| Документ | Тема |
| --- | --- |
| `01-architecture.md` | концепция, таблицы `federation_connections/invitations/participants`, жизненный цикл |
| `02-api-specification.md` | federation endpoints, WebSocket protocol, rate limiting |
| `03-websocket-isolation.md` | изоляция WS-каналов по токенам |
| `04-security.md` | векторы атак, токены 256-bit, constant-time, rate limiting |
| `05-ui-ux.md` | отображение участников из разных аккаунтов |
| `06-deployment.md` | план внедрения, тесты, откат |
| `07-error-handling.md` | сетевые ошибки, конфликты, синхронизация |
| `08-federation-invites.md` | типы приглашений (локальные, cross-instance) |
| `09-transit-proxy.md` | транзит через Home Instance при блокировках между кластерами |
| `10-federation-establishment.md` | out-of-band установление соединения сервер-сервер |

## Если потребуется реализовать

Спека предполагает: единый источник истины на хосте (сообщения только в его Feed API), токен-based авторизация на каждый запрос, отдельный WS-канал `federation-<feedId>-<token>`, прокси-участники (`remote-<domain>-<userId>`), транзитное проксирование для заблокированных регионов. Ограничения: ≤100 инстансов и ≤1000 участников на чат.

Это **вне** объёма воссоздания по `spec/` — текущий проект федерацию не содержит.
