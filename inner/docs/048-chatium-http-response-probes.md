# Эмпирические форматы HTTP-ответов Chatium (UGC-приложение)

Документ зафиксирован по **реальным** ответам публичного GET (curl, без следования редиректам), workspace `s.chtm.khudoley.pro`, приложение `p/units/aom/lava_gc_integration`, каталог проб **`temp/`** (см. репозиторий: `temp/*/index.tsx`). Дата снятия: **2026-03-29** (UTC в заголовках ответа — 2026-03-29 22:15Z).

**Назначение:** при проектировании интеграционных проверок маршрутов (серверный `request`, мониторинг, скрипты) не полагаться только на «классический» HTTP-статус в первой строке ответа: у Chatium часть сценариев отдаёт **HTTP 200** при том, что в коде роута задумывались **4xx** или метаданные редиректа переносятся в тело JSON.

## Важные выводы для разработки и тестов

### 1. `ctx.resp.json(body, statusCode)` (второй аргумент — код) при внешнем GET

Пробы использовали именно вызов **`ctx.resp.json(..., 400)`** с кодом **вторым аргументом**. Для проб **400 / 401 / 403** фактический **HTTP-статус был 200**, при этом тело оставалось «ошибочным» JSON приложения. То есть:

- **`response.ok` в `fetch` (проверка 200–299) не отличает успех от задуманного 4xx**, если опираться только на статус.
- Для «честного» 4xx на проводе предпочтительно **`return ctx.resp.status(statusCode).json(body)`** или объект **`{ statusCode, rawHttpBody, headers }`**; либо парсить JSON и проверять поля (`error`, `success`, коды предметной области).

### 2. Редирект `ctx.resp.redirect`

Статус **302** и **`Location`** отражаются корректно, но **тело не пустое**: приходит служебный JSON с полем **`statusCode`: 302** и `simplifiedContentType`. Клиенты, ожидающие пустое тело при редиректе, могут быть удивлены.

### 3. Явный статус через `rawHttpBody`

Ответы **404** и **500** с объектом `{ statusCode, rawHttpBody, headers }` в **первой строке HTTP** совпали с заданным кодом.

### 4. Необработанное исключение в роуте

`throw` привёл к **HTTP 500** и JSON вида:

```json
{
  "success": false,
  "reason": "Unknown error during request to `https://…/temp/throws-error/index`: runUgcRequest … failed (500): temp probe: intentional server error",
  "statusCode": 500
}
```

Текст `reason` формируется платформой (в т.ч. URL внутри строки может отличаться от запрошенного хоста в зависимости от окружения). Для автотестов полезны поля **`success`**, **`statusCode`** в теле.

### 5. Страницы `app.html`

Для интеграционной проверки «страница открылась» недостаточно смотреть только на **200**: оболочка Chatium может быть общей; смысловую разметку приложения нужно искать внутри HTML или проверять отдельные маркеры.

## Воспроизведение

Пробы живут в проекте **`lava_gc_integration`**, путь **`./temp`** относительно `PROJECT_ROOT`. Пример базового URL:

`https://s.chtm.khudoley.pro/p/units/aom/lava_gc_integration/temp`

Проверка без следования редиректам:

```bash
curl -sS -D - -o /tmp/body.txt --max-redirs 0 \
  'https://<хост>/p/units/aom/lava_gc_integration/temp/http-400'
```
