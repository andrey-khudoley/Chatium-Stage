# Документация проекта в Chatium (README, docs, ADR, LLM)

Этот документ описывает **как вести документацию проекта** в условиях Chatium: фиксированный стек, без ручной серверной инфраструктуры, деплой автоматически по пушу.

---

## 1) Общие принципы

- Документация — часть продукта, обновляется вместе с кодом.
- Не пишите «воду»: только то, что помогает понимать и сопровождать проект.
- Если функциональности ещё нет — так и пишите: «не используется».
- Один источник правды для каждого типа информации (см. разделы ниже).

---

## 2) Корневой README.md (обязателен)

**README.md всегда в корне проекта.**  
Это **главная техническая карточка проекта** и место для информации, которая не подходит под `docs/`.

### Обязательная структура README.md

- **Название и назначение** (1–2 абзаца).
- **Ограничения платформы** (Chatium, нельзя менять зависимости, авто‑деплой).
- **Текущее состояние** (что реально работает сейчас).
- **Ссылки на docs/** (архитектура, API, данные, запуск).
- **TODO** (короткий список ближайших задач).
- **Changelog** (хронология ключевых изменений).

### Чего не хранить в README

- Историю диалогов (для этого `docs/LLM/`).
- Технические решения (для этого `docs/ADR/`).
- Подробную архитектуру и таблицы API (для этого `docs/`).

---

## 3) Папка docs/ (документация проекта)

### 3.1 `docs/architecture.md`

**Цель:** быстро объяснить устройство проекта.

**Минимальная структура:**

- Назначение проекта
- Ограничения платформы
- Основные сценарии
- Роутинг (основные страницы)
- Структура каталогов
- Интеграции (если есть)

### 3.2 `docs/api.md`

**Цель:** список всех API‑эндпоинтов.

**Минимальная структура:**

- Таблица публичных эндпоинтов
- Таблица внутренних эндпоинтов
- События/вебхуки (если есть)

Если API ещё нет — пишите «не реализовано».

### 3.3 `docs/data.md`

**Цель:** описание данных.

**Минимальная структура:**

- Таблица Heap‑таблиц (имя, файл, назначение, поля)
- Файлы/хранилище (если используется)
- Индексы/поиск (если есть)

### 3.4 `docs/run.md`

**Цель:** как запускать и проверять.

**Минимальная структура:**

- Требования (Chatium, фиксированный стек)
- Запуск/деплой (пуш → авто‑деплой)
- Конфигурация (`config.json`, если есть)
- Проверки (ручные шаги)

### 3.5 `docs/imports.md`

**Цель:** контроль циклических импортов.

**Минимальная структура:**

- Список импортов для каждого TSX‑роута
- Список импортов для Vue‑страниц

Обновлять **каждый раз**, когда меняются импорты в роутах/страницах.

## 4) ADR (архитектурные решения)

**Папка:** `docs/ADR/`  
**Назначение:** объяснить, **почему** решение было принято.

### Формат файлов

- Имена: `0001-...`, `0002-...` (сквозная нумерация).
- Один файл = одно решение.

### Минимальная структура ADR

- Контекст
- Решение
- Последствия (плюсы/минусы)

---

## 5) LLM (история диалогов с AI)

**Папка:** `docs/LLM/`  
**Назначение:** хранить историю диалогов с ИИ.

### Правила

- Имена файлов: сквозная нумерация `0001-...`, `0002-...`, `0003_дата_время.md` и т.д. Каждый **новый** файл — **следующий** номер после **максимального** уже лежащего в `docs/LLM/` (не создавать очередной `0001`, если в папке уже есть `0002+`). Перед добавлением файла посмотреть список в каталоге.
- **Только текст**: без копипасты кода.
- Каждый файл — краткое резюме диалога: задача, результат, договоренности.

### Минимальная структура

- Дата
- Агент
- Запрос
- Итог
- Важные договоренности

---

## 6) Файл .CHATIUM-LLM.md

Короткий «паспорт» проекта:

- что это;
- где ключевые документы;
- критические правила.

---

## 7) Важные особенности Chatium (для всей документации)

- Серверную инфраструктуру менять нельзя.
- Установленные библиотеки фиксированы.
- Любые изменения деплоятся сразу после пуша.
- URL всегда относительные, используйте `config/routes.tsx`.
- Не храните секреты в репозитории.
