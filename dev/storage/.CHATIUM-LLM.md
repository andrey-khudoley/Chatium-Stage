## Обзор проекта

Storage - это система для хранения JavaScript скриптов и CSS стилей в Heap-таблице с возможностью динамической отдачи через HTTP и удобным веб-интерфейсом для управления.

### Ключевые преимущества

- 🌐 **Веб-интерфейс** - управление скриптами через браузер с drag-and-drop загрузкой
- 📦 **Централизованное хранение** - все скрипты в одном месте
- 🔗 **Динамические ссылки** - подключение через простые URL
- 🔄 **CORS поддержка** - использование с любого origin
- 🧪 **100% покрытие тестами** - 23 автоматических теста
- ♻️ **Автоочистка** - удаление тестовых данных

### POST /api/scripts/create

Создать новый скрипт вручную

**Параметры**:
- `name` (required) - уникальное имя
- `type` (required) - "script" или "style"
- `content` (required) - содержимое
- `description` (optional) - описание

**Ответ**:
```json
{
  "success": true,
  "script": {...},
  "url": "https://s.chtm.khudoley.tech/dev/storage/serve~my-script.js"
}
```

### POST /api/scripts/upload (NEW v2.0.0)

Загрузить скрипт из файла

**Параметры**:
- `filename` (required) - имя файла с расширением (.js или .css)
- `content` (required) - содержимое файла

**Функциональность**:
- Автоматическое определение типа по расширению
- Проверка поддерживаемых форматов (.js, .css)
- Проверка уникальности имени
- Генерация описания с именем файла

**Ответ**:
```json
{
  "success": true,
  "script": {
    "id": "...",
    "name": "my-script",
    "description": "Загружен из файла my-script.js",
    "type": "script",
    "content": "..."
  },
  "url": "https://s.chtm.khudoley.tech/dev/storage/serve~my-script.js"
}
```

**Ошибки**:
```json
{
  "success": false,
  "error": "Поддерживаются только файлы .js и .css"
}
```

### POST /api/scripts/update

Обновить скрипт

**Параметры**:
- `id` (required)
- `name`, `description`, `type`, `content` (optional)

### POST /api/scripts/delete

Удалить скрипт

**Параметры**:
- `id` (required)

### GET /api/scripts/get/:id

## Технические детали

### File-based роутинг в Chatium

Chatium использует специфический подход к роутингу с тильдой (`~`):

```
Структура файлов → Базовый путь → Итоговый URL
```

**Примеры**:

| Файл | Роут внутри | Итоговый URL |
|------|-------------|--------------|
| `ui/index.tsx` | `app.html('/')` | `/dev/storage/ui` |
| `serve.tsx` | `app.get('/:filename')` | `/dev/storage/serve~{filename}.js` |

**Важно:**
- Тильда (`~`) отделяет базовый путь от явного маршрута
- Файл `index.tsx` в папке создает чистый URL без тильды
- Параметры `:filename` также используют тильду

### Возврат ответа с кастомными заголовками

Для отдачи текстового контента (скрипты, стили) с правильными заголовками используйте формат:

```typescript
// ✅ Правильно - возврат файла с Content-Type
export const serveScriptRoute = app.get('/:filename', async (ctx, req) => {
  const { filename } = req.params
  const extension = filename.match(/\.(js|css)$/)?.[1]
  const name = filename.replace(/\.(js|css)$/, '')
  
  const script = await ScriptsTable.findOneBy(ctx, { name })
  if (!script) {
    return 'Script not found'
  }
  
  // Определяем Content-Type
  let contentType = 'text/plain; charset=utf-8'
  if (extension === 'css') {
    contentType = 'text/css; charset=utf-8'
  } else if (extension === 'js') {
    contentType = 'application/javascript; charset=utf-8'
  }
  
  // Возвращаем с правильными заголовками
  return {
    statusCode: 200,
    rawHttpBody: script.content,
    headers: {
      'Content-Type': contentType
    }
  }
})

// ❌ Неправильно - ctx.header() не существует
ctx.header('Content-Type', 'text/css')
return script.content

// ❌ Неправильно - простой return не устанавливает Content-Type
return script.content
```

**Ключевые моменты**:
- Используйте `rawHttpBody` для текстового контента
- Явно устанавливайте `Content-Type` через объект `headers`
- Firefox строго проверяет MIME-типы для CSS и JS файлов
- Неправильный Content-Type вызывает ошибки `NS_ERROR_FAILURE` или HTTP 500

### Загрузка файлов (NEW v2.0.0)

#### Drag-and-Drop

```vue
<div 
  @drop.prevent="handleDrop"
  @dragover.prevent="isDragging = true"
  @dragleave.prevent="isDragging = false"
>
  <!-- Drop zone -->
</div>

<script setup>
async function handleDrop(e) {
  isDragging.value = false
  const files = e.dataTransfer.files
  if (files.length > 0) {
    await uploadFile(files[0])
  }
}
</script>
```

#### Выбор файла

```vue
<input 
  type="file" 
  accept=".js,.css" 
  @change="handleFileSelect"
  class="hidden"
/>

<script setup>
async function handleFileSelect(e) {
  const files = e.target.files
  if (files.length > 0) {
    await uploadFile(files[0])
  }
  e.target.value = '' // Сброс для повторной загрузки
}
</script>
```

#### Чтение файла

```javascript
function readFileContent(file) {
  return new Promise((resolve, reject) => {
    const reader = new FileReader()
    
    reader.onload = (e) => {
      resolve(e.target.result)
    }
    
    reader.onerror = () => {
      reject(new Error('Ошибка чтения файла'))
    }
    
    reader.readAsText(file)
  })
}
```

## Решенные проблемы

### 1. Ошибка "element is not a function"

**Проблема**: TypeError при загрузке Vue компонента с множественными импортами API роутов

**Решение**: 
- Создан упрощенный компонент UIPage.vue
- Импортируется ограниченное количество API роутов
- Постепенное добавление функциональности

### 2. Конфликт роутов

**Проблема**: Роут `serve.tsx` с `/:filename` перехватывал все запросы

**Решение**:
- Создана подпапка `ui/` с `index.tsx`
- Правильная структура file-based роутинга
- Чистый URL `/dev/storage/ui`

### 3. Not Found на /dev/storage/ui

**Проблема**: URL возвращал 404 ошибку

**Решение**:
- Перемещен файл из `ui.tsx` в `ui/index.tsx`
- Исправлен импорт Vue компонента
- Правильный относительный путь

### 4. Циклические зависимости при импорте

**Проблема**: Импорт нескольких API роутов вызывал конфликты

**Решение**:
- Минимизация импортов в компонентах
- Использование динамических импортов где возможно
- Правильная структура зависимостей

### 5. Неправильный Content-Type для CSS и JS файлов

**Проблема**: 
- CSS файлы отдавались с `Content-Type: text/plain` вместо `text/css`
- JS файлы отдавались с `Content-Type: text/plain` вместо `application/javascript`
- Firefox отклонял CSS файлы с ошибкой `NS_ERROR_FAILURE` из-за неправильного MIME-типа
- Скрипты также падали с ошибкой 500

**Причина**:
- Использовался несуществующий метод `ctx.header()` для установки заголовков
- Простой `return` строки не устанавливает Content-Type автоматически

**Решение**:
- Использование правильного формата возврата ответа в Chatium:
  ```typescript
  return {
    statusCode: 200,
    rawHttpBody: script.content,
    headers: {
      'Content-Type': contentType  // text/css или application/javascript
    }
  }
  ```
- Определение Content-Type на основе расширения файла (.js или .css)
- Все тесты (23/23) успешно проходят после исправления

**Документация**:
- Правильный способ возврата файлов с кастомными заголовками в Chatium
- Использование `rawHttpBody` вместо обычного return для текстового контента
- Явная установка заголовков через объект `headers`

## Важные замечания

### Требования

1. **Имена скриптов** должны быть уникальными
2. **CORS включен** - можно подключать с любого origin
3. **Тесты обязательны** - обновляйте при изменении функционала
4. **File-based роутинг** - используйте тильду в URL для параметров

### Ограничения

1. Поддерживаются только .js и .css файлы
2. Нет пагинации (все скрипты загружаются сразу)
3. Нет поиска и фильтрации в UI
4. Максимальный размер файла определяется настройками платформы

### Безопасность

- Требуется авторизация (requireAnyUser)
- Проверка уникальности имен
- Валидация типов файлов
- Санитизация входных данных
