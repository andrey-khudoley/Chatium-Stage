## Назначение

**Система аналитики** — платформа для работы с событиями из ClickHouse, предоставляющая:

- 📊 **Дашборды** — визуализация данных с графиками и таблицами
- 🔍 **Фильтры** — гибкая фильтрация по параметрам и датам
- 🎯 **Custom Events** — пользовательские события для специфичной бизнес-логики
- 🤖 **AI-ассистент** — помощь в настройке подписок на события
- ⚡ **Real-time** — обновления дашбордов через WebSocket
- 🎨 **Конструктор** — создание собственных дашбордов drag-and-drop

### Целевая аудитория

- **Администраторы** — полный доступ ко всем функциям (текущая реализация)
- **Аналитики** — просмотр и создание дашбордов (планируется)
- **Менеджеры** — просмотр готовых отчётов (планируется)

### Исправленная проблема

#### 🔍 Поиск пользователей не работал

**Симптомы**:
- При вводе текста в поле поиска страница просто обновлялась без изменений
- Результаты не фильтровались
- Network requests показывали запросы БЕЗ параметра `search`

**Корневая причина**:
Метод `.run()` для GET-роутов не передаёт параметры `query` в URL query string. Код в `AllUsers.vue` был:

```typescript
// НЕПРАВИЛЬНО - параметры не передаются в URL
apiGetUsersRoute.run(ctx, {
  query: {
    search: searchQuery.value,
    limit: limit.value,
    offset: offset.value
  }
})
```

**Решение**:
Использование прямого `fetch()` с `URLSearchParams` для формирования правильного query string:

```typescript
// ПРАВИЛЬНО - параметры передаются в URL
const params = new URLSearchParams()
if (searchQuery.value) {
  params.append('search', searchQuery.value)
}
params.append('limit', String(limit.value))
params.append('offset', String(offset.value))

const usersUrl = apiGetUsersRoute.url() + '?' + params.toString()

const usersResponse = await fetch(usersUrl).then(r => r.json())
```

**Файлы изменены**:
- `pages/AllUsers.vue` - исправлен вызов API с использованием `fetch()` и `URLSearchParams`
- `api/users.ts` - добавлено логирование для отладки поиска
- `tests/api/run-tests.ts` - добавлен тест `users_search_api`, все вызовы переписаны на `request()`
- `tests/shared/test-definitions.ts` - добавлен новый тест в список
- `index.tsx` - добавлен импорт `./tests/ai.tsx`

**Дополнительные исправления**:

1. **API Backend (api/users.ts)**:
   - Убран параметр `offset` из метода `searchBy()` - Heap не поддерживает offset для полнотекстового поиска
   - Добавлена ручная пагинация результатов поиска через `slice()`
   - Исправлен подсчёт `totalCount` для режима поиска
   - Добавлено подробное логирование для отладки

2. **Тесты (tests/api/run-tests.ts)**:
   - Создан новый тест `users_search_api` для проверки функционала поиска
   - Все HTTP вызовы в тестах переписаны с `.run()` на `request()` с абсолютными URL
   - Добавлена правильная обработка ответов `request()` через `.body`
   - Тесты проверяют:
     * Поиск по email (должен найти пользователя)
     * Поиск по имени (может найти несколько пользователей)
     * Пагинацию результатов поиска

### Исправленные проблемы

#### 1️⃣ Переход на resolved_user_id для подсчета пользователей
**Было**: Некоторые пользователи показывали 8 событий (должно быть 10)  
**Причина**: Подсчет по `user_id` вместо `resolved_user_id` (главного идентификатора GetCourse)  
**Решено**: Группировка и поиск по `COALESCE(resolved_user_id, user_id)`  
**Результат**: ✅ Все 10 событий отображаются корректно

**Детали**: См. раздел [Критические исправления F-1.4 → Проблема 5](#проблема-5-использование-resolved_user_id-вместо-user_id)

#### 2️⃣ Мульти-идентификаторный поиск событий
**Было**: Некоторые пользователи показывали 1 событие (должно быть 8)  
**Причина**: Поиск только по `user_email`, игнорировал события с `user_email = null`  
**Решено**: Двухэтапный поиск по ВСЕМ идентификаторам (user_id, uid, gc_visitor_id, email)  
**Результат**: ✅ Все 8 событий отображаются корректно

**Детали**: См. раздел [Критические исправления F-1.4 → Проблема 1](#проблема-1-неполное-отображение-событий-пользователя)

#### 3️⃣ Оптимизация загрузки страницы пользователей
**Было**: Timeout 30-60 секунд при загрузке `/dev/analytics~users`  
**Причина**: 50 запросов к ClickHouse для подсчёта событий каждого пользователя  
**Решено**: Батчи по 25, ленивая загрузка событий только при клике  
**Результат**: ✅ Загрузка <1 секунда, открытие спойлера 1-2 сек

**Детали**: См. раздел [Критические исправления F-1.4 → Проблема 2](#проблема-2-таймауты-при-загрузке-страницы-пользователей)

#### 4️⃣ Универсальный поиск пользователей  
**Было**: Поиск некоторых пользователей не работал (пользователь в ClickHouse, но не в Heap)  
**Причина**: Поиск только в Heap, fallback на ClickHouse отсутствовал  
**Решено**: Двухуровневый поиск с автосозданием в Heap из ClickHouse  
**Результат**: ✅ Поиск работает для ВСЕХ пользователей

**Детали**: См. раздел [Критические исправления F-1.4 → Проблема 3](#проблема-3-поиск-не-работал-для-неимпортированных-пользователей)

#### 5️⃣ Стабильная работа SQL запросов
**Было**: `totalCount: 8`, но `events: []` (пустой массив)  
**Причина**: SELECT несуществующих или несовместимых полей (uid, gc_visitor_id, gc_deal_id)  
**Решено**: Минимальный безопасный набор полей в SELECT  
**Результат**: ✅ Все запросы работают стабильно

**Детали**: См. раздел [Критические исправления F-1.4 → Проблема 4](#проблема-4-оптимизация-select-для-избежания-ошибок)

#### 6️⃣ Автоматическая синхронизация пользователей из GetCourse
**Было**: Новые пользователи из GetCourse не загружались автоматически в Heap  
**Причина**: Сложный SQL запрос с `GROUP BY` и `anyIf` возвращал 0 результатов  
**Решено**: Упрощенный запрос с фильтрацией по `urlPath = 'event://getcourse/user/created'`  
**Результат**: ✅ Синхронизация работает, пользователи успешно загружаются

**Детали**: См. раздел [Критические исправления v1.4.3 → Проблема 6](#проблема-6-синхронизация-пользователей-из-getcourse)

## Архитектура

### Технологический стек

- **Frontend**: Vue 3.5.13 (Composition API), Chart.js
- **Backend**: Chatium framework, TypeScript
- **Database**: 
  - ClickHouse — хранение событий (планируется)
  - Heap — конфигурация дашбордов, permissions (планируется)
- **Real-time**: WebSocket (@app/socket) (планируется)
- **AI**: Chatium Agents (@app/agents) (планируется)
- **Стили**: Inline styles (TailwindCSS-like), FontAwesome 6.4.0

### Основные принципы

1. **SSR** — серверный рендеринг Vue компонентов
2. **API-first** — все данные через REST API
3. **Role-based доступ** — проверка прав на уровне роутов
4. **Модульность** — переиспользуемые компоненты
5. **Типизация** — TypeScript для всего кода
6. **Тестируемость** — полное покрытие unit и интеграционными тестами

**`POST /api/users~update-historical`** - Импорт исторических данных из ClickHouse

**Описание**: Загружает пользователей из событий `event://getcourse/user/created` в ClickHouse и создаёт записи в Heap. Обрабатывает события за последние 7 дней для оптимизации производительности.

**Алгоритм** (v1.4.3 - упрощенный):
1. Загружает все существующие email из Heap в Set (пагинация по 1000)
2. Запрашивает события создания пользователей из ClickHouse (без GROUP BY)
3. Фильтрует по `urlPath = 'event://getcourse/user/created'` и последним 7 дням
4. Проверяет каждый email в Set и пропускает существующих
5. Создаёт новых пользователей с данными из события
6. Добавляет email в Set для дедупликации в пределах батча

**Возвращает**:
```typescript
{
  success: true,
  processed: 8,        // Обработано событий
  created: 8,          // Создано новых пользователей
  updated: 0,          // Обновлено существующих (не используется)
  totalInSystem: 4886  // Всего пользователей в системе
}
```

**SQL запрос** (v1.4.3):
```sql
SELECT
  user_id,
  user_email,
  user_first_name,
  user_last_name,
  user_phone,
  uid,
  ts,
  dt
FROM chatium_ai.access_log
WHERE 
  urlPath = 'event://getcourse/user/created'
  AND user_email IS NOT NULL
  AND user_email != ''
  AND dt >= '${последние 7 дней}'
ORDER BY ts DESC
LIMIT 10
```

**Особенности**:
- Упрощенный SELECT без `GROUP BY`, `anyIf`, `COALESCE` для стабильной работы
- Фильтрация только по событиям создания пользователей
- Временное окно 7 дней для оптимизации (настраивается)
- Использование Set вместо Map для эффективной проверки существующих email
- Lowercase сравнение email для корректной дедупликации
- Подробное логирование каждого создания пользователя
- Идемпотентность - можно запускать многократно без дублей

### Обновление тестов

**Когда добавлять тесты**:
- ✅ При создании нового API — добавить тест в категорию `api`
- ✅ При создании новой таблицы — добавить тесты в категорию `database`
- ✅ При добавлении функции — добавить в `functional`
- ✅ При создании потока — добавить в `integration`

**Как добавить новый тест**:

1. Добавить в `tests/shared/test-definitions.ts`:
```typescript
{
  name: 'api',
  tests: [
    // ... существующие
    { name: 'new_endpoint', description: 'POST /api/new-endpoint' }
  ]
}
```

2. Реализовать в `tests/api/run-tests.ts`:
```typescript
async function executeTest(ctx, category, testName) {
  switch (category) {
    case 'api':
      await runApiTest(ctx, testName)
      break
  }
}

async function runApiTest(ctx, testName) {
  switch (testName) {
    case 'new_endpoint':
      const result = await apiNewEndpointRoute.run(ctx, { data: 'test' })
      if (!result.success) throw new Error('Endpoint не работает')
      break
  }
}
```

3. Готово! Тест появится автоматически в обоих интерфейсах.

### Безопасность тестов

**Принцип**: Клиент передаёт ТОЛЬКО имя теста, все данные генерируются на сервере.

✅ **Правильно**:
```typescript
// Клиент
await apiRunSingleTestRoute.run(ctx, {
  category: 'infrastructure',
  testName: 'index_route_exists'  // ← Только имя теста
})

// Сервер
async function runTest(ctx, testName) {
  // Все параметры генерируются на сервере
  const testId = 'test-' + Date.now()
  const userId = ctx.user.id
  // ...
}
```

❌ **Неправильно**:
```typescript
// НЕ передавайте параметры от клиента!
await apiRunTestRoute.run(ctx, {
  testName: 'get_order',
  orderId: '12345'  // ← ОПАСНО! Инъекция!
})
```

#### Проблема 5: Использование resolved_user_id вместо user_id

**Симптомы**:
- Некоторые пользователи показывали только 8 событий
- По факту в ClickHouse было 10 событий с тем же `resolved_user_id`
- Аналогичные проблемы могли быть у других пользователей

**Причина**:
GetCourse использует два типа идентификаторов:
- `user_id` — может меняться или быть разным для одного пользователя (например, при разных сессиях)
- `resolved_user_id` — главный, стабильный идентификатор пользователя в GetCourse

Старый код группировал и считал события по `user_id`, пропуская события с другими значениями `user_id`, но тем же `resolved_user_id`.

**Пример**:
```
Событие                           | user_id          | resolved_user_id
event://getcourse/user/created    | 219894:476472048 | 219894:476472048
event://getcourse/message/sent    | 219894:476472048 | 219894:476472048
...еще 6 событий с одним user_id  | 219894:476472048 | 219894:476472048
event://https                     | 219894:476472050 | 219894:476472048  ← другой user_id!
event://https                     | 219894:476472050 | 219894:476472048  ← другой user_id!
```

Старый запрос (`GROUP BY user_id`) создавал две отдельные группы (8 + 2), новый (`GROUP BY resolved_user_id`) — одну группу из 10 событий.

**Решение**:
Изменена логика на использование `COALESCE(resolved_user_id, user_id)` во всех запросах:

**В api/users.ts** — синхронизация с ClickHouse:
```sql
SELECT
  COALESCE(resolved_user_id, user_id) as user_id,  -- Используем resolved_user_id как главный
  anyIf(user_email, user_email IS NOT NULL AND user_email != '') as user_email,
  any(user_first_name) as user_first_name,
  any(user_last_name) as user_last_name,
  any(user_phone) as user_phone,
  any(uid) as uid,
  min(ts) as ts
FROM chatium_ai.access_log
WHERE 
  COALESCE(resolved_user_id, user_id) IS NOT NULL
  AND COALESCE(resolved_user_id, user_id) != ''
  ${lastUserId ? `AND COALESCE(resolved_user_id, user_id) > '${lastUserId.replace(/'/g, "''")}'` : ''}
GROUP BY COALESCE(resolved_user_id, user_id)  -- Группируем по resolved_user_id
ORDER BY COALESCE(resolved_user_id, user_id)
LIMIT ${BATCH_SIZE}
```

**В api/users.ts** — подсчет событий:
```sql
-- Шаг 1: Находим ВСЕ resolved_user_id по email
SELECT DISTINCT

FROM chatium_ai.access_log

  AND user_email != ''

-- Шаг 2: Считаем события по resolved_user_id
```

**В api/events.ts** — получение событий:

**Уровень 1**: Поиск в Heap (быстро):
```typescript

```

**Уровень 2**: Если не нашли в Heap → поиск в ClickHouse + автосоздание:
```typescript
if (!users || users.length === 0) {
  // Ищем в ClickHouse по точному совпадению email
  const exactEmailQuery = `
    WITH user_ids AS (
      SELECT DISTINCT user_id
      FROM chatium_ai.access_log
      WHERE lowerUTF8(user_email) = lowerUTF8('${search}')
        AND user_id IS NOT NULL
      LIMIT 1
    )
    SELECT
      anyIf(user_email, user_email != '' AND user_email IS NOT NULL) as user_email,
      any(user_first_name) as user_first_name,
      any(user_last_name) as user_last_name,
      min(ts) as ts,
      max(ts) as last_ts
    FROM chatium_ai.access_log
    WHERE user_id IN (SELECT user_id FROM user_ids)
    GROUP BY user_id
    HAVING length(user_email) > 0
  `
  
  const result = await gcQueryAi(ctx, exactEmailQuery)
  
  // Создаём найденных пользователей в Heap
  for (const row of result.rows) {
    const newUser = await UsersTable.create(ctx, {
      email: row.user_email,
      firstName: row.user_first_name || undefined,
      lastName: row.user_last_name || undefined,
      registrationDate: new Date(row.ts),
      lastEventDate: new Date(row.last_ts),
      eventsCount: 0
    })
    users.push(newUser)
  }
}
```

- Некоторые события имеют `user_email = null`
- Но все события одного пользователя имеют одинаковый `user_id`
- `anyIf(user_email, ...)` выбирает первый непустой email из группы
- `HAVING length(user_email) > 0` гарантирует, что email нашёлся

#### Проблема 6: Синхронизация пользователей из GetCourse

**Симптомы**:
- Функция `apiUpdateHistoricalDataRoute` возвращала "Обработано: 0, Создано: 0, Обновлено: 0"
- Новые пользователи из GetCourse (события `event://getcourse/user/created`) не загружались в Heap
- События были видны в ClickHouse, но SQL запрос возвращал 0 строк

**Причина**:
Изначально использовался сложный SQL запрос с агрегацией:
```sql
-- ❌ Этот запрос возвращал 0 результатов
SELECT
  COALESCE(resolved_user_id, user_id) as user_id,
  anyIf(user_email, user_email IS NOT NULL AND user_email != '') as user_email,
  any(user_first_name) as user_first_name,
  any(user_last_name) as user_last_name,
  any(user_phone) as user_phone,
  any(uid) as uid,
  min(ts) as ts
FROM chatium_ai.access_log
WHERE 
  COALESCE(resolved_user_id, user_id) IS NOT NULL
  AND COALESCE(resolved_user_id, user_id) != ''
GROUP BY COALESCE(resolved_user_id, user_id)
ORDER BY user_id
```

Проблема: запрос не фильтровал по типу события и искал по всей таблице `access_log`, что приводило к конфликтам с `anyIf` и возвращало пустой результат.

**Решение**:
Упрощенный запрос с четкой фильтрацией по событиям создания пользователей:

```sql
-- ✅ Упрощенный запрос, который работает
SELECT
  user_id,
  user_email,
  user_first_name,
  user_last_name,
  user_phone,
  uid,
  ts,
  dt
FROM chatium_ai.access_log
WHERE 
  urlPath = 'event://getcourse/user/created'  -- Только события создания
  AND user_email IS NOT NULL
  AND user_email != ''
  AND dt >= '${последние 7 дней}'
ORDER BY ts DESC
LIMIT 10
```

**Ключевые изменения**:
1. Добавлен фильтр `urlPath = 'event://getcourse/user/created'` - только события создания пользователей
2. Убран `GROUP BY` - каждое событие обрабатывается отдельно
3. Убраны `COALESCE`, `anyIf`, `any()` - прямой SELECT полей
4. Добавлен фильтр по дате `dt >= '${последние 7 дней}'` - только недавние события
5. Сортировка по времени `ORDER BY ts DESC` - новые события первыми
6. Проверка email в Set вместо Map для эффективной дедупликации

**Изменения в логике обработки**:
```typescript
// Загружаем email существующих пользователей в Set
const existingEmailsSet = new Set<string>()
for (const user of heapUsers) {
  existingEmailsSet.add(user.email.toLowerCase())

// Проверяем, существует ли email

// Добавляем в Set чтобы не создавать дубли в пределах одного батча
existingEmailsSet.add(emailLower)
```

## Особенности реализации F-1.2

### Умное отображение деталей события

Компонент `EventsTable.vue` использует интеллектуальную фильтрацию полей:

**Функции проверки**:
```typescript
function hasValue(value) {
  // Проверяет что значение существует и не пустое
  if (value === null || value === undefined) return false
  if (value === '') return false
  if (value === '—') return false
  if (typeof value === 'number' && value === 0) return true
  return true
}

function hasUserData(event) {
  // Проверяет наличие хотя бы одного поля пользователя
  return hasValue(event.user_id) || 
         hasValue(event.user_email) || 
         hasValue(event.user_first_name) || 
         hasValue(event.user_last_name) || 
         hasValue(event.user_phone)
}

function hasEventParams(event) {
  // Проверяет наличие хотя бы одного параметра события
  return hasValue(event.action_param1) || 
         hasValue(event.action_param2) || 
         hasValue(event.action_param3) || 
         hasValue(event.action_param1_int) || 
         hasValue(event.action_param2_int) || 
         hasValue(event.action_param1_float) || 
         hasValue(event.action_param2_float) || 
         hasValue(event.action_param1_arrstr)
}
```

**Результат**: 
- Секции показываются только если есть данные
- Поля внутри секций показываются только если заполнены
- Компактное и читаемое отображение

### Секции деталей события

1. **Тип события** - всегда
2. **Название** - только если есть title
3. **Данные пользователя** - только если есть user_* поля
4. **Параметры события** - только если есть action_param* поля
5. **Дополнительные параметры (JSON)** - только если есть action_params
6. **Технические данные** - всегда (дата, время, GC IDs если есть)

## Особенности реализации F-1.4

### Batch Processing для импорта данных

Импорт исторических данных обрабатывает большие объёмы данных без таймаутов:

**Параметры**:
```typescript
const BATCH_SIZE = 150           // Размер батча для ClickHouse
const MAX_TOTAL_RECORDS = 5000   // Максимум записей за один запуск
const HEAP_LIMIT = 1000          // Лимит для пагинации Heap
```

**Алгоритм**:
1. **Предзагрузка пользователей**: Загружаются все email из Heap в Map постранично (по 1000)
2. **Итеративная обработка**: Запросы к ClickHouse батчами по 150 записей
3. **Email-based пагинация**: `ORDER BY user_email` и `WHERE user_email > 'last_email'`
4. **Создание/обновление**: Проверка в Map и создание новых или обновление дат регистрации
5. **Логирование**: Детальные логи каждой итерации для отладки

**Преимущества**:
- ✅ Избегает таймаутов (обработка порциями)
- ✅ Надёжная пагинация (по email, а не offset)
- ✅ Нет N+1 проблемы (предзагрузка в Map)
- ✅ Идемпотентность (можно запускать многократно)

### Агрегация данных из ClickHouse

Используется `GROUP BY` для получения уникальных пользователей:

```typescript
const query = `
  SELECT
    user_email,
    any(user_first_name) as user_first_name,
    any(user_last_name) as user_last_name,
    any(user_phone) as user_phone,
    any(uid) as uid,
    min(ts) as ts  -- Самая ранняя дата регистрации
  FROM chatium_ai.access_log
  WHERE
    user_email IS NOT NULL
    AND user_email != ''
    ${lastEmail ? `AND user_email > '${lastEmail.replace(/'/g, "''")}'` : ''}
  GROUP BY user_email
  ORDER BY user_email
  LIMIT ${BATCH_SIZE}
`
```

**Особенности**:
- `any()` — берёт любое значение из группы (для ФИО, телефона, uid)
- `min(ts)` — выбирает самую раннюю дату (дата первой регистрации)
- Экранирование апострофов в email для безопасности
- Фильтрация пустых email

### Обработка NULL из ClickHouse

ClickHouse может возвращать `null` для строковых полей:

```typescript
// ❌ НЕПРАВИЛЬНО - передача null в Heap
await UsersTable.create(ctx, {
  firstName: event.user_first_name  // может быть null
})

// ✅ ПРАВИЛЬНО - замена на undefined
await UsersTable.create(ctx, {
  firstName: (event.user_first_name && event.user_first_name !== null && event.user_first_name !== '') 
    ? event.user_first_name 
    : undefined
})
```

**Heap ожидает**:
- Заполненное значение: `string` (например, `"Ivan"`)
- Отсутствие значения: `undefined` (НЕ `null`, НЕ `""`)

### Автоматическое обновление через хуки

При записи нового события автоматически обновляются данные пользователя:

```typescript
app.hook('@start/after-event-write', async (ctx, eventData) => {
  if (eventData.email) {
    await handleUserRegistrationEvent(ctx, eventData)
  }
})

async function handleUserRegistrationEvent(ctx, eventData) {
  const existingUser = await UsersTable.findAll(ctx, {
    where: { email: eventData.email },
    limit: 1
  })
  
  if (existingUser.length > 0) {
    // Обновляем существующего
    await UsersTable.update(ctx, {
      id: existingUser[0].id,
      lastEventDate: new Date(),
      eventsCount: existingUser[0].eventsCount + 1
    })
  } else {
    // Создаём нового
    await UsersTable.create(ctx, {
      email: eventData.email,
      firstName: eventData.firstName || undefined,
      // ... остальные поля
      registrationDate: new Date(),
      eventsCount: 1
    })
  }
}
```

- ✅ Обновление даты последней активности
