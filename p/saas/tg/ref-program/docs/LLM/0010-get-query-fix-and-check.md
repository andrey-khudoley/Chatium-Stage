# LLM 0010 — исправление передачи query для GET и проверка @check

Диалог: предупреждение при загрузке админки (api/settings/get — отсутствует key), выяснение причины, исправление вызова GET с query по inner/docs, проверка остальных участков, запуск оркестратора @check.

---

### Пользователь

При загрузке страницы админки я получил warn: api/settings/get — Валидация не пройдена: отсутствует или пустой key, в логе query пустой. Почему?

### Ассистент

Разобрал код: в api/settings/get сервер ожидает req.query.key; в AdminPage в loadProjectName вызывается getSettingRoute.run с передачей query: key project_name. Значит, с точки зрения кода key передаётся. Предупреждение с пустым query могло быть из-за того, что рантайм Chatium при .run() сначала шлёт запрос без параметров, или из-за сериализации. Предложил вариант текста для запроса в поддержку.

---

### Пользователь

Приведи текст для запроса в поддержку с примерами кода.

### Ассистент

Составлен текст запроса в поддержку: описание проблемы, ожидаемое и фактическое поведение, пример лога, фрагмент клиентского вызова и серверной валидации, вопрос — может ли рантайм отправлять дополнительный GET без query.

---

### Пользователь

Тут проблема явно в том, что ты неправильно передаёшь query. Он передаётся не также, как body в post. Изучи документацию inner/docs и убедись в этом.

### Ассистент

Проверены inner/qna/001-run.md и inner/docs/002-routing.md. В таблице типов вызовов указано: GET с query — синтаксис route.query({ key: val }).run(ctx), а POST с body — route.run(ctx, { body }). То есть для GET параметры query передаются через метод .query(), а не вторым аргументом в .run(). Второй аргумент .run() — это body запроса. Текущий вызов getSettingRoute.run(ctx, { query: { key: 'project_name' } }) ошибочен: query так не попадает в URL. Правильно: getSettingRoute.query({ key: 'project_name' }).run(ctx).

---

### Пользователь

Отлично. Вот и исправь. Проверь остальные участки проекта на наличие аналогичных ошибок с query.

### Ассистент

**Сделано:**
- В pages/AdminPage.vue вызов заменён на getSettingRoute.query({ key: 'project_name' }).run(ctx).
- По проекту проверены все вызовы .run(): saveSettingRoute.run(ctx, { key, value }) — POST, body передаётся верно; listSettingsRoute нигде не вызывается; других GET с query не найдено.
- Линтер ошибок не показывает.

**Итог:** единственная некорректная передача query была в loadProjectName в AdminPage.vue, исправлена.
