# ADR-0002: Настройки в Heap и слоистая архитектура API

Дата: 2026-02-01

## Контекст

Некоторые настройки проекта будут часто меняться (название проекта, уровень логирования и т.п.). Хранить их в коде или конфиг-файлах неудобно — требуется пересборка и деплой при каждом изменении. Нужна возможность изменять настройки без деплоя, при этом сохраняя валидацию, дефолты и разграничение прав доступа.

## Решение

### 1. Настройки в Heap (key-value)

Создана таблица `t__template-project__setting__7Fk2Qw` с полями:
- `key` (string) — ключ настройки;
- `value` (Heap.Any) — значение любого типа.

Дефолты (если записи в БД нет):
- `project_name` — A/Ley Services;
- `project_title` — A/Ley;
- `log_level` — Info;
- `logs_limit` — 100.

### 2. Разделение слоёв

Принцип разделения ответственности при работе с данными:

| Слой | Каталог | Ответственность |
| --- | --- | --- |
| **Таблицы** | `tables/` | Схемы Heap. Только определение структуры данных. |
| **Репозитории** | `repos/` | Работа с БД: CRUD, запросы. Без бизнес-логики. |
| **Бизнес-логика** | `lib/` | Правила, дефолты, валидация, вычисления. Вызывает репозитории. |
| **API** | `api/` | HTTP-эндпоинты, парсинг, первичная валидация, проверка прав. Вызывает lib. |

Поток данных: `HTTP → API → lib → repos → Heap`.

### 3. File-based структура API

Вместо одного файла `api/settings.ts` используются отдельные файлы:
- `api/settings/list.ts` — GET, список настроек;
- `api/settings/get.ts` — GET, одна настройка (query: `key`);
- `api/settings/save.ts` — POST, сохранение (body: `{ key, value }`).

Правило: один файл — один эндпоинт с путём `/`; URL формируется из структуры каталогов (например, `/api/settings/list`).

### 4. Проверка прав

Все эндпоинты настроек доступны только роли **Admin** (`requireAccountRole(ctx, 'Admin')`).

## Последствия

**Плюсы:**
- настройки можно менять без деплоя;
- чёткое разделение слоёв упрощает поддержку и тестирование;
- file-based API соответствует принятому роутингу и облегчает навигацию по коду;
- циклических зависимостей нет: tables ← repos ← lib ← api.

**Минусы:**
- больше файлов (по одному на эндпоинт);
- требуется дисциплина при добавлении новых настроек (обновление lib, дефолтов, валидации).
