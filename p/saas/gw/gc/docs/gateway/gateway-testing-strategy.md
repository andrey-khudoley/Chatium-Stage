# Интеграционное и юнит-тестирование gateway GetCourse

Документ задаёт **норматив по тестам** слоя gateway: как уменьшить ручной ввод в Heap за счёт **цепочек вызовов** API, как вести **массовый прогон** без лишних созданий сущностей в GetCourse, как **убирать или маркировать** следы тестов и как соотносится это с [gateway-operation-manual](./gateway-operation-manual.md) (**§5.8**, **§9–§10**, **§12** «Открытые вопросы», **§13** «История изменений»).

Машиночитаемый перечень обязательных полей запросов по **`op`**: [gc-required-fields-by-op.json](./gc-required-fields-by-op.json). Пересборка: [extract-gc-required-fields.py](./extract-gc-required-fields.py). Статусы операций **`availability`** и UI страницы тестов — **§9** ниже и [gateway-operation-manual](./gateway-operation-manual.md) **§2.11**, **§5.9**.

## 1. Минимальный Heap и общая идея «цепочек»

### 1.1. Всегда обязательный ввод (уровень A)

До любого интеграционного вызова **`/v1/{op}`** в Heap должны быть заданы только (**§5.8** уровень A):

| Ключ | Назначение |
| --- | --- |
| **`gc_developer_api_key`** | Ключ разработчика |
| **`gc_test_school_api_key`** | Ключ API тестовой школы → заголовок **`X-Gc-School-Api-Key`** |
| **`gc_test_school_host`** | Хост школы → **`X-Gc-School-Host`** |

Иные ключи **`gc_itest_*`** считаются **не обязательными по умолчанию**, если тот же результат можно получить вызовом другого **`op`** в рамках одного прогона (см. **§2**).

### 1.2. Контекст прогона (память процесса, не Heap)

Интеграционный раннер держит объект **контекста сессии** (в памяти тестового процесса): например `dealId`, `userId`, `offerId`, `groupId`, `lessonId`, `lessonAnswerId`, `webinarId`, `webinarCommentId`, `exportTaskId`, и т.д. Значения заполняются **результатами успешных вызовов** gateway и переиспользуются следующими тестами. **Исключение — `offerId`**: в GetCourse это **уникальный код предложения**, задаваемый вручную в админке школы; его **не** брать из **`getOffers`** как внутренний **`id`** (см. **§2**). Для прочих полей ориентир — колонка «Как получить без Heap» в **§2**: если там описана цепочка API, отдельный ввод в Heap не нужен; если там указан **`gc_itest_*`** — значение задают в настройках приложения.

Правило: **сначала производитель ID (или список), потом потребитель**; порядок тестов в файле/сьюте явно задаётся или группируется по фикстуре «setup-модуль». Детальная фазовая модель — **§3.1**.

### 1.3. Статусы `availability` и допустимость интеграционного прогона

Эффективный статус каждого **`op`** берут из каталога gateway (поле **`availability`** в **`GET /v1/operations`** и в сборке приложения — [gateway-operation-manual](./gateway-operation-manual.md) **§2.11**, **§3.4**).

- **Интеграционные тесты**, которые выполняют **реальный** HTTP к **`/v1/{op}`** и инициируют исходящий вызов к GetCourse, **разрешено** ставить в очередь прогона **только** для операций со статусом **`enabled`** или **`beta`**.
- Для операций **`disabled`** или **`unsupported`** **запрещено** запускать интеграционные тесты **при любой конфигурации** (включая отладку, ручной «форс» в UI и временные флаги в коде). Такие строки на странице тестирования **не** получают кнопку запуска интеграционного сценария; CI-раннер **обязан** завершаться ошибкой, если обнаружен интеграционный кейс для **`disabled`**/**`unsupported`**.

## 3. Порядок прогона: фазы, цепочки вызовов, реестр зависимостей

Интеграционный сценарий — это **не** набор независимых вызовов **`/v1/{op}`**. Чтобы выполнить тест для операции **B**, часто нужно **сначала** вызвать **A**, разобрать **ответ** (или тело по контракту GC), положить извлечённые **`id`** и структуры в **контекст сессии** и **только потом** вызвать **B** с этими значениями. Машиночитаемый перечень обязательных полей по каждому **`op`**: [gc-required-fields-by-op.json](./gc-required-fields-by-op.json); ниже — **семантика цепочек** по [new_api_schema.json](./new_api_schema.json) и [legacy_api_schema.json](./legacy_api_schema.json).

### 3.1. Четыре фазы одного интеграционного сьюита

1. **Фаза 1 — только то, что задаёт администратор (Heap), без «производителей» в API.** Сюда относятся вызовы, которым для **`args`** достаточно уровня A (**§1.1**) и ключей **`gc_itest_*` / `gc_itest_offer_id`**, если **иного способа** получить значение в GetCourse нет (типичный пример — **`getOfferById`** с **`offerId`** из Heap — **§2**). Такие тесты идут **первыми**, если от них не зависят другие; иначе их оставляют до момента, когда нужен только «справочный» вызов.
2. **Фаза 2 — производители идентификаторов и структур.** Операции, которые **создают** сущность или **возвращают** **`id`** / массив для дальнейших шагов: например **`createDeal`** → **`dealId`** в контексте, **`addUser`** → **`userId`**, **`exportUsers`** → **`exportId`**, **`getAllGroups`** → **`groupId`**, **`getAllWebinars`** → **`webinarId`**, **`getLessonAnswers`** → **`lessonAnswerId`**, **`addDealPositions`** / **`getDealFields`** → **`positionIds`**, и т.д. (**§2**, **§3.4**). Ответ каждого такого шага **парсится**, нужные поля **сохраняются в памяти прогона** (не в Heap).
3. **Фаза 3 — потребители.** Все **`op`**, которым нужны **`dealId`**, **`userId`**, **`dialogId`**, **`webinarId`**, **`lessonAnswerId`**, **`exportId`**, **`positionIds`**, **`commentId`** и пр., получают их **из контекста** после фазы 2 (или из фазы 1 для **`offerId`**). Один и тот же **`dealId`** / **`userId`** в рамках сьюита **переиспользуют** по правилам **§3.3**.
4. **Фаза 4 — деструкторы и приведение к безопасному состоянию.** В **`afterAll`** / **`finally`**: откат мутаций, где есть обратный **`op`**, либо **терминальный статус** сущности. Подробно — **§3.5**.

**Правило запуска одного теста:** если в [gc-required-fields-by-op.json](./gc-required-fields-by-op.json) у **`op`** есть обязательное поле, которое **не** берётся из Heap уровня A/B (**§5.8** manual), раннер **обязан** либо выполнить **предыдущий** шаг цепочки и заполнить контекст, либо **явно пропустить** кейс с диагностикой «не собран контекст».

### 3.2. Обязательная модель «предыдущий ответ → следующий запрос»

- **Интеграционный кейс** описывают как упорядоченный список шагов **`(op₁, op₂, …)`**; для шага **`opₖ`** каждый аргумент либо **литерал сьюита** (разрешённые константы: **`transport`**, **`action`** модерации и т.д.), либо **`context[field]`**, заполненный при **`opᵢ`**, **`i < k`**.
- **Запрещено** подставлять в **`opₖ`** «магические» **`id`**, не полученные в этом же прогоне или из Heap по **§2** / **§5.8**, если документ не относит поле к **`gc_itest_*`**.
- Для **асинхронных** цепочек (**`exportUsers`** / **`exportDeals`** → **`getExportResult`**) между запуском экспорта и опросом результата выдерживать **§6** (throttle) и логику готовности задачи в ответе GC.

### 3.3. Массовый прогон: одна сделка на сьюит

Частный случай фаз 2–3 для **`dealId`**:

1. **Один раз** в **`beforeAll`** (фаза 2) вызвать **`createDeal`** (Legacy) **или** выбрать существующую сделку из **`exportDeals`**, если политика школы это допускает; сохранить **`dealId`** в контексте.
2. Все тесты фазы 3, которым нужен **`dealId`** (**`getDealFields`**, **`addCommentToDeal`**, **`addDealPositions`**, … — **§3.4**), используют **этот же** идентификатор.
3. **Не** вызывать **`createDeal`** перед каждым кейсом без причины (изоляция только при необратимой порче данных).

### 3.4. Реестр зависимостей по `op` (new + legacy)

Ниже — **все операции** из [gc-required-fields-by-op.json](./gc-required-fields-by-op.json): что требуется на проводе и **типичный предшественник** (цепочка). Детали тел Legacy — в **`params`** по [legacy_api_schema.json](./legacy_api_schema.json) и help GetCourse; для **`new`** дополнительные поля из **`allOf`** / **`UserIdentifier`** см. в [new_api_schema.json](./new_api_schema.json) (например **`userId`** **или** **`email`** там, где схема допускает идентификацию пользователя по email).

**Контур `legacy`**

| `op` | Что обязательно на проводе | Откуда берут значения (фаза / предшественник) |
| --- | --- | --- |
| **`addUser`** | `action`, `key`, `params` (блок `user`, минимум email) | Фаза 2: в **`user.email`** — хардкод **`tester@khudoley.pro`**; **`userId`** из **`result`** — в контекст для **`new`**-операций |
| **`createDeal`** | `action`, `key`, `params` (блоки `user`, `deal` с **`offer_code`/`offer_id`** = уникальный код предложения) | Фаза 1: **`gc_itest_offer_id`**; после вызова — **`dealId`** в контекст |
| **`exportDeals`**, **`exportUsers`**, **`exportPayments`** | query `key` | Фаза 2–3: из ответа — **`exportId`** (или аналог) → **`getExportResult`** |
| **`exportGroupUsers`** | path **`groupId`**, query `key` | Фаза 2: **`getAllGroups`** → **`groupId`** |
| **`getCustomFields`** | query `key` | Фаза 1–3: без предшественника по **`id`** |
| **`getExportResult`** | path **`exportId`**, query `key` | Фаза 2: **`exportUsers`** / **`exportDeals`** / … |

**Контур `new` — GET, обязательные query/path**

| `op` | Параметры | Предшественник / источник |
| --- | --- | --- |
| **`getDealCalls`**, **`getDealComments`**, **`getDealCustomFields`**, **`getDealFields`** | query **`dealId`** | **`createDeal`** или общий контекст сьюита (**§3.3**) |
| **`getDialogHistory`** | query **`dialogId`** | Heap **`gc_itest_*`**, вебхук **`setUri`**, отдельный сьюит |
| **`getLessonAnswers`** | query **`lessonId`** | **`getTrainings`** / Heap **`gc_itest_lesson_id`** |
| **`getOfferById`** | query **`offerId`** | Heap **`gc_itest_offer_id`** (**§2**) |
| **`getUserByTelegramChatId`** | query **`chatId`** | Heap или **`skip`** |
| **`getWebinarsByIds`** | body **`ids`** (массив) | **`getAllWebinars`** |

Остальные **`new`** **GET** в json без обязательных query/body — фаза 2 как **каталоги** или фаза 3 как **чтение** без внешних **`id`** из контекста (кроме случаев, когда тест сам задаёт фильтры по опциональным полям схемы). Полный явный перечень из реестра [gc-unified-op-registry-v0](./gc-unified-op-registry-v0.md), чтобы ни один **`op`** не остался без покрытия:

- **Каталоги без аргументов:** **`getAllGroups`**, **`getAllPersonalManagers`**, **`getAllWebinars`**, **`getTrainings`**, **`getOffers`**, **`getOffersTags`**, **`getDealCancelReasons`**, **`getDealsTags`**.

### 3.6. Падение теста и всё равно teardown

Шаги фазы 4 выполняют в **`afterAll`** / **`finally`**, чтобы при падении середины сьюита **попытаться** перевести общую сделку в **`status: "false"`** (или иной согласованный teardown). Если teardown упал — запись в лог по **§5** (тип сущности, **`op`**, **`id`**).

Аналогичный принцип **«один раз создать → много раз использовать → завершить деструктором»** — для **`userId`** (общий тестовый пользователь на сьюит), **вебинаров** (по возможности только чтение), **уроков** (минимизировать мутации ответов — **§4**).

## 4. Мутации и минимизация мусора

- Предпочитать **GET** и **чтение** там, где это проверяет контракт gateway.
- Для **POST**, меняющих данные (**комментарии**, **баланс**, **группы**, **позиции сделки**), группировать в конце сьюта или использовать **отдельную** «расходную» сделку/пользователя, чтобы не ломать общую фикстуру раньше времени.

## 5. Обязательная политика очистки и учёта артефактов

После **любого** интеграционного теста, который создал или изменил данные в GetCourse, действует лестница ответственности:

1. **Откат или удаление.** Если через доступные **`op`** gateway можно **удалить** или **вернуть состояние** (отмена заказа, удаление позиций, обнуление тестового поля и т.д.) — сделать это в **`afterEach`** / **`afterAll`** того же сьюта или в общем teardown.
2. **Пометка как тестовых данных.** Если удалить нельзя — создавать сущности с **явной пометкой**: для пользовательских сценариев — заранее согласованный тестовый email **`tester@khudoley.pro`**; для прочих сущностей — комментарии с префиксом **`[gateway-itest]`**, номера/поля, видимые администратору школы.
3. **Невозможно ни удалить, ни пометить.** В конце прогона вывести **явное предупреждение** в лог раннера (и при падении CI — в артефакт отчёта): тип сущности, **`op`**, известный **ID** из ответа GC, текст «артефакт остался в системе GetCourse».

Запрещено молча оставлять созданные объекты без одной из трёх мер выше.

То же относится к **асинхронным экспортам**: после проверки **`getExportResult`** фиксировать, что задача завершена; не копить повторные экспорты без нужды (лимиты [help](https://getcourse.ru/help/api)).

## 6. Лимит частоты запросов к GetCourse

Между двумя последовательными вызовами, каждый из которых приводит к **одному** исходящему HTTP от gateway к GetCourse, интервал **≥ 1000 ms** (**1 rps** на процесс). Параллельные воркеры могут суммировать нагрузку — в CI предпочтительно **последовательное** выполнение интеграционных тестов или глобальный семафор.

Подробнее об общей архитектуре ответов и ошибок — [gateway-operation-manual](./gateway-operation-manual.md) **§9–§10**. Норматив по **непрерывному** серверному логированию (каждая существенная ветка, код этапа + текст + безопасные «сырые» данные в **`payload`**, согласование с уровнем из **`template_project`**) — **§7.2.1** того же manual.

## 7. Юнит-тесты

Юнит-прогон **без** реальных вызовов **`@app/request`** к GetCourse и **без** обращения к Heap настроек тестовой школы. Он **не заменяет** интеграционный (**§3**).

### 7.1. Назначение

Проверить **чистую логику gateway** при фиксированных входах: построение исходящего HTTP (метод, путь, query, заголовки, тело), разбор и нормализация ответов, ветвление **Legacy** vs **new**, маппинг кодов ошибок к контракту [gateway-operation-manual](./gateway-operation-manual.md) (**§9–§10**).

### 7.2. Сценарии по слоям (каталог кейсов)

Каждый юнит-кейс — **Arrange → Act → Assert**; в Arrange задают **`ctx`** (или его мок), **`op`**, **`args`**, при необходимости **заглушку ответа** «как будто от GC».

| Слой / тема | Что проверять | Типовой Arrange |
| --- | --- | --- |
| **Маршрутизация `op` → HTTP** | Совпадение с [gc-op-http-mapping.json](./gc-op-http-mapping.json): путь, метод, наличие Bearer vs form `key` | Одна пара **`(op, contour)`** |
| **Сериализация тела / query** | Обязательные поля из [gc-required-fields-by-op.json](./gc-required-fields-by-op.json) попадают в исходящий запрос; опциональные — по правилам схемы | **`args`** с минимальным валидным набором |
| **Контур Legacy, `params`** | Base64(JSON) или иной контракт шаблона соответствует [legacy_api_schema.json](./legacy_api_schema.json) и внутренним хелперам gateway | Строки **`action`**, фиктивный **`key`** в моке |
| **Валидация входа** | Отклонение невалидных **`args`** до сети (типы, обязательность) | Наборы граничных значений |
| **Разбор ответа GC** | Успех: извлечение **`result`** / **`data`** в форму ответа gateway; ошибка: классификация по **§10** manual | Фикстуры JSON ответов GC |
| **Ошибки сети / таймаут** | Поведение без тела ответа; повторяемость сообщений для логов | Мок **`request`** бросает / возвращает статус |

### 7.3. Связь с интеграционными фазами

В юнит-тестах **нет** фаз **§3.1** в смысле живого GC: цепочка **`createDeal` → `dealId` → `getDealFields`** не выполняется. Вместо этого для слоя маппера задают **пары** `(входные args, ожидаемый фрагмент HTTP)`; для слоя разбора ответа — `(сырой ответ GC, ожидаемая структура для клиента)`. Таблица **§3.4** используется как **чеклист**: для каждого **`op`** должен существовать хотя бы один юнит на **сборку запроса** и при наличии нетривиального разбора — на **разбор ответа**.

### 7.4. Пороги и регрессия

Минимальное покрытие веток и пороги — в репозитории приложения; при добавлении **`op`** в каталог — одновременно дополнять юнит-кейсы по **§7.2**.

## 9. Страница тестирования в админке: фильтры, переключатели, визуальные состояния

Норматив для **Vue-страницы** тестирования gateway в приложении Chatium (`pages/` + **`fetch`** к `api/*`, данные настроек через существующие эндпоинты, роль **`Admin`** — [gateway-operation-manual](./gateway-operation-manual.md) **§0**, **§5.9**). Цель — не допустить «тихого» нарушения правил **§1.3** и **§6** (throttle).

### 9.1. Интеграционные тесты: только `enabled` и `beta`

- Каждая строка сценария, привязанная к **`op`**, отображает текущий **`availability`** из каталога (тот же источник, что раннер).
- **Запуск** интеграционного сценария активен **только** если **`availability`** ∈ { **`enabled`**, **`beta`** }.
- Для **`disabled`** и **`unsupported`** элемент управления запуском **отключён** (disabled), фон строки — **нейтральный серый** (`muted` / `slate` / `zinc` в духе Tailwind: например **`bg-slate-100`** в светлой теме и **`bg-slate-800/60`** в тёмной), без анимаций «пульс», чтобы визуально отличать от доступных сценариев.

### 9.2. Юнит-тесты: глобальный переключатель «форс всех»

- На странице размещается переключатель **«Запускать юнит-тесты для всех операций»** (или эквивалентная подпись). **По умолчанию — выключен** (`false`).
- **Выключен (по умолчанию):** в автоматический юнит-прогон (кнопка «Запустить юниты» / CI hook) попадают **только** тесты, отнесённые к **`op`** со статусом **`enabled`** или **`beta`** (согласовано с **§1.3** по духу осторожности).
- **Включён:** разрешается запуск **полного** набора юнит-тестов репозитория, включая кейсы для операций **`disabled`**/**`unsupported`** (например проверка, что маппер возвращает **`INVOKE_OP_DISABLED`** / **`INVOKE_OP_UNSUPPORTED_BY_GC`** без сети — [gateway-operation-manual](./gateway-operation-manual.md) **§2.11**, **§10**). Переключатель **не** влияет на **интеграционные** тесты (**§9.1**).

### 9.3. Жёлтое предупреждение: аргумент только от выключенного «производителя»

Условие **одновременно**:

1. интеграционный сценарий для **`op`** со статусом **`enabled`** или **`beta`** требует аргумент **`A`**;
2. по [gateway-operation-manual](./gateway-operation-manual.md) и таблицам **§2** этого документа типичный **производитель** значения **`A`** — операция **`opProducer`**, у которой **`availability`** ∈ { **`disabled`**, **`unsupported`** };
3. в Heap **нет** записи с ключом, который нормативно задаёт **`A`** для этого сценария (например ожидаемый **`gc_itest_*`** из **§5.8** manual или иной ключ, явно указанный в метаданных сценария).

Тогда строка сценария (или вложенная карточка аргумента) отображается с **фоном предупреждения**: тёплый **приглушённый** янтарь/песок, **без** насыщенного «кислотного» жёлтого — ориентир **`bg-amber-50`** / **`bg-amber-950/20`** (светлая/тёмная тема) или близкий токен дизайн-системы страницы; контраст текста — не ниже разумного минимума для чтения (подогнать под фактическую тему админки).

**Обязательный текст на карточке (русский):**

- какой **`op`** (производитель) «обрезан» статусом **`availability`**;
- **имя ключа Heap** (или иной **строковый ключ настроек**), который нужно задать вручную в админке (**[gateway-operation-manual](./gateway-operation-manual.md) §5.9**), чтобы выполнить сценарий **без** активации выключенного метода-«геттера».

### 9.4. Произвольные ключи настроек для обхода блокировок

Если для устранения состояния **§9.3** администратор задаёт значение в Heap под указанным ключом через форму **«ключ — значение»** (**§5.9** manual), жёлтое предупреждение для данного аргумента **снимается** при следующей загрузке страницы (или после refetch настроек), без изменения **`availability`** у **`opProducer`**.
