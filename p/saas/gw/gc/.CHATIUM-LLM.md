# Проект `p/saas/gw/gc` — GetCourse Gateway

## Что это

Chatium-приложение: gateway к API GetCourse на каркасе, изначально скопированном из `template_project` (страницы, админка, логи, тесты). **Рабочий репозиторий и SSOT по коду** — **`p/saas/gw/gc`** (`PROJECT_ROOT` в `config/routes.tsx`), а не каталог `p/template_project`.

## Норматив разработки gateway

Единый техдок слоя (маршруты `/v1/{op}`, заголовки, ошибки, маппинг на GC) и JSON-артефакты лежат в **`docs/gateway/`** приложения. Ключевые файлы:

- `docs/gateway/gateway-operation-manual.md` — SSOT разработки (§0–§13: жизненный цикл `/v1/{op}`, каталог `GET /v1/operations`, контуры, авторизация, наблюдаемость, коды ошибок; §9.3 версионирование, §12 журнал решений, §13 «История изменений»). Ключевые нормативы: таймаут к GC = **10 000 мс** (§8.1, `GW_OUTBOUND_TIMEOUT_MS`); лимит входящего тела `POST /v1/{op}` = **1 MiB** (§8.7, `GW_MAX_REQUEST_BODY_BYTES`, превышение → `INVOKE_BODY_TOO_LARGE` HTTP 413); идемпотентность и серверные ретраи в gateway **запрещены** (§8.6); единственная версия публичного API — `/v1/` (§9.3); единый SSOT каталога — **`lib/gateway/operationsCatalog.ts`** (`export const operationsCatalog = { schemaVersion, entries } as const`, в каждой записи — `op`/`httpMethod`/`contour`/`availability`/`pathTemplate`/`legacyImportAction` и live-объект `argsSchema`); генерируемые источники — `lib/gateway/gcOpHttpMapping.generated.ts` (entries+schemaVersion) и `lib/gateway/v1OpArgsSchemas.generated.ts` (схемы args), плюс ручное перекрытие `addUser`; схемы `args` — на платформенном билдере `s` из `@app/schema` (§3.1, §3.5); и `GET /v1/operations`, и `/v1/{op}` берут запись через `findOperationCatalogEntry` из того же модуля (§3.5); ответ роутов — объект `TuneHttpHeadersResponse` (`rawHttpBody`, `headers?`, `statusCode?`, §9.0); всегда один файл — один роут (§2.1); общая логика вызова GC — **`lib/gateway/handleV1OpRoute.ts`**.
- `docs/gateway/gateway-testing-strategy.md` — стратегия юнит- и интеграционного тестирования (фазы прогона, цепочки `op`, 1 rps, страница тестов в админке).
- `docs/gateway/implementation-plan.md` — пошаговый план реализации (Прототип → MVP → Прод). **Точка старта разработки**; закрытие всех чек-боксов = готовый продукт. (Прогресс по чек-боксам — в самом файле; раздел 1.1: страница тестов шаблона закрыта.)
- `docs/gateway/gc-unified-op-registry-v0.md`, `gc-op-http-mapping.json`, `gc-required-fields-by-op.json`, `new_api_schema.json`, `legacy_api_schema.json`, `extract-gc-required-fields.py` — машиночитаемые контракты.

## Публичный `/v1/{op}`

- Роуты: **`api/v1/{op}.ts`** (59 операций, каждый файл экспортирует `{op}Handler: V1GcHandler` + `{op}Route`) и **`api/v1/operations.ts`** (`GET /v1/operations`, ответ `fields[]` через `toOperationSummaries()`) — без сессии Chatium.
- Каталог: **`lib/gateway/operationsCatalog.ts`** — рукописный статический массив 59 записей (`op`/`contour`/`httpMethod`/`pathTemplate`/`availability`/`legacyImportAction`/`argsValidator (s.object)`/`argsSchema.fields[]`). Экспорты: `CATALOG_SCHEMA_VERSION=1`, `findOperationCatalogEntry`, `findOperation`, `toOperationSummaries`, типы `OperationEntry`/`OperationCatalogEntry`. Нет `*.generated` файлов — codegen-пайплайн удалён.
- Общий обработчик: **`lib/gateway/handleV1Op.ts`** (`handleV1Op(ctx, req, op, handler)` + `handleV1OpWithGcDiagnostic`): конвейер проверок → вызов per-op handler → запись raw-журналов в `finally`. НЕ импортирует реестр.
- Реестр: **`lib/gateway/v1OpHandlers.ts`** (`V1_OP_HANDLERS: Record<op, V1GcHandler>`, `findV1OpHandler`) — единый путь для роутов и тест-раннера.
- Совместимостный shim: **`lib/gateway/handleV1OpRoute.ts`** (`handleV1OpRoute` / `handleV1OpRouteWithGcDiagnostic` через реестр) — для `v1AddUserHandler.ts` и тест-раннера.
- Общий слой `lib/gateway/`: `constants.ts`, `utf8Base64.ts`, `v1IncomingPost.ts`, `v1GatewayQuery.ts`, `v1TuneResponse.ts` (CORS), `v1ErrorCatalog.ts`, `legacyGcFormBody.ts`, `legacyGcImportClient.ts`, `legacyGcExportGet.ts`, `newGcApiClient.ts`, `pathTemplate.ts`, `interpretGcV1Response.ts`, `legacyGcSemantic.ts`, `newGcSemantic.ts`.
- Заголовки: `shared/gatewayHttpHeaders.ts` (`// @shared`).
- **Наблюдаемость.** В `finally` блока `handleV1Op` пишутся две Heap-записи: `gatewayRequestLog` (входящий запрос) и `gatewayUpstreamLog` (исходящий вызов GC, если был). PII/секреты маскируются через `shared/redactRaw.ts` (`redactRawDeep`). Admin-эндпоинты: `api/admin/raw/requests/recent`, `api/admin/raw/requests/get`, `api/admin/raw/upstream/recent`, `api/admin/raw/upstream/get`, `api/admin/dashboard/gatewayCounts` (KPI за 24ч через `countBy`). События workspace (`@start/sdk`) не используются.

## Тесты и логирование

Как в шаблоне: `shared/testCatalog.ts`, `lib/tests/templateUnitSuite.ts`, `lib/tests/integrationSuite.ts`, роуты `api/tests/*`; SSR `logLevel`, `browserRemoteLogger` на аутентифицированных страницах. HTTP-вкладка: для `GET /` проверяются `window.__BOOT__` и фрагмент шапки ` / Главная` (не текст «Шаблон проекта»). Интеграция с GetCourse: кейс **`gateway_v1_addUser_live`** в `integrationSuite.ts` (см. `docs/gateway/gateway-testing-strategy.md` §1.1). См. README и `docs/LLM/` при необходимости деталей UI.

**Сьюит /v1/{op} (`POST /api/tests/v1-ops/run`)**: на вкладке HTTP страницы `/web/tests` есть отдельный тест на каждый из 59 роутов `api/v1/*`. Сьюит реализован в `lib/tests/gateway/v1OpsScenarios.ts` (фазовый порядок 1–4, цепочки producer→consumer→destructor по `gateway-testing-strategy.md` §3.4) и `lib/tests/gateway/v1OpsSuiteRunner.ts` (throttle 1 rps ожиданием по `Date.now()` без таймеров — см. `docs/architecture.md`; не `@app/jobs`), вызов **`handleV1OpRouteWithGcDiagnostic`** для поля **`gcUpstream`** (сырое тело ответа школы в JSON прогона), контекст в памяти процесса, чтение секретов школы и `gc_itest_*` Heap-ключей). UI — `pages/TestsPage.vue`, секция «Gateway /v1/{op}»: для каждой строки видны HTTP-статус, длительность, `requestId`, фаза и раскрывающийся блок с **args**, ответом **gateway** (`parsedResponse`) и **сырым ответом GetCourse** (`gcUpstream`). Список 59 op для UI генерируется в `shared/v1OpsList.generated.ts` (// @shared) скриптом `scripts/gen-gc-op-http-mapping.cjs`. В шаблоне не использовать inline-объект `{ key: value }` внутри `{{ … }}` — ведущая `{` у UGC/Vue может парситься как блок и давать пустой `ReferenceError:`; маппинг строк — через функцию в `<script setup>` (см. `v1OpsPreflightRunStatusShortLabel`). В шаблоне секции `/v1/{op}` не использовать **`?.`** (optional chaining) — в UGC выражениях шаблона это может давать пустой `ReferenceError:` после появления `row.result`; писать **`row.result && row.result.field`**.

**Префлайт `/v1/{op}` (`GET /api/tests/v1-ops/preflight`)**: статичный анализ готовности сьюита без сетевых вызовов. Возвращает состояние «уровня A» (Heap-ключи `gc_*` — manual §5.8, стратегия §1.1), список заполненных `gc_itest_*` (только имена, без значений — manual §5.7) и для каждого op — `runStatus ∈ {ready, blocked-availability, warn-heap, warn-deps}`, `dependsOn`, `heapKeys`, `missingHeapKeys`, `blockedDependencies`, `blockReason`. UI разбивает строки по фазам, для `disabled`/`unsupported` отключает кнопку `Run` и красит фон серым (стратегия §9.1), для `warn-*` — янтарным с подсказкой (§9.3). Реализация — `lib/tests/gateway/v1OpsPreflight.ts`.

**Произвольные настройки Heap (manual §5.9)**: универсальный ввод «ключ — значение» в админке (`pages/AdminPage.vue`, секция «Произвольные настройки Heap»). Эндпоинты: `POST /api/settings/save` (произвольные ключи проходят без валидации значения), `POST /api/settings/delete`, `GET /api/settings/list-arbitrary`. Используется для `gc_itest_*` (стратегия §5.8 уровень B) и обхода жёлтых предупреждений на странице тестов (стратегия §9.4). Секреты (`gc_developer_api_key`, `gc_test_school_api_key`) в этот список не попадают (manual §5.7) — они задаются в форме «Настройки GetCourse» уровнем выше.

## Настройки gateway в Heap

Имена полей `key` в таблице настроек Heap (строки вида `gc_*`) экспортируются из `shared/gatewaySettingKeys.ts` (первая строка `// @shared`); это не значения секретов. Примеры: `GC_DEVELOPER_API_KEY`, `GC_TEST_SCHOOL_API_KEY`, `GC_TEST_SCHOOL_HOST`. `lib/settings.lib.ts` импортирует их в `SETTING_KEYS`, чтобы те же строки использовались на сервере и во Vue без импорта `lib/` с клиента. Ввод трёх настроек уровня A — карточка в `pages/AdminPage.vue`; проверка хоста — `shared/gcSchoolHostValidation.ts`.

## Критические правила

- Не хардкодить базовый путь приложения: `config/routes.tsx` (`PROJECT_ROOT`, `withProjectRoot` — **утилиты шаблона** `template_project`, не платформы; платформенный примитив — `route.url()`).
- Клиентский бандл: не тянуть `lib/`, `repos/`, `tables/` из Vue — только `shared/` с `// @shared` и контракты `api/*/route`.
- Логирование в прикладном коде: через `lib/logger.lib` (`writeServerLog`), не `console.log`. Для «залогировать и выбросить» — `throwLoggedServerError`. В Heap поле `payload` пишется JSON-строкой при любом уровне логирования, если `payload` передан; в account.log / сокет / вебхук структурированный контекст — только при Debug.
