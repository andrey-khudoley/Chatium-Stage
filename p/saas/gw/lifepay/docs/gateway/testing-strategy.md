# Интеграционное и юнит-тестирование payments-gateway (LifePay)

Документ задаёт **норматив по тестам** слоя payments-gateway: как уменьшить ручной ввод в Heap за счёт цепочек вызовов API, как вести массовый прогон без лишних созданий счетов в LifePay, как убирать или маркировать следы тестов и как соотносится это с [operation-manual](./operation-manual.md) (§5.8, §9-§10, §12 «Открытые вопросы», §13 «История изменений»).

Машиночитаемый перечень обязательных полей запросов по `op`: [lp-required-fields-by-op.json](./lp-required-fields-by-op.json). Статусы операций `availability` и UI страницы тестов - §9 ниже и [operation-manual](./operation-manual.md) §2.11, §5.9.

**Актуализация документа:** 14-05-2026 (начальная версия). Заимствует структуру и нормативы из образца testing-strategy проекта `course-chatium-gc-integration-3fa7c2` _(вне этого воркспэйса)_ приложения `p/saas/gw/gc`, с заменой контуров (`new`/`legacy` → `bills_v1`/`ecom_v1`), имён ключей Heap (`gc_*` → `lp_*`), `op` (3 операции вместо 59) и формата фикстур (`tester@khudoley.pro` остаётся как канон email тестового покупателя по проекту).

## 1. Минимальный Heap и общая идея «цепочек»

### 1.1. Всегда обязательный ввод (уровень A)

До любого интеграционного вызова `/v1/{op}` в Heap должны быть заданы только (§5.8 manual):

| Ключ             | Назначение                                                                    |
| ---------------- | ----------------------------------------------------------------------------- |
| `lp_test_apikey` | API-ключ тестового магазина LifePay → заголовок `X-Lp-Apikey`                 |
| `lp_test_login`  | Логин (телефон владельца) тестового магазина LifePay → заголовок `X-Lp-Login` |

Иные ключи `lp_itest_*` считаются не обязательными по умолчанию, если тот же результат можно получить вызовом другого `op` в рамках одного прогона (см. §2).

### 1.2. Контекст прогона (память процесса, не Heap)

Интеграционный раннер держит объект **контекста сессии** (в памяти тестового процесса): `billNumber` (из ответа `createBill`), `paymentUrl`, `paymentUrlWeb`, опционально `orderNumber` (искусственно сгенерированный для каждого прогона, чтобы не конфликтовать с боевыми заказами GetCourse). Значения заполняются результатами успешных вызовов gateway и переиспользуются следующими тестами.

**Исключение для `orderNumber`.** Это не идентификатор LifePay, а внешний ключ, который gateway передаёт LifePay в поле `order.number` запроса `createBill` и который потом приходит в webhook как `order_id`. Для тестов `orderNumber` генерируется в начале каждого прогона по шаблону `itest-{ISO_timestamp}-{rand4}` (например `itest-2026-05-14T13-22-05-a1b2`), хранится в контексте прогона. Не пишется в Heap.

Правило: сначала производитель ID (`createBill` → `billNumber`), потом потребитель (`getBillStatus(billNumber)`, `cancelBill(billNumber)`). Порядок тестов в файле / сьюте явно задаётся или группируется по фикстуре `beforeAll`. Детальная фазовая модель - §3.1.

### 1.3. Статусы `availability` и допустимость интеграционного прогона

Эффективный статус каждого `op` берут из каталога gateway (поле `availability` в `GET /v1/operations` и в сборке приложения - [operation-manual](./operation-manual.md) §2.11, §3.4).

- **Интеграционные тесты**, которые выполняют реальный HTTP к `/v1/{op}` и инициируют исходящий вызов к LifePay, **разрешено** ставить в очередь прогона только для операций со статусом `enabled` или `beta`.
- Для операций `disabled` или `unsupported` **запрещено** запускать интеграционные тесты при любой конфигурации (включая отладку, ручной «форс» в UI и временные флаги в коде). Такие строки на странице тестирования не получают кнопку запуска интеграционного сценария; CI-раннер обязан завершаться ошибкой, если обнаружен интеграционный кейс для `disabled` / `unsupported`.

Типовые компенсирующие методы для контура `bills_v1`. Если в колонке «альтернатива» указан способ получения значения из ответа предыдущего `op`, ключ `lp_itest_*` для этого поля в Heap **не нужен** при автоматизированном прогоне.

| ------------------------------ | ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `billNumber`                   | `getBillStatus`, `cancelBill`                                            | Один раз вызвать `createBill` с тестовым `orderNumber` и минимальной суммой; из тела ответа взять поле `number` (LifePay возвращает `paymentUrl`, `paymentUrlWeb`, `number`). Хранить в контексте сессии.         | Не вызывать `createBill` перед каждым тестом без причины - см. §3.3.                                                                                                                           |
| `paymentUrl`, `paymentUrlWeb`  | Сценарии «дочитать QR / открыть deeplink» (не gateway-кейсы)             | Из ответа `createBill`.                                                                                                                                                                                           | В юнитах gateway не используется; нужно только в end-to-end тестах client-приложения с QR-страницей школы.                                                                                     |
| `orderNumber`                  | Тело `createBill` (`args.orderNumber` маппится в `order.number` LifePay) | Сгенерировать в `beforeAll` по шаблону `itest-{ISO_timestamp}-{rand4}` (§1.2).                                                                                                                                    | Уникальность в рамках магазина важна, чтобы не конфликтовать с боевыми заказами; для одного прогона достаточно одного `orderNumber`, если оплата не нужна (только проверка создания / отмены). |
| `amount`                       | Тело `createBill` (`args.amount`)                                        | Литерал `1.00` (одна копейка - LifePay требует копейки или целые рубли с двумя знаками, уточняется на тестовом счёте).                                                                                            | Литерал в коде тестов, не Heap.                                                                                                                                                                |
| `customerEmail`                | Тело `createBill` (`args.customerEmail`)                                 | Хардкод `tester@khudoley.pro`.                                                                                                                                                                                    | Канон email тестового покупателя по проекту; не генерировать, не класть в Heap.                                                                                                                |
| `callbackUrl`                  | Тело `createBill` (`args.callbackUrl`)                                   | Литерал `https://example.invalid/webhook` для юнит-сьюита (webhook от LifePay не понадобится, так как тесты gateway не ждут результата оплаты); для end-to-end тестов client-приложения - реальный URL приёмника. | Не Heap. Не реальный URL в gateway-юнитах.                                                                                                                                                     |
| `tid` (LifePay transaction ID) | Только в webhook от LifePay (приходит на client, не на gateway)          | В gateway-тестах **не используется**: gateway никогда не видит `tid`, потому что webhook идёт на client (§6 manual).                                                                                              | Для тестов client-приложения - отдельный сьют вне этого норматива.                                                                                                                             |

**Литералы для тестового счёта.**

| Поле                 | Литерал                                                                                                                                            | Источник                                                     |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| `args.amount`        | `1.00`                                                                                                                                             | Минимум для реального СБП-теста (1 ₽).                       |
| `args.customerEmail` | `tester@khudoley.pro`                                                                                                                              | Канон проекта.                                               |

| `args.description`   | `"itest createBill"` или подобное.                                                                                                                 | Чтобы человек-админ LifePay сразу видел тестовое назначение. |
| `args.orderNumber`   | Контекст прогона по шаблону §1.2.                                                                                                                  | -                                                            |
| `args.callbackUrl`   | `https://example.invalid/webhook` для юнитов; реальный URL приёмника для end-to-end.                                                               | -                                                            |

## 3. Порядок прогона: фазы, цепочки вызовов, реестр зависимостей

Интеграционный сценарий - **не** набор независимых вызовов `/v1/{op}`. Для `getBillStatus` или `cancelBill` сначала нужно вызвать `createBill`, получить `billNumber`, положить его в контекст и потом использовать. Машиночитаемый перечень обязательных полей по каждому `op`: [lp-required-fields-by-op.json](./lp-required-fields-by-op.json).

### 3.1. Четыре фазы одного интеграционного сьюита

1. **Фаза 1. Только то, что задаёт администратор (Heap), без «производителей» в API.** Сюда относятся вызовы, которым для `args` достаточно уровня A (§1.1) и литералов из §2. В payments-gateway v0 в этой фазе нет операций без предшественника, кроме самой `createBill` (которая сама является производителем `billNumber`).
2. **Фаза 2. Производители идентификаторов и структур.** В payments-gateway v0 это ровно одна операция: `createBill` → `billNumber` в контекст. В будущем сюда могут попасть операции вроде `getServiceInfo` (если появится в каталоге).
3. **Фаза 3. Потребители.** `getBillStatus(billNumber)`, `cancelBill(billNumber)` получают `billNumber` из контекста после Фазы 2. Один и тот же `billNumber` в рамках сьюита переиспользуется по правилам §3.3.
4. **Фаза 4. Деструкторы и приведение к безопасному состоянию.** В `afterAll` / `finally`: вызов `cancelBill(billNumber)` для каждого `billNumber`, который не был оплачен и остаётся в статусе `pending`. Если счёт уже оплачен (в случае, когда тестировщик намеренно дошёл до оплаты QR в банковском приложении) - запись в лог raннера с пометкой «остался оплаченный тестовый счёт на сумму 1 ₽, ручная сверка». Подробно §3.5.

**Правило запуска одного теста.** Если в [lp-required-fields-by-op.json](./lp-required-fields-by-op.json) у `op` есть обязательное поле (`billNumber` для `getBillStatus`, `cancelBill`), раннер обязан либо выполнить предыдущий шаг цепочки и заполнить контекст, либо явно пропустить кейс с диагностикой «не собран контекст».

### 3.2. Обязательная модель «предыдущий ответ → следующий запрос»

- **Интеграционный кейс** описывают как упорядоченный список шагов `(op₁, op₂, ...)`; для шага `opₖ` каждый аргумент либо литерал сьюита (разрешённые константы: `tester@khudoley.pro`, `1.00`, и т.п.), либо `context[field]`, заполненный при `opᵢ`, `i < k`.
- **Запрещено** подставлять в `opₖ` «магические» `billNumber`, не полученные в этом же прогоне или из Heap по §2 / §5.8.
- Для **асинхронных** сценариев (опрос статуса счёта в случае настоящей оплаты в банке) между шагами выдерживают §6 (throttle).

### 3.3. Массовый прогон: один счёт на сьют

Частный случай Фаз 2-3 для `billNumber`:

1. **Один раз** в `beforeAll` (Фаза 2) вызвать `createBill` с тестовым `orderNumber` и минимальной суммой; сохранить `billNumber` в контексте.
2. Все тесты Фазы 3, которым нужен `billNumber` (`getBillStatus`, опциональный пробег `cancelBill` в середине, если проверяют переход состояния), используют **этот же** идентификатор.
3. **Не** вызывать `createBill` перед каждым кейсом без причины. Изоляция только при необратимой порче данных (например намеренно проверяют двойную отмену - тогда новый счёт).

### 3.4. Реестр зависимостей по `op` (контур `bills_v1`)

Полный перечень `op` v0 из [lp-required-fields-by-op.json](./lp-required-fields-by-op.json):

| `op`            | Что обязательно в `args`                                                                                                                         | Откуда берут значения (фаза / предшественник)                                                                                                            |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `createBill`    | `amount` (число > 0), `customerEmail` (строка), `orderNumber` (строка), `callbackUrl` (строка-URL). Опционально: `customerPhone`, `description`. | Фаза 2. Литералы по §2; `orderNumber` из контекста (генерируется в `beforeAll`). После вызова: `billNumber`, `paymentUrl`, `paymentUrlWeb` → в контекст. |
| `getBillStatus` | `billNumber` (строка).                                                                                                                           | Фаза 3. `billNumber` из контекста после `createBill`.                                                                                                    |
| `cancelBill`    | `billNumber` (строка).                                                                                                                           | Фаза 3 или Фаза 4 (деструктор). `billNumber` из контекста.                                                                                               |

**Замечание про `method`.** Поле LifePay `method` всегда `"sbp"` в контуре `bills_v1` (см. ADR 0001 и §5.2 manual); gateway выставляет его автоматически на этапе сборки исходящего тела, в `args` оно не передаётся. Тесты на ситуацию «другой `method`» в v1 не делаются.

### 3.5. Деструкторы и «чистка» по типам сущностей

| Сущность                                                                           | Типовой деструктор (Фаза 4)                                                                                                      | Если `op` недоступен                                                                              |
| ---------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| **Счёт (`billNumber`)** в статусе `pending`                                        | `cancelBill(billNumber)` в `afterAll`.                                                                                           | §5: предупреждение, учёт артефакта.                                                               |
| **Счёт оплаченный** (если тестировщик дошёл до реальной оплаты в банке)            | Отмена через API в v1 невозможна (LifePay не возвращает деньги через API); возврат вручную в `my.life-pay.ru` → касса → возврат. | §5: запись в лог раннера, пометка «остался оплаченный тестовый счёт на сумму X ₽, ручная сверка». |
| **Webhook-следы в client-приложении** (если тесты ходили end-to-end через webhook) | Зачистка таблиц `payment-events` / `processed-webhooks` client-приложения.                                                       | Не относится к payments-gateway, делается в сьюите client.                                        |

### 3.6. Падение теста и всё равно teardown

Шаги Фазы 4 выполняют в `afterAll` / `finally`, чтобы при падении середины сьюита **попытаться** отменить созданные счета. Если teardown упал - запись в лог по §5 (тип сущности, `op`, `billNumber`).

Аналогичный принцип «один раз создать → много раз использовать → завершить деструктором» - для тестовых счетов; в payments-gateway v0 нет более долгоживущих сущностей.

## 4. Мутации и минимизация мусора

- В контуре `bills_v1` каждая операция несёт некоторый side-effect: `createBill` создаёт счёт в LifePay, `cancelBill` отменяет, `getBillStatus` читает. Чистых GET без побочных эффектов в v0 каталога - только `getBillStatus`.
- **Предпочитать** `getBillStatus` для проверочных кейсов; **группировать** `createBill` в одном `beforeAll` сьюита и переиспользовать `billNumber` всеми последующими тестами.
- Тестовая сумма всегда минимальная (1 ₽), описание `"itest createBill"`, email `tester@khudoley.pro` - чтобы человек-админ LifePay в `my.life-pay.ru` сразу видел в журнале интеграционные следы.

## 5. Обязательная политика очистки и учёта артефактов

После любого интеграционного теста, который создал или изменил данные в LifePay, действует лестница ответственности:

1. **Откат или удаление.** Если через доступные `op` gateway можно отменить или вернуть состояние (счёт в `pending` → `cancelBill`) - сделать это в `afterEach` / `afterAll` того же сьюта или в общем teardown.
2. **Пометка как тестовых данных.** Если отменить нельзя (счёт уже оплачен) - создавать сущности с **явной пометкой**: `customerEmail` = `tester@khudoley.pro`, `description` начинается с `"itest "`, `orderNumber` начинается с `"itest-"`.
3. **Невозможно ни удалить, ни пометить.** В конце прогона вывести **явное предупреждение** в лог раннера (и при падении CI - в артефакт отчёта): тип сущности, `op`, известный `billNumber` из ответа LifePay, текст «артефакт остался в системе LifePay».

Запрещено молча оставлять созданные объекты без одной из трёх мер выше.

То же относится к **оплаченным счетам**: после успешной оплаты тестового счёта в 1 ₽ запись в лог раннера с суммой и `tid` (из ответа `getBillStatus`); ручной возврат денег - вне рамок прогона, через `my.life-pay.ru` → касса → возврат.

## 6. Лимит частоты запросов к LifePay

Между двумя последовательными вызовами, каждый из которых приводит к одному исходящему HTTP от gateway к LifePay, интервал **≥ 1000 ms** (1 rps на процесс). Параллельные воркеры могут суммировать нагрузку - в CI предпочтительно последовательное выполнение интеграционных тестов или глобальный семафор.

LifePay в публичной документации не объявляет точные лимиты по rate-limit для контура `bills_v1`; ограничение 1 rps на тестах выбрано из общей практики работы с провайдерами и из соображений «не вызывать раздражения у поддержки LifePay при массовом прогоне». Если в проекте появится подтверждение более высокого лимита, значение пересматривается с обновлением этого раздела.

Подробнее об общей архитектуре ответов и ошибок - [operation-manual](./operation-manual.md) §9-§10. Норматив по непрерывному серверному логированию - §7.2.1 manual.

## 7. Юнит-тесты

Юнит-прогон **без** реальных вызовов `@app/request` к LifePay и **без** обращения к Heap настроек тестового магазина. Он **не заменяет** интеграционный (§3).

### 7.1. Назначение

Проверить чистую логику payments-gateway при фиксированных входах: построение исходящего HTTP (метод, путь, query, заголовки, тело), разбор и нормализация ответов, маппинг кодов ошибок к контракту [operation-manual](./operation-manual.md) (§9-§10).

### 7.2. Сценарии по слоям (каталог кейсов)

Каждый юнит-кейс - **Arrange → Act → Assert**; в Arrange задают `ctx` (или его мок), `op`, `args`, при необходимости заглушку ответа «как будто от LifePay».

| Слой / тема                                                            | Что проверять                                                                                                                                                                                                                                                                                                                                                                                                                                                                            | Типовой Arrange                                                                                                                             |
| ---------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Маршрутизация `op` → HTTP                                              | Совпадение с [lp-op-http-mapping.json](./lp-op-http-mapping.json): путь, метод.                                                                                                                                                                                                                                                                                                                                                                                                          | Одна пара `(op, contour)`.                                                                                                                  |
| Сериализация тела / query                                              | Обязательные поля из [lp-required-fields-by-op.json](./lp-required-fields-by-op.json) попадают в исходящий запрос; `apikey` / `login` подставляются из заголовков, не из `args`; для `bills_v1.createBill` - в теле JSON, для `bills_v1.getBillStatus` - в query.                                                                                                                                                                                                                        | `args` с минимальным валидным набором; заголовки `X-Lp-Apikey` / `X-Lp-Login` в моке `ctx.req.headers`.                                     |

### 7.3. Связь с интеграционными фазами

В юнит-тестах **нет** фаз §3.1 в смысле живого LifePay: цепочка `createBill` → `billNumber` → `getBillStatus` не выполняется. Вместо этого для слоя маппера задают пары `(входные args, ожидаемый фрагмент HTTP)`; для слоя разбора ответа - пары `(сырой ответ LifePay, ожидаемая структура для клиента)`. Таблица §3.4 используется как чек-лист: для каждого `op` должен существовать хотя бы один юнит на сборку запроса и при наличии нетривиального разбора - на разбор ответа.

## 9. Страница тестирования в админке: фильтры, переключатели, визуальные состояния

Норматив для Vue-страницы тестирования gateway в приложении Chatium (`pages/` + `fetch` к `api/*`, данные настроек через существующие эндпоинты, роль `Admin` - [operation-manual](./operation-manual.md) §0, §5.9). Цель - не допустить «тихого» нарушения правил §1.3 и §6 (throttle).

### 9.1. Интеграционные тесты: только `enabled` и `beta`

- Каждая строка сценария, привязанная к `op`, отображает текущий `availability` из каталога (тот же источник, что раннер).
- **Запуск** интеграционного сценария активен **только** если `availability` ∈ { `enabled`, `beta` }.
- Для `disabled` и `unsupported` элемент управления запуском **отключён** (disabled), фон строки - нейтральный серый (`bg-slate-100` в светлой теме, `bg-slate-800/60` в тёмной), без анимаций «пульс».

### 9.2. Юнит-тесты: глобальный переключатель «форс всех»

- На странице размещается переключатель **«Запускать юнит-тесты для всех операций»**. По умолчанию выключен (`false`).
- **Выключен (по умолчанию).** В автоматический юнит-прогон попадают только тесты, отнесённые к `op` со статусом `enabled` или `beta`.
- **Включён.** Разрешается запуск полного набора юнит-тестов репозитория, включая кейсы для операций `disabled` / `unsupported` (например проверка, что маппер возвращает `INVOKE_OP_DISABLED` / `INVOKE_OP_UNSUPPORTED_BY_LP` без сети). Переключатель не влияет на интеграционные тесты (§9.1).

### 9.3. Жёлтое предупреждение: аргумент только от выключенного «производителя»

В payments-gateway v0 этот сценарий маловероятен (только три `op`, единственный «производитель» - `createBill`), но норматив сохраняется на будущее: если интеграционный сценарий для `op` со статусом `enabled` / `beta` требует аргумент `A`, типичный производитель которого имеет статус `disabled` / `unsupported`, и в Heap нет ключа, нормативно задающего `A`, - строка отображается с фоном предупреждения (тёплый приглушённый янтарь, `bg-amber-50` / `bg-amber-950/20`).

Текст карточки (русский): какой `op`-«производитель» обрезан статусом `availability`; имя ключа Heap, который нужно задать вручную в админке (§5.9 manual).

### 9.4. Произвольные ключи настроек для обхода блокировок

Если для устранения состояния §9.3 администратор задаёт значение в Heap через форму «ключ - значение» (§5.9 manual), жёлтое предупреждение для данного аргумента снимается при следующей загрузке страницы.

### 9.5. Вкладка «Тест запроса»: ручной прогон через UI

**Расположение и доступ.** Четвёртая вкладка `pages/TestsPage.vue` (рядом с «Юнит / Интеграция / HTTP»), доступная только при роли `Admin` (`requireRealUser(ctx).is('Admin')` в `web/tests/index.tsx`). Для не-Admin вкладка не рендерится. Компонент - `components/RequestTestTab.vue`. Норматив в [operation-manual](./operation-manual.md) §11.0.

**Источник списка `op`.** Каталог из `lib/gateway/operationsCatalog.ts` приходит на страницу через SSR-проп `operationsCatalog` (типизация через `shared/operationsCatalogShared.ts`); параллельно тот же каталог доступен через `GET /v1/operations`. Для `op` со статусом `availability != 'enabled'` (§2.11 manual) кнопка «Отправить» **заблокирована**, рядом - жёлтое предупреждение «Метод недоступен (availability = ...). Отправка заблокирована».

**Источник секретов.** `X-Lp-Apikey` и `X-Lp-Login` по умолчанию вводятся **вручную** в полях формы (без автоподстановки при открытии вкладки). Рядом с полями есть кнопка **«Использовать тестовые»** - по явному клику подгружает значения `lp_test_apikey` / `lp_test_login` из Heap (§5.5 manual). Это удобный shortcut для регулярного тестирования, не нарушающий принцип «осознанного выбора»: тестер каждый раз явно выбирает источник секретов (кнопка vs ручной ввод). Автоматические интеграционные тесты §3 продолжают читать пару из Heap напрямую через серверный код, без UI.

**Транспорт.** Реальный браузерный `fetch` напрямую на `/${PROJECT_ROOT}/api/v1/{op}` с методом из каталога. Без `credentials: 'include'`, заголовки `X-Lp-Apikey`/`X-Lp-Login` + `Content-Type: application/json` (для POST). Гарантируется тот же путь обработки, что для боевого вызова из серверного кода клиентского приложения (виджет, приёмник, тестовая страница): цепочка §2 manual, классификация транспорта/семантики, ответ по §9.1.

**Валидация на лету.** Клиент:

- `X-Lp-Apikey` непустой;
- `X-Lp-Login` соответствует `^7[0-9]{10}$` (§2.5 manual);
- обязательные поля `args` непустые; поля `type: 'number'` - валидное число.
  Сервер всё равно валидирует ещё раз через `argsValidator.safeParse` каталога (§3.5 manual) - клиентская валидация не заменяет серверную.

**Вывод результата.**

- Статус «УСПЕХ» / «ПРОВАЛ» (по `body.ok` если есть, иначе по HTTP-статусу 2xx).
- Сырой JSON запроса: `{ method, url, headers, body | query }`. Заголовки `X-Lp-*` отображаются в открытом виде (страница Admin-only; маскирование намеренно отсутствует).
- Сырой JSON ответа: `{ statusCode, headers, body }`. Поле `body` - JSON-контракт §9.1.
- Оба snapshot - в `<pre>` с отступами `JSON.stringify(..., null, 2)`.

**Граница с интеграционными тестами.** Ручной канал **не входит** в автоматический прогон страницы `/web/tests` (кнопки «Запустить набор» и «Полный прогон» не действуют на вкладке «Тест запроса»). Это two-way street: интеграция (фазы §3) - для регулярного автомата на тестовом магазине LifePay с teardown; ручной канал - для отладки конкретного случая, воспроизведения бага, ручного создания счёта при разработке.

**Сценарии использования:**

- Проверка нового `op` после добавления в каталог - отправить минимальный запрос, увидеть путь до LifePay и обратно.
- Воспроизведение бага из боевого окружения - подставить те же `args`, что были у пользователя, увидеть, какой ответ LifePay приводит к конкретной ошибке.
- Демонстрация API внешнему интегратору - живой пример, как должен выглядеть запрос и ответ.
