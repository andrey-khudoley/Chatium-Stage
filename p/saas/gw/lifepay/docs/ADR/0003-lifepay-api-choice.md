# ADR 0001. Выбор API LifePay для платёжной страницы

## Статус

**Принято 13.05.2026.** Основной путь интеграции — вариант B (старый API `POST /v1/bill` через `my.life-pay.ru`). Запасной путь — вариант A (ECOM через `home.life-pay.ru`), включается при выгодных условиях по итогам ответа инженеров LifePay (заявка от 12.05.2026 22:11). Ответ инженеров не блокирует разработку: при невыгодных условиях остаёмся на B.

## Контекст

LifePay предлагает две независимые поверхности API для приёма онлайн-платежей. У ИП Яковлева (клиент) подключена только облачная фискализация в `my.life-pay.ru`. Эквайринг в `home.life-pay.ru` не подключён, требует отдельной заявки.

Выбор поверхности API определяет: способы оплаты, авторизацию, формат webhook, фискализационный поток, UX страницы оплаты в GetCourse, стоимость для клиента (комиссии). Решение нужно принять до старта разработки (после получения договора, аванса и доступа к GetCourse).

Подробный ландшафт в [cabinets](../../../../../units/aayakovleva/project-docs/knowledge/lifepay/cabinets.md).

## Варианты

### Вариант A. ECOM API через `home.life-pay.ru`

`POST /v1/auth` → JWT (TTL 3 ч.) → `POST /v1/invoices/` → редирект покупателя на `form_link`.

Плюсы:

единый API для всех способов (карты, СБП, Mir Pay, Yandex Pay в одном invoice), хостед-форма LifePay снимает с интегратора PCI-нагрузку, актуальная документация (`docs.life-pos.ru`), официально рекомендуемый путь LifePay для новых интеграций.

Минусы:

требует отдельной заявки на подключение эквайринга (срок неизвестен, может затянуть проект), комиссии по СБП и картам через эквайринг не подтверждены, могут быть выше 0.4%, фискализация требует дополнительной настройки (либо `send_receipt_through` в invoice, либо Cloud Print API после webhook), webhook ECOM требует уточнения формата после получения доступа в кабинет.

### Вариант B. Старый API `POST /v1/bill` через `my.life-pay.ru`

`POST /v1/bill` с `apikey + login + method: "sbp"` → ответ с `paymentUrl` (qr.nspk.ru) и `paymentUrlWeb` (web.qr.nspk.ru).

Плюсы:

работает прямо сейчас без подключения эквайринга, комиссия по СБП 0.4% (та, что Николай согласовал на созвоне 08.05), автофискализация уже настроена на стороне `my.life-pay.ru` (нужна только галочка в «Фискализация → Сотрудники»), QR-код отображается прямо на странице GetCourse, на мобильном открывается deeplink в банковское приложение, формат webhook документирован (`apidoc.life-pay.ru/notification/index`).

Минусы:

только СБП, нет карт и брендовых кошельков (но в скоупе клиента карты идут через GC и не требуются от LifePay), `apikey` в заголовке каждого запроса (менее современный паттерн, чем JWT), автофискализация ограничена одной товарной позицией (в скоупе подходит, но при расширении на сложные продукты потребует Cloud Print API), `apidoc.life-pay.ru` менее актуально визуально, чем новая документация ECOM, хотя API работает.

## Критерии выбора

Стоимость для клиента (комиссия, доход с оборота школы). Время до старта разработки (зависимость от инженеров LifePay). Покрытие способов оплаты, нужных в скоупе (по факту достаточно СБП, остальное идёт через GC и Lava Top). Совместимость с действующей облачной фискализацией. Простота сопровождения.

## Решение

**Принято: вариант B как основной, вариант A как fallback.**

Обоснование:

всё необходимое для B уже подключено у клиента (`my.life-pay.ru`, облачная фискализация, `apikey + login` доступны прямо сейчас). Разработка не зависит от внешних сроков. Комиссия 0.4% по СБП совпадает с тем, что Николай согласовал на созвоне 08.05. Покрытия только СБП достаточно: карты и прочее идут через GC и Lava Top, что зафиксировано в утверждённом скоупе. Ограничение автофискализации одной позицией под текущий продукт (один курс = один чек) не блокирующее.

Вариант A остаётся в проекте как fallback. Если инженеры LifePay подтвердят, что эквайринг подключается быстро и комиссия по СБП не выше 0.4%, B можно заменить на A в виде second-iteration. Решение о замене оформляется отдельным ADR (0002), не пересматривает 0001.

## Последствия

Закреплено в архитектуре:

[payment-scheme](../../../../../units/aayakovleva/sbp-client/docs/architecture/payment-scheme.md) описывает бизнес-логику виджета (раскладка способов оплаты), [operation-manual](../gateway/operation-manual.md) - технический норматив payments-gateway-приложения `p/saas/gw/payments/lifepay` под контур `bills_v1`. Главный файл проекта [Бриф проекта](../../../../../units/aayakovleva/project-docs/brief.md) фиксирует решение в реестре ADR и в журнале.

Что делает разработка:

Payments-gateway (`p/saas/gw/lifepay`) публикует три операции контура `bills_v1`: `createBill` (создание счёта через `POST /v1/bill` с `method: "sbp"`), `getBillStatus` (проверка статуса), `cancelBill` (отмена). Виджет-бандл на странице оплаты школы обращается не к gateway, а к промежуточной клиентской панели `p/units/aayakovleva/sbp-client` (`POST /api/lp/invoke`); панель хранит секреты магазина, делает прямой HTTP-запрос к gateway с заголовками `X-Lp-Apikey` / `X-Lp-Login`, пишет журнал запросов и показывает дашборд - см. [operation-manual §12.1](../gateway/operation-manual.md), [implementation-plan §1.8](../gateway/implementation-plan.md). На десктопе виджет рисует QR-код, на мобильном делает редирект на СБП-deeplink. Webhook от LifePay принимает та же клиентская панель (`POST /web/webhook?token=...`); аутентификация - токен в query (LifePay подпись webhook не публикует, MD5 не реализуется - решение 15-05-2026), дедупликация по `number` LifePay; в MVP - обновление статуса заказа в GetCourse через смежный GC-gateway. Отдельных SDK / widget / webhook-приложений нет. Чек уходит автоматически из `my.life-pay.ru` при включённой галочке в «Фискализация → Сотрудники».

Что не делает:

JWT-авторизация, ECOM `/invoices/`, хостед-форма редиректа, Cloud Print API. Эти материалы остаются в базе знаний как fallback под вариант A.

Параллельный трек (не блокер):

Ждём ответ инженеров LifePay по условиям эквайринга. После получения данных принимается отдельное решение, нужен ли переезд на вариант A. Если да - оформляется ADR 0002 со ссылкой на этот и активацией контура `ecom_v1` в [operation-manual](../gateway/operation-manual.md) §4.4.
