## 1. Общая идея проекта

Проект представляет собой панель управления и систему аналитики для партнёрской программы.

Партнёрская программа предназначена для:

*   управления партнёрами;
*   трекинга полного пути клиента;
*   учёта стандартных и нестандартных событий;
*   аналитики регистраций, заказов и оплат.

---

## 2. Пользователи, роли и доступы

### 2.1 Авторизация и базовые роли

*   Любой пользователь может авторизоваться через веб-интерфейс.
*   После авторизации создаётся пользователь с ролью **member**.

### 2.2 Проекты

*   Пользователь с ролью **member** может создать проект.
*   Проект служит:

    *   центром управления функциями;
    *   механизмом изоляции доступов между пользователями.

### 2.3 Роли внутри проекта

*   Пользователь, создавший проект, получает роль **project-owner**.
*   Пользователь с ролью **admin** в системе имеет полный доступ к любому проекту (наравне с project-owner).
*   Пользователь может быть добавлен в проект с ролью **project-member**.

### 2.4 Управление участниками

*   **project-owner** может:

    *   сформировать ссылку-приглашение;
    *   передать права владельца любому project-member;
    *   удалить проект;
    *   не может выйти из проекта (может только удалить проект).

*   **project-member** может:

    *   выйти из проекта в любое время.

### 2.5 Механика приглашений

*   project-owner формирует ссылку-приглашение.
*   Неавторизованный пользователь:

    *   при переходе по ссылке попадает на страницу авторизации;
    *   после авторизации возвращается на страницу вступления через параметр `?return`.
*   На странице вступления пользователь видит:

    *   приглашение;
    *   кнопки **[Вступить]** и **[Отказаться]**.
*   При нажатии **[Вступить]** пользователь добавляется в проект с ролью **project-member**.

### 2.6 Удаление проекта

*   Перед удалением проекта показываются три последовательных предупреждения:

    1.  «Удалить проект?»
    2.  «Вы уверены?»
    3.  «Вы АБСОЛЮТНО уверены?»

## 3. Разделы внутри проекта

### A. Боты

Раздел для управления Telegram-ботами партнёрской программы.

#### Функциональность:

*   Добавление одного или нескольких ботов.
*   Боты могут дублировать функционал друг друга.
*   При добавлении бота:

    *   регистрируется Telegram-вебхук;
    *   вебхук уникален для каждого бота;
    *   учитываются особенности роутинга Chatium (адресация через `~` и `/:id`, @002-routing).

#### Апдейты:

*   В таблице ботов есть кнопка перехода на страницу апдейтов.
*   Отображаются **последние 25 входящих апдейтов**.
*   Апдейты сохраняются даже после перезагрузки страницы.
*   Реализация:

    *   отдельная heap-таблица для хранения хуков;
    *   обработчик входящих вебхуков, обновляющий таблицу.

---

### B. Обратные вызовы

Раздел для создания **нестандартных событий**, которые учитываются в реферальной системе.

---

### C. Партнёры

*   Список всех партнёров, участвующих в программе.
*   Возможность перейти в профиль конкретного партнёра.

---

### D. Рефералы

Раздел со списком всех пользователей, для которых было зафиксировано хотя бы одно событие.

#### Фильтры:

*   по дате;
*   по партнёру;
*   по количеству заказов;
*   по количеству оплат;
*   по продукту (ручной ввод названия);
*   по событиям (множественный выбор).

Дополнительно:

*   фильтр по партнёру включается автоматически при переходе из профиля партнёра;
*   фильтр по событиям реализован в виде выпадающего списка с множественным выбором.

#### Поля таблицы:

*   время регистрации;
*   имя;
*   `tg_id`;
*   `gc_id`;
*   количество заказов;
*   сумма заказов;
*   сумма оплат;
*   кнопка открытия модального окна со списком нестандартных событий.

Каждый столбец поддерживает сортировку `asc/desc`.

---

### E. Доступные ссылки

Раздел управления страницами, на которые партнёры могут вести трафик.

#### Функциональность:

*   список всех доступных страниц;
*   кнопка добавления новой ссылки;
*   таблица со сводной статистикой:

    *   количество лидов;
    *   количество заказов;
    *   сумма оплат;
*   ссылка на подробную аналитику (аналог страницы рефералов по партнёру).

#### Добавление ссылки:

Модальное окно содержит:

*   название;
*   URL;
*   формат обратного вызова (POST / GET);
*   имя параметра:

    *   GET: параметр URL;
    *   POST: параметр в body или headers.

Дополнительно:

*   в URL можно использовать `{key}`, который заменяется уникальным ключом пользователя;
*   после добавления показывается URL вебхука для подтверждения регистрации;
*   вебхук размещается:

    *   в настройках формы подписки;
    *   или в настройках Telegram-бота;
*   после добавления страницы партнёры сразу могут создавать ссылки.

---

### F. Настройки проекта

Основные параметры партнёрской программы:

*   название проекта;
*   признак нового клиента (true / false):

    *   если true – учитываются только новые клиенты;
    *   реализуется через heap-таблицу пользователей;
    *   таблица заполняется загрузкой CSV;
    *   поле загрузки CSV появляется только при `true`;
*   все изменения применяются мгновенно, без кнопки сохранения;
*   выбор продуктов:

    *   если заданы, фильтр продуктов на странице рефералов блокируется;
*   время закрепления реферала за партнёром (в днях):

    *   при `false` – закрепление навсегда;
*   уровень логирования: `info`, `warn`, `error`, `disable`.

## 4. Логика работы бота

### 4.1 Команда `/start`

При отправке `/start` бот отвечает сообщением:

*   приветствие с полным именем из Telegram;
*   статус регистрации в партнёрской программе (с указанием названия проекта, если задано);
*   показатели:

    *   количество регистраций;
    *   количество заказов;
    *   количество оплаченных заказов;
    *   сумма оплат;
    *   сумма партнёрских начислений;
    *   сумма к выводу;
*   кнопки:

    *   **Управлять ссылками**
    *   **Просмотр детализации**
    *   **Запросить вывод средств**

---

### 4.2 Управление ссылками (мини-приложение)

*   список всех созданных пользователем ссылок;
*   если ссылок нет – соответствующее сообщение;
*   фиксированная кнопка **[Создать новую ссылку]**.

#### Создание новой ссылки:

*   поля:

    *   название;
    *   выбор страницы (одна);
*   если страниц нет:

    *   создание недоступно;
    *   показывается предупреждение;
*   после создания:

    *   страница подтверждения;
    *   кнопка возврата к списку.

#### Просмотр ссылок:

*   название;
*   URL;
*   кнопка просмотра статистики переходов;
*   детализация аналогична разделу рефералов в админке.

---

### 4.3 Просмотр детализации (мини-приложение)

*   статистика по каждой ссылке:

    *   регистрации;
    *   заказы;
    *   оплаты;
*   кнопка перехода к подробной таблице лидов и событий.

---

### 4.4 Запрос вывода средств

*   на текущем этапе не реализуется.

## 5. Трекинг и аналитика пути клиента

### 5.1 Переход по партнёрской ссылке

*   партнёр размещает ссылку;
*   пользователь переходит по ссылке;
*   сервер:

    *   определяет fingerprint;
    *   сохраняет компоненты fingerprint раздельно;
    *   генерирует уникальный ключ (до 8 url-совместимых символов);
*   данные сохраняются в heap-таблице визитов;
*   пользователь перенаправляется на целевой URL с подстановкой ключа.

---

### 5.2 Регистрация во внешней системе

*   пользователь совершает действие (форма, бот);
*   внешняя система отправляет вебхук;
*   вебхук содержит уникальный ключ;
*   запись визита дополняется событием регистрации.

---

### 5.3 Заказы и оплаты

#### Заказ:

*   внешний сервис отправляет вебхук;
*   данные:

    *   номер заказа;
    *   название продукта;
    *   сумма заказа;
    *   уникальный ключ.

#### Оплата:

*   отправляется вебхук;
*   данные:

    *   номер заказа;
    *   название продукта;
    *   сумма заказа;
    *   сумма оплаты;
    *   уникальный ключ.

---

### 5.4 Нестандартные события

*   могут приходить через GET или POST;
*   GET: ключ передаётся в параметре `key`;
*   POST: ключ передаётся в `body` с ключом `key`.
