# План разработки

## 0. Контекст и цель

Нужно разработать веб-приложение (админ-панель) и Telegram-часть (бот + мини-приложение), которые вместе реализуют партнёрскую программу с трекингом пути клиента:

*   партнёр привлекает трафик по уникальным партнёрским ссылкам
*   система фиксирует клики и связывает их с последующими событиями во внешней системе по ключу
*   фиксируются стандартные события: Регистрация, Создание заказа, Оплата
*   поддерживаются нестандартные события (custom callbacks)
*   есть аналитика по партнёрам, рефералам, ссылкам, заказам и оплатам

Технологический контекст: 
проект реализуется в Chatium (Heap-таблицы, роутинг, серверный TS в V8, клиент на Vue).

## 1. Термины

*   **Система**: разрабатываемое приложение (панель + API + боты).
*   **Проект**: то же самое, что Система.
*   **Кампания**: контейнер данных и доступов внутри системы.
*   **Участник кампании**: пользователь с доступом к кампании.
*   **Партнёр**: Telegram-пользователь, который вступил в партнёрскую программу через бота и создаёт партнёрские ссылки.
*   **Реферал**: конечный пользователь (лид/клиент), для которого зафиксирован клик и хотя бы одно событие.
*   **Событие**: стандартное или нестандартное уведомление о действиях реферала.
*   **Реферальный идентификатор (ref)**: уникальный идентификатор реферала в рамках трекинга (не длиннее 8 URL-safe символов).

## 2. Роли и доступы

### 2.1 Роли системы

*   `admin`: системная роль. Имеет полный доступ ко всем кампаниим.
*   `member`: роль по умолчанию после авторизации. Может создавать кампанияы.

### 2.2 Роли в рамках кампании

*   `campaign-owner`: владелец кампании.
*   `campaign-member`: участник кампании.

### 2.3 Матрица прав (минимум)

| Действие | admin | campaign-owner | campaign-member |
| :--- | :---: | :---: | :---: |
| Создать кампания | ✅ | ✅ | ✅ |
| Просматривать кампания | ✅ | ✅ | ✅ |
| Редактировать настройки кампании | ✅ | ✅ | ✅ (по умолчанию ✅, можно ограничить позже) |
| Управлять ботом | ✅ | ✅ | ✅ |
| Управлять ссылками и событиями | ✅ | ✅ | ✅ |
| Список участников кампании | ✅ | ✅ | ✅ |
| Приглашать участников | ✅ | ✅ | ✅ (по умолчанию ✅, можно ограничить позже) |
| Передать владение | ✅ | ✅ | ❌ |
| Выйти из кампании | ✅ | ✅ (❌) | ✅ |
| Удалить кампания | ✅ | ✅ | ❌ |

**Примечание**: `campaign-owner` не может выйти из кампании, но может удалить кампания или передать владение.

## 3. Авторизация и общая логика кампанияов

### 3.1 Создание пользователя

*   Любой пользователь может авторизоваться через веб-интерфейс.
*   После успешной авторизации в системе создаётся пользователь с ролью `member`.

### 3.2 Кампания

*   Любой `member` может создать кампанию.
*   Кампания изолирует пользователей, ботов, ссылки, партнёров, рефералов и события.
*   Создатель кампании становится `campaign-owner`.
*   `admin` имеет доступ к кампаниим как `campaign-owner`.

### 3.3 Приглашение в кампанию

**Требование**: `campaign-owner` или `campaign-member` может создать ссылку-приглашение.

**Поток**:

1.  Текущий участник кампании генерирует invite-ссылку.
2.  Если неавторизованный пользователь переходит по invite-ссылке:
    *   попадает на страницу авторизации
    *   после логина возвращается на страницу вступления через параметр `?return=...`
3.  Страница вступления показывает:
    *   описание приглашения (название кампании, кто пригласил)
    *   кнопки: **[Вступить]** и **[Отказаться]**
4.  При нажатии **[Вступить]**:
    *   пользователь добавляется в кампанию как `campaign-member`
    *   после успешного добавления пользователь автоматически редиректится на первую страницу этой кампании
5.  При **[Отказаться]**:
    *   доступ не выдаётся

**Технические требования**:

*   invite-ссылка должна быть одноразовой или многоразовой (выбрать одно, по умолчанию: многоразовая) с возможностью отключения.
*   Ссылка действительна 24 часа.
*   Ссылка может быть отозвана пользователями с правами `admin`, `campaign-owner` или тем пользователем, который её создал.
*   если пользователь уже состоит в кампанияе, повторное вступление не создаёт дубль.

### 3.4 Управление участниками

*   `campaign-owner` и `campaign-member` видят список участников кампании.
*   `campaign-owner` может передать права владельца любому `campaign-member`.
*   `campaign-member` может выйти из кампании в любое время.

### 3.5 Удаление кампании

*   `campaign-owner` может удалить кампания.
*   Перед удалением: 3 последовательных предупреждения в модальных окнах:
    1.  "Удалить кампанию?"
    2.  "Вы уверены?"
    3.  "Вы АБСОЛЮТНО уверены?"

**Технические требования**:

*   удаление должно быть транзакционным (либо полностью, либо откат)
*   при удалении удаляются данные кампании: боты, ссылки, партнёры, рефералы, события, визиты, логи

## 4. Разделы внутри кампании

### A) Партнёры

**Цель**: список всех партнёров кампании (участвуют через Telegram-бота).

**Функции**:

*   таблица партнёров
*   переход в профиль партнёра

**Минимальные поля партнёра**:

*   `tg_id`
*   `username`
*   `full_name`
*   дата регистрации в партнёрской программе
*   показатели: регистрации, заказы, оплаты, сумма оплат, сумма начислений

**Профиль партнёра**:

*   основные данные
*   список созданных партнёрских ссылок
*   краткая сводка по каждой ссылке
*   кнопка перехода к списку рефералов с автоматическим фильтром по партнёру

### B) Рефералы

**Цель**: список рефералов (лидов/клиентов), для которых зафиксирован клик и минимум одно событие.

**Страница рефералов: фильтры**

*   фильтр по дате (диапазон)
*   фильтр по партнёру
*   фильтр по кол-ву заказов
*   фильтр по кол-ву оплат
*   фильтр "продукт" (ввод названия вручную)
*   фильтр по выбранным событиям (multiselect)

**Поведение фильтров**:

*   если пользователь перешёл со страницы партнёра, фильтр по партнёру включается автоматически
*   если в настройках кампании задан список продуктов, для фильтра по продукту на странице рефералов становится доступен только этот список

**Таблица рефералов: колонки**

*   Время регистрации
*   Имя
*   `tg_id` (id в Telegram)
*   `gc_id` (id в GetCourse)
*   Количество заказов
*   Сумма заказов
*   Сумма оплат
*   Кнопка: открыть модальное окно со списком всех событий, которые представлены в виде лога всего, что происходило с этим рефералом

**Сортировка**:

*   каждый столбец поддерживает сортировку asc/desc

**Технические требования**:

*   фильтрация и сортировка делаются сервер-сайд
*   пагинация обязательна (по умолчанию pageSize = 25)

**Модалка "Лог событий реферала"**:

*   список всех событий с датой, названием, payload (в коротком виде) и ссылкой на полный JSON
*   события представлены в хронологическом порядке (от старых к новым или наоборот)

### C) Доступные ссылки (страницы для трафика)

**Цель**: админ кампании задаёт список целевых страниц, на которые партнёры смогут генерировать свои ссылки.

**Экран "Доступные ссылки"**:

*   таблица страниц
*   кнопка "Добавить ссылку" (модалка)

**Модалка добавления страницы**:

*   Название
*   URL

**Шаблон URL**:

*   в URL допускается строка `{ref}`
*   перед редиректом `{ref}` заменяется на уникальный реферальный идентификатор

**После добавления страницы**:

После успешного добавления страницы показывается модальное окно "об успехе" (для улучшения UX), которое содержит:

*   **Webhook URL для подтверждения регистрации**: адрес вида `https://srv.aley.pro/p/gc/partnership/hook~...?key=<секрет_кампании>&ref=<ref>`, который внешняя система должна вызывать для отправки событий (регистрация, заказ, оплата)
    *   Примечание: webhook URL одинаковый для всех кампаний и ссылок, отличаться они будут только параметрами `key` (секрет кампании) и `ref` (реферальный идентификатор)
    *   Секрет кампании (`key`) отображается в этой модалке и также доступен в настройках кампании
*   **Формат обратного вызова**: система принимает как POST, так и GET запросы
*   **Параметр `ref`**:
    *   `ref` — это уникальный идентификатор реферала (длиной до 8 символов), который система генерирует при клике по партнёрской ссылке
    *   `ref` используется для отслеживания пути клиента от клика до регистрации, заказа и оплаты
    *   `ref` всегда передаётся в query-параметре: `?ref=qwe12345` (для GET и POST запросов)
    *   внешняя система должна извлекать `ref` из URL реферала и передавать его в webhook при отправке событий

Администратор копирует webhook URL и вставляет его во внешнюю форму или бота для настройки интеграции.

**Таблица страниц должна включать**:

*   название, URL, формат, дата создания
*   краткая сводка:
    *   привлечено лидов
    *   создано заказов
    *   оплат на сумму N
*   действие: перейти к подробной сводке (переход к рефералам с фильтром по этой странице и партнёру при необходимости)

### D) Бот

**Цель**: подключение Telegram-бота для партнёрской программы. К одной кампании может быть подключен только один бот.

**Функции**:

*   Добавление бота по токену.
*   Бот относится к конкретной кампании.
*   При добавлении регистрируется Telegram webhook, уникальный для этого бота.

**Технические требования**:

*   webhook URL должен содержать идентификатор бота в системе (`botId`), чтобы маршрутизировать апдейты.
*   Учитывать роутинг Chatium (адресация через `~` и `/:id`, ссылка на правила маршрутизации: `@002-routing`).

**Экран "Настройки бота"**:

*   информация о боте с основными полями (минимум): название, username, botId, дата добавления, статус webhook.
*   кнопка "Апдейты".

**Экран "Контроль вебхука"**:

*   отображается требуемый проектом вебхук (webhook URL, который должен быть установлен)
*   отображается текущий вебхук на стороне Telegram (получается через Telegram Bot API)
*   зелёный маркер, если вебхуки совпадают
*   кнопка "Обновить", если вебхуки отличаются (при нажатии устанавливает требуемый вебхук в Telegram)

**Экран "Апдейты бота"**:

*   поток последних входящих апдейтов (Telegram updates)
*   всегда отображаются 25 последних апдейтов, даже после перезагрузки
*   апдейты берутся из Heap-таблицы (хранение истории)

**Обработчик webhook**:

*   принимает update от Telegram (входящие сообщения, команды, нажатия кнопок и другие события от бота)
*   сохраняет каждый update в таблицу `bot_updates` для истории
*   обеспечивает ограничение размера истории: автоматически удаляет старые записи, оставляя только последние N записей (количество записей для хранения устанавливается в настройках кампании, по умолчанию: 25)

**Обязательные детали хранения**:

*   `campaignId`, `botId`, `update_id`, тип апдейта, `tg_user_id` (если есть), payload JSON, `createdAt`

**Ошибки**:

*   неверный токен
*   бот уже добавлен (если в кампании уже есть бот)
*   webhook не установился (показать статус + retry)

### E) Обратные вызовы (Custom events)

**Зачем это нужно**: чтобы учитывать в аналитике **любые дополнительные события** из внешних систем (помимо стандартных «Регистрация / Заказ / Оплата») и показывать их в логе реферала.

**Функции**:

*   CRUD-операции с событиями: админ кампании управляет справочником «типов событий»
*   Для каждого типа события задаются:
    *   название (например, «Просмотр видео», «Скачивание файла»)
    *   описание (что означает событие и когда внешняя система должна его отправлять)
    *   формат входных данных (описание структуры payload для документации интеграции)
*   Приём событий: система принимает webhook запросы **и GET, и POST** (одновременно)
*   Каждый тип события принадлежит конкретной кампании (события разных кампаний не смешиваются)

**Требования**:

*   каждое событие имеет уникальный `eventId`
*   каждый webhook нестандартного события должен уметь принимать `ref` (реферальный идентификатор)

### F) Настройки кампании

Настройки применяются сразу при изменении (без отдельной кнопки "Сохранить"), включая динамику UI.

**Поля**:

1.  **Название кампании**
2.  **Признак нового клиента** (boolean)
    *   если `true`, в партнёрской программе учитываются только рефералы, которых не было ранее
    *   для этого используется Heap-таблица существующих клиентов
    *   загрузка CSV доступна только когда признак `true`
3.  **Загрузка CSV** (появляется только при `true`)
    *   CSV содержит список уже существующих клиентов
    *   минимальный набор полей: идентификатор (`telegram_id` или `email`/`phone`), дата регистрации
4.  **Выбор продуктов, участвующих в программе**
    *   если список задан, на странице рефералов фильтр по продукту ограничен продуктами из этого списка
5.  **Время закрепления реферала за партнёром**
    *   числовое поле (дни)
    *   переключатель "закреплять навсегда" (если `false`, действует ограничение по дням)
6.  **Количество записей истории апдейтов бота**
    *   числовое поле (по умолчанию: 25)
    *   определяет, сколько последних апдейтов от Telegram бота хранить в истории
    *   старые записи автоматически удаляются при превышении лимита
7.  **Секрет кампании для webhook**
    *   неизменяемое поле (генерируется автоматически при создании кампании)
    *   длиной 8 URL-safe символов
    *   используется для защиты webhook endpoints: передаётся в query-параметре `?key=<секрет>`
    *   отображается в модалке "об успехе" после добавления страницы (раздел C)
8.  **Уровень логирования**: `info`, `warn`, `error`, `disable`

## 5. Telegram-бот: бизнес-логика

### 5.1 /start

После того, как бот добавлен в кампанию, он активен и начинает принимать команды.

При сообщении `/start`:

*   система определяет Telegram пользователя
*   если партнёр ещё не зарегистрирован в этом кампанияе: создаёт запись партнёра
*   отправляет сообщение:

> Здравствуйте, (полное имя пользователя из Telegram)!
>
> Вы зарегистрированы в партнёрской программе.
> (или, если в настройках кампании задано имя)
> Вы зарегистрированы в партнёрской программе Название кампании.
>
> Вами привлечено:
> Регистраций - (кол-во регистраций)
> Заказов - (кол-во заказов)
> Из них оплачено (кол-во оплаченных заказов) на сумму (сумма всех платежей)
>
> Сумма партнёрских зачислений составляет - (сколько заработано)
> Не выплачено ещё - (сколько можно вывести)
>
> [Кнопка: Управлять ссылками]
> [Кнопка: Просмотр детализации]
> [Кнопка: Запросить вывод средств]

*Примечание: расчёт "партнёрских зачислений" и "не выплачено" пока берётся как заглушка, если финансовая модель не определена.*

### 5.2 Кнопки

**[Управлять ссылками] -> Telegram Mini App**

**Страница "Мои ссылки"**:

*   список ссылок партнёра
*   если ссылок нет: показываем текст "Ссылки ещё не созданы"
*   внизу фиксированная кнопка **[Создать новую ссылку]**

**Создание новой ссылки**:

*   форма: название ссылки
*   выбор целевой страницы (из списка страниц кампании)
    *   можно выбрать только одну
    *   если доступна только одна страница, она выбрана по умолчанию
    *   если страниц нет, завершение создания недоступно + предупреждение

**После создания**:

*   страница подтверждения:
    *   "Ваша партнёрская ссылка создана!"
    *   Название
    *   URL
    *   кнопка возврата к списку

**Элемент списка ссылки**:

*   название
*   URL
*   кнопка "Подробно" (переходы)

**Подробно (переходы)**:

*   таблица рефералов по этой конкретной ссылке
*   формат аналогичен админке (рефералы конкретного партнёра)

**[Просмотр детализации] -> Telegram Mini App**

**Страница аналитики по ссылкам**:

*   список ссылок партнёра
*   по каждой ссылке:
    *   регистраций
    *   заказов
    *   оплат
*   для каждой ссылки кнопка "Детализация"

**Детализация**:

## 6. Партнёрские ссылки, клики, fingerprint и ref

### 6.1 Генерация партнёрской ссылки

Партнёр создаёт ссылку внутри Mini App.

Рекомендуемый формат ссылки в системе:

*   публичный URL вида: `https://srv.aley.pro/p/gc/partnership/r~<partnerLinkId>`
    *   Формат основан на file-based роутинге Chatium: файл `r.tsx` в `p/gc/partnership/` с маршрутом `/:id` формирует URL через тильду `~`
*   `partnerLinkId` уникален и указывает на связку: `campaignId` + `partnerId` + `pageId`

При открытии `https://srv.aley.pro/p/gc/partnership/r~<partnerLinkId>`:

1.  сервер определяет fingerprint пользователя для дедупликации кликов
2.  создаёт (или переиспользует) `ref` длиной <= 8 URL-safe
3.  сохраняет визит в Heap-таблицу
4.  редиректит на целевой URL страницы, подставляя `{ref}`

### 6.2 Fingerprint

**Требование**: fingerprint должен предотвращать множественные записи одного человека.

**Рекомендация реализации**:

*   вычислять `fingerprintHash` из набора признаков (ip, ua, accept-language, platform, timezone, возможно client hints)
*   хранить данные компонентов, чтобы каждый можно было сравнивать отдельно
*   хранить данные в хэшированном виде, но учитывать, что библиотека crypo недоступна в окружении Chatium

**Важно**:

*   хранить персональные данные в открытом виде минимально

### 6.3 Атрибуция

*   `ref` связывает: клик -> регистрация -> заказ -> оплата
*   закрепление за партнёром действует по настройкам кампании:
    *   навсегда
    *   или N дней

## 7. Webhooks от внешних систем

Система принимает события от внешней системы по webhook.

### 7.1 Общие требования к webhook API

*   Приём GET и POST одновременно (система принимает любой формат для всех webhook endpoints)
*   Везде обязателен `ref`, который всегда передаётся в query-параметре как `?ref=qwe12345` (для GET и POST запросов)
*   Вебхуки должны быть идемпотентны (повторный вызов не создаёт дублей)
*   Каждый webhook должен иметь защиту:
    *   секрет всегда передаётся в query-параметре с именем `key`: `?key=<секрет>`
    *   секрет уникален для каждой кампании, длиной 8 URL-safe символов
    *   секрет генерируется автоматически при создании кампании и отображается в настройках кампании как неизменяемое поле

### 7.2 Стандартные события

#### 7.2.1 Регистрация

**Входные данные (минимум)**:

*   `ref`
*   `tg_id` (если регистрация в Telegram)
*   `gc_id` (если GetCourse)
*   `name`
*   `email`/`phone` (если есть)

**Поведение**:

*   найти визит по `ref`
*   расширить запись визита: `registeredAt`, `referralProfile`
*   создать/обновить реферала
*   записать событие регистрации в отдельную Heap-таблицу стандартных событий (см. раздел 8 — таблица регистраций)

#### 7.2.2 Создан заказ

**Входные данные**:

*   `ref`
*   `order_id`
*   `product_name`
*   `order_sum`

**Поведение**:

*   создать (или обновить, если уже есть) заказ с `order_id`
*   привязать заказ к рефералу по `ref` (заказ всегда связан с лидом/рефералом через `ref`)
*   заказ может существовать без оплаты — это нормальная ситуация
*   увеличить агрегаты

#### 7.2.3 Совершена оплата

**Входные данные**:

*   `ref`
*   `order_id`
*   `product_name`
*   `order_sum` (опционально)
*   `payment_sum`

**Поведение**:

*   создать (или обновить, если уже есть) оплату
*   привязать оплату к рефералу по `ref` (оплата всегда связана с лидом/рефералом через `ref`)
*   сохранить `order_id` в записи оплаты, чтобы позже связать оплату с заказом
*   попытаться связать оплату с заказом по `order_id`, если заказ уже существует
*   если оплата пришла раньше заказа — это не ошибка: оплата остаётся сохранённой и будет связана с заказом, когда заказ появится
*   увеличить агрегаты

### 7.3 Нестандартные события

*   событие может приходить как GET, так и POST (система принимает оба формата одновременно)
*   `ref` всегда передаётся в query-параметре как `?ref=qwe12345` (для GET и POST запросов)
*   секрет кампании всегда передаётся в query-параметре `?key=<секрет>` для защиты webhook

Требование к хранению:

*   для **каждого типа нестандартного события** создаётся **отдельная Heap-таблица** (см. раздел 8)
*   таблица хранит фиксированные базовые поля (`campaignId`, `eventId`, `ref`, `createdAt`) + динамический payload через `Heap.Any()` (в документации также встречается формулировка “heap.any”)

## 9. Нефункциональные требования

### 9.1 Производительность

*   таблицы со списками: пагинация и server-side фильтры
*   агрегаты (сводки) должны считаться быстро (либо материализовать, либо оптимизировать запросы)

### 9.2 Идемпотентность

*   регистрация: не дублировать по `ref`
*   заказ: не дублировать по (`campaignId`, `order_id`)
*   оплата: не дублировать по (`campaignId`, `order_id`, `payment_sum`, `timestamp`) или отдельному `payment_id`

### 9.3 Логирование

*   уровень логирования берётся из настроек кампании
*   минимум логов:
    *   входящие webhook события
    *   ошибки парсинга
    *   попытки доступа без прав

### 9.4 Безопасность

*   webhook endpoints защищаются секретом кампании:
    *   секрет передаётся в query-параметре `?key=<секрет>`
    *   секрет уникален для каждой кампании, длиной 8 URL-safe символов
    *   секрет генерируется автоматически при создании кампании и не может быть изменён
*   токены ботов хранятся зашифрованными
*   доступ к кампаниим проверяется на каждом запросе
*   rate limiting на webhook endpoints

### 9.5 UX

*   все изменения в настройках применяются сразу
*   состояния загрузки/ошибок обязательны
*   не блокировать интерфейс долгими запросами (использовать фоновые обновления)

## 10. Критичные edge-cases (учесть в разработке)

1.  Партнёр создаёт ссылку, но в кампанияе нет доступных страниц.
2.  Внешняя система отправила webhook без `ref`.
3.  Внешняя система повторно отправляет один и тот же webhook.
4.  Реферал кликает несколько раз:
    *   один и тот же браузер
    *   разные устройства
5.  `ref` устарел по сроку закрепления.
6.  Удаление кампании: корректно удалить все дочерние сущности.
7.  Передача владения: новый owner получает доступ сразу.
8.  Бот токен неправильный или webhook не регистрируется.

## 11. Приёмка (минимальный набор сценариев)

1.  Авторизация -> создание кампании -> owner назначен.
2.  Инвайт: переход без авторизации -> логин -> вступление -> пользователь стал member кампании.
3.  Передача владения -> новый owner видит права.
4.  Добавление бота -> webhook зарегистрирован -> `/start` работает.
5.  Mini App: партнёр создал ссылку -> получил URL.
6.  Клик по партнёрской ссылке -> визит создан -> редирект с подстановкой `ref`.
7.  Пришёл webhook регистрации -> реферал создан -> показан в таблице.
8.  Пришёл webhook заказа/оплаты -> агрегаты обновились.
9.  Фильтры рефералов работают, сортировки работают.
10. Удаление кампании с 3 подтверждениями.

## 12. Рекомендуемые соглашения (чтобы избежать разночтений)

*   **Валюта и формат сумм**: хранить в минимальных единицах (копейки) целым числом.
*   **Время**: хранить как ISO строку UTC + выводить в интерфейсе по таймзоне пользователя.
*   **Длина ref**: 8 символов, алфавит base62 (A-Z a-z 0-9).
*   **Стандартный URL для редиректа**: `https://srv.aley.pro/p/gc/partnership/r~<slug>` (тильда `~` вместо слеша `/` из-за параметра пути `/:id` в роутинге Chatium).
*   **Все webhook endpoints используют формат**: `https://srv.aley.pro/p/gc/partnership/hook~<тип_события>?key=<секрет>&ref=<ref>`
    *   где `key` — секрет кампании (8 URL-safe символов), `ref` — реферальный идентификатор
