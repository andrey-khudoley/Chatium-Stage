# Проект lava_gc_integration (Интеграция Lava.top и GC)

## Что это
Приложение Chatium по пути `p/units/aom/lava_gc_integration`: интеграция Lava.top и GetCourse. Структура унаследована от шаблона (страницы, API настроек и логов, Heap). Heap: помимо `settings` и `logs` — таблицы `lava_payment_contract`, `lava_webhook_event`, `lava_lock_log` (файлы в `tables/`), репозитории в `repos/` (`lava_payment_contract.repo.ts`, `lava_webhook_event.repo.ts`, `lava_lock_log.repo.ts`); в каждом из трёх `lava_*` репозиториев — `writeServerLog` severity 7 на вход/выход операций (без рекурсии с `logs.repo.create`, в отличие от записи в таблицу `logs` из `logger.lib`). В `lib/settings.lib.ts` объявлены ключи и геттеры для секретов/URL Lava и GetCourse (`lava_api_key`, `lava_base_url`, `lava_product_id`, `lava_offer_id`, `lava_webhook_secret`, `gc_api_key`, `gc_account_domain`, `gc_service_token`). Типы внешних контрактов (валюта, webhook, payment-link) — `lib/lava-types.ts`. Мастер на админке Lava: ввод ключа и URL → POST `/api/admin/lava/catalog` (каталог продуктов/офферов) → выбор → сохранение id в настройки; блок GetCourse: поля `gc_api_key` и `gc_account_domain` (`shared/gcSettingKeys.ts`) → POST `/api/admin/getcourse/verify` (`verifyGcPlApiAccess`) → «Сохранить в настройки»; `shared/lavaBaseUrl.ts` и `lib/lava-api.client.ts` (исходящий HTTP к Lava — `request({…})` без `ctx`, не `request(ctx,…)`): `updateOfferPrice` (PATCH цены оффера), `createContract` (POST invoice v3), `getProducts` (GET первая страница / диагностика), `fetchLavaProductsCatalog` (пагинация для каталога админки). Основной поток ссылки на оплату — `lib/lava-payment.service.ts`: `createPaymentLink` (идемпотентность по активному контракту GC, `runWithExclusiveLock` + `lava_lock_log`, PATCH/POST Lava, запись `lava_payment_contract`). Приём webhook Lava — `lib/lava-webhook.service.ts`: `processWebhook` (секрет `lava_webhook_secret` в заголовке `X-Api-Key`, дедупликация по `dedupe_key`, повтор обработки при сохранённой ошибке `contract_not_found`, маппинг статусов в Heap, вызов `updateDealStatus` для `paid`/`failed`). HTTP-вход для webhook — `POST /api/integrations/lava/webhook` (`api/integrations/lava/webhook/index.ts`, `lavaWebhookRoute`): тело через `.body` (`@app/schema`), вызов `processWebhook`. Исходящие вызовы в GetCourse PL API — `lib/getcourse-api.client.ts`: `normalizeGcAccountDomain`, `verifyGcPlApiAccess` (проверка ключа/домена для админки), `updateDealStatus` (POST form `action`/`key`/Base64 `params` на `/pl/api/deals`, без throw при ошибке ответа); **environment-agnostic Base64**: функция `encodeBase64()` использует `Buffer` (Node.js) / `btoa` (браузер) / ручную реализацию (sandbox) — полностью самодостаточная, работает во всех средах выполнения Chatium (UTF-8, суррогатные пары, RFC 4648), критична для компиляции `// @shared-route` в клиентский бандл. Входящий запрос ссылки на оплату от GetCourse — `POST /api/integrations/lava/payment-link` (`api/integrations/lava/payment-link/index.ts`, `lavaPaymentLinkRoute`): `X-Service-Token` или `Authorization: Bearer` = `gc_service_token`, тело через `.body` (`@app/schema`), `createPaymentLink`.

## Тесты интеграции Lava
`lib/settings-save-credentials.lib.ts` — чистые функции слияния полей для `POST /api/settings/save` (`resolveGcCredentialsForSave`, `resolveLavaCredentialsForSave`, `shouldVerifyCredentialPair`). Категории в `GET /api/tests/list`: `unit-save-credentials` (GET `settings-save-credentials-unit` — юнит без сети), `integration-credentials-both` (GET `integration-credentials-both` — обе живые проверки из Heap), `integration-check` — по отдельности `integration-gc-credentials` и `integration-lava-credentials` (`runGcCredentialCheckFromSettings` / `runLavaCredentialCheckFromSettings`). Прочие GET в `api/tests/endpoints-check/` (`lava-settings-getters`, `lava-repos`, …) — в `docs/api.md`, часть не дублируется в list. Страница `./web/tests` (`TestsPage.vue`): вкладка «Юнит» — сценарии слияния ключей; «Интеграция» — два GET (`integration-gc-credentials`, `integration-lava-credentials`); «Устаревшее» — шесть слоёв + два отдельных GET ключей; «Запустить все тесты» зависит от активной вкладки. Слоёвые GET — query `testId=<id>`. Отдельно: GET `integration-credentials-both` (оба чекера в одном ответе) — без обязательного дублирования в UI.

## Маршруты интеграции (константы)
`PROJECT_ROOT` вынесен в **`shared/projectRoot.ts`** — Vue (`TestsPage.vue`, …) и `web/login` импортируют **только** его, не весь `config/routes.tsx` (иначе в клиентский бандл могли попадать пути/API-модули → ошибка при оценке `payment-link/index.ts`). В `config/routes.tsx`: **`ROUTES`** / **`ROUTE_PATHS`** только страницы; URL интеграций — в `docs/api.md` (`/api/integrations/lava/payment-link`, webhook). `config/project.tsx` — `// @shared`.

## Критические правила
- Не хардкодить URL, использовать `config/routes.tsx`.
- Документацию обновлять при любых изменениях структуры/логики.
