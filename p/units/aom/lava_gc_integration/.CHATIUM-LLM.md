# Проект lava_gc_integration (Интеграция Lava.top и GC)

## Что это
Приложение Chatium по пути `p/units/aom/lava_gc_integration`: интеграция Lava.top и GetCourse. Структура унаследована от шаблона (страницы, API настроек и логов, Heap). Heap: помимо `settings` и `logs` — таблицы `lava_payment_contract`, `lava_webhook_event`, `lava_lock_log` (файлы в `tables/`), репозитории в `repos/` (`lava_payment_contract.repo.ts`, `lava_webhook_event.repo.ts`, `lava_lock_log.repo.ts`); `create` в этих репозиториях: публичный вход `HeapCreateInput` / экспорт `*CreateInput`, внутри — приведение к `Parameters<typeof …Table.create>[1]` (typings `@app/heap`); в `lava-payment.service` — именованные `create`, `findActiveByGcOrderAmountAndCurrency` и `deactivateActiveContractsForGcOrderId` из `lava_payment_contract.repo`; в каждом из трёх `lava_*` репозиториев — `writeServerLog` severity 7 на вход/выход операций (без рекурсии с `logs.repo.create`, в отличие от записи в таблицу `logs` из `logger.lib`). В `lib/settings.lib.ts` объявлены ключи и геттеры для секретов/URL Lava и GetCourse (`lava_api_key`, `lava_base_url`, `lava_product_id`, `lava_offer_id`, `lava_webhook_secret`, `gc_api_key`, `gc_account_domain`, `gc_order_flag_addfield_id`). Типы внешних контрактов (валюта, webhook, payment-link) — `lib/lava-types.ts`. Мастер на админке Lava: ввод ключа и URL → POST `/api/admin/lava/catalog` (каталог продуктов/офферов) → выбор → сохранение id в настройки; поле `lava_webhook_secret` — автогенерация hex (кнопки «Сгенерировать»/«Обновить»), показ по «Показать», сохранение в Heap через `POST /api/settings/save`; блок GetCourse: поля `gc_api_key`, `gc_account_domain` и `gc_order_flag_addfield_id` (`shared/gcSettingKeys.ts`) → POST `/api/admin/getcourse/verify` (`verifyGcPlApiAccess`) → «Сохранить в настройки»; `shared/lavaBaseUrl.ts` и `lib/lava-api.client.ts` (исходящий HTTP к Lava — `request({…})` без `ctx`, не `request(ctx,…)`): `updateOfferPrice` (PATCH цены оффера), `createContract` (POST invoice v3), `getProducts` (GET первая страница / диагностика), `fetchLavaProductsCatalog` (пагинация для каталога админки). Основной поток ссылки на оплату — `lib/lava-payment.service.ts`: `createPaymentLink` (идемпотентность по заказу GC + сумма + валюта; при валюте `USD`/`EUR` входная сумма трактуется как RUB и конвертируется через `lib/cbr-rates.client.ts` по актуальному курсу ЦБ (`https://www.cbr-xml-daily.ru/daily_json.js`) перед PATCH/contract; затем проверка диапазона суммы для Lava — `lib/lava-amount-limits.lib.ts` (USD/EUR 5…10000 по ответу API Lava, RUB от 50), при нарушении — `AMOUNT_OUT_OF_RANGE` с непустым `message`; при смене суммы — отмена прочих активных контрактов по заказу; `runWithExclusiveLock` + `lava_lock_log`, PATCH/POST Lava, запись `lava_payment_contract`). Приём webhook Lava — `lib/lava-webhook.service.ts`: `processWebhook` (секрет `lava_webhook_secret` в заголовке `X-Api-Key`, дедупликация по `dedupe_key`, повтор обработки при сохранённой ошибке `contract_not_found`, маппинг статусов в Heap, вызов `updateDealStatus` для `paid`/`failed`). HTTP-вход для webhook — `POST /api/integrations/lava/webhook` (`api/integrations/lava/webhook/index.ts`, `lavaWebhookRoute`): тело через `.body` (`@app/schema`), вызов `processWebhook`; тот же путь **GET** — `lavaWebhookInfoRoute`, JSON-проба доступности для браузера. Исходящие вызовы в GetCourse PL API — `lib/getcourse-api.client.ts`: `normalizeGcAccountDomain`, `verifyGcPlApiAccess` (проверка ключа/домена для админки), `updateDealStatus`, `probeGcOrderPlApi` (диагностическая проба заказа по `deal_number` + email, `deal_status=payment_waiting`; GET тест `api/tests/endpoints-check/integration-gc-order-pl-api.ts`) (POST form `action`/`key`/Base64 `params` на `/pl/api/deals`, без throw при ошибке ответа). При заданном `gc_order_flag_addfield_id` в payload заказа передаётся `deal.addfields = { [id]: true }`. **environment-agnostic Base64**: функция `encodeBase64()` использует `Buffer` (Node.js) / `btoa` (браузер) / ручную реализацию (sandbox) — полностью самодостаточная, работает во всех средах выполнения Chatium (UTF-8, суррогатные пары, RFC 4648), критична для компиляции `// @shared-route` в клиентский бандл. Входящий запрос ссылки на оплату — `POST /api/integrations/lava/payment-link` (`api/integrations/lava/payment-link/index.ts`, `lavaPaymentLinkRoute`): **без** заголовков авторизации (в т.ч. вызов из браузера/JS на странице оплаты), тело через `.body` (`@app/schema`): заказ `gcOrderId` или `orderNumber`, email `buyerEmail` или `email`, поля GetCourse `offer`/`product` (копятся в Heap как `gc_offer_title`/`gc_product_title`; `product` уходит в Lava PATCH как имя оффера); валюта нормализуется `lib/lava-currency.lib.ts`; опционально `integrationTestDryRun: true` — ответ успеха без Lava/Heap. Для расследования инцидентов email в debug добавлены сырые email-логи и полный payload перед `POST /api/v3/invoice` в `createContract` (включая `email`, `emailLowercase`, длину и `charCode`-массив). Тесты: GET `payment-link-dry-run-unit` (`lavaPaymentLinkRoute.run(ctx, { gcOrderId, … })` — поля тела на верхнем уровне, не `{ body: { … } }`), POST `payment-link-http-integration` (`request()` на тот же URL, тело с опциональным `paymentLinkOverrides`), лайв: GET `payment-link-heap-settings-read`, GET `payment-link-full-route-run`, POST `payment-link-full-http-integration` (`payment-link-live-test.lib`, сброс контрактов `gc_order_id=test` перед полным путём), `lib/app-public-url.lib.ts`.
В webhook-потоке обрабатывается только успешная оплата (`payment.success` + `completed`): заказ GetCourse переводится в статус `in_work` («В работе») и передаётся `deal_is_paid=1`; неуспешные/нерелевантные события webhook игнорируются.

## Тесты интеграции Lava
`lib/settings-save-credentials.lib.ts` — чистые функции слияния полей для `POST /api/settings/save` (`resolveGcCredentialsForSave`, `resolveLavaCredentialsForSave`, `shouldVerifyCredentialPair`). Категории в `GET /api/tests/list`: `unit-save-credentials`, `unit-page-routes`, `integration-credentials-both`, `integration-check`, `payment-link-integration` (dry-run юнит + HTTP POST), `webhook-route-unit` (GET `lava-webhook-route`: GET `lavaWebhookInfoRoute`; POST-сценарии — `request()` на URL webhook или fallback `processWebhook`; валидация тела — `lavaWebhookRoute.run` — не путать с GET `lava-webhook-service`, где только `processWebhook`), `payment-link-live` (чтение Heap для Lava, полный `route.run` и HTTP без dry-run — см. `lib/payment-link-live-test.lib.ts`, `deactivateActiveContractsForGcOrderId` в `lava_payment_contract.repo`; плюс POST/GET `webhook-live-test-arm` / `webhook-live-test-status` и `lib/webhook-live-test.lib.ts` для лайв-проверки webhook после оплаты), прочие — см. `docs/api.md`. Страница `./web/tests` (`TestsPage.vue`): вкладка «Юнит» — страницы (`route.run`), слияние ключей save, карточка **Lava webhook (route.run)** (GET `…/lava-webhook-route`, опционально `?testId=`); вкладка «Интеграция» — блоки payment-link dry-run, карточка **GetCourse по номеру заказа** (ввод `gcOrderId`, опционально email; GET `…/integration-gc-order-pl-api`), отдельная карточка «Лайв: payment-link» (триада Heap + route.run + HTTP), карточка «Лайв: webhook после оплаты» (URL POST webhook, ссылка на оплату, вооружение по `lava_contract_id`); «Запустить все тесты» на интеграции **не** запускает лайв-триаду (отдельная кнопка). Слоёвые GET — query `testId=<id>`. Отдельно: GET `integration-credentials-both`. **Страница заказов:** `./web/orders` (`OrdersPage.vue`, только Admin), метрики и постраничный список контрактов по `lava_payment_contract` — `lib/admin/orders-metrics.lib.ts`, `lib/admin/orders-list.lib.ts`; API `GET /api/admin/orders-metrics`, `GET /api/admin/orders-filter-options`, `GET /api/admin/orders-list` (по 50 строк, `page` с 1); для фильтра `to` используется включающая верхняя граница выбранной минуты (`+59.999 сек.`), чтобы UI не терял записи в конце минуты.

Логи в админке (`pages/AdminPage.vue`): кнопка «Загрузить ещё 50» загружает историю через `GET /api/admin/logs/before`. При накоплении большого объёма в клиенте список ограничивается `MAX_LOG_ENTRIES=5000`; обрезка выполняется без дополнительного логирования (чтобы не было рекурсии через `setLogSink`). Исторические пакеты объединяются без дублей по `id`, а курсор пагинации (`oldestLogTimestamp`) пересчитывается по фактическому массиву логов.

Фильтры `Info/Warn/Error` в блоке логов админки теперь запускают полную серверную перезагрузку истории по выбранным уровням: клиент строит `severities` (CSV кодов 0..7) и запрашивает `GET /api/admin/logs/recent` + `GET /api/admin/logs/before` постранично до конца. В API `recent`/`before` добавлена опциональная query `severities`; в `repos/logs.repo` (`findAll`, `findBeforeTimestamp`) добавлена фильтрация `severity: { $in: [...] }`.

## Временные HTTP-пробы (`temp/`)
Каталог **`temp/`** (не прод): маршруты для ручной проверки, как Chatium отдаёт разные статусы и тела (200/3xx/4xx/5xx, JSON при 200 с `ok: false`, редиректы, `throw`). Оглавление: **`./temp`** относительно `PROJECT_ROOT`. См. `docs/testing.md`. Сводка по фактическим HTTP-ответам (curl): **`inner/docs/048-chatium-http-response-probes.md`**.

## Маршруты интеграции (константы)
`PROJECT_ROOT` вынесен в **`shared/projectRoot.ts`** — Vue (`TestsPage.vue`, …) и `web/login` импортируют **только** его, не весь `config/routes.tsx` (иначе в клиентский бандл могли попадать пути/API-модули → ошибка при оценке `payment-link/index.ts`). В `config/routes.tsx`: **`ROUTES`** / **`ROUTE_PATHS`** только страницы; URL интеграций — в `docs/api.md` (`/api/integrations/lava/payment-link`, webhook). `config/project.tsx` — `// @shared`.

## Критические правила
- Не хардкодить URL, использовать `config/routes.tsx`.
- Документацию обновлять при любых изменениях структуры/логики.
