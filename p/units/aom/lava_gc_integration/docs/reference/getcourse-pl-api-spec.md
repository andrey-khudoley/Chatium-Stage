# GetCourse: формат PL API (импорт / сделки)

**Первоисточник (полная документация):** [Документация по формату вызова и ответа API](https://getcourse.ru/help/api).

Этот файл — **сжатая спецификация** для разработки интеграции в `lava_gc_integration`: что нужно знать про вызовы GetCourse из Chatium (обновление заказа после оплаты через Lava) и про ограничения API. Детали полей, все коды статусов и примеры — на стороне GetCourse по ссылке выше.

---

## 1. Общие правила

| Параметр | Значение |
| --- | --- |
| Протокол | Только **HTTPS**. |
| Назначение | **Import API** — оперативное создание/обновление объектов (в т.ч. заказы). **Export API** — выгрузки пачками, с отдельными лимитами. |
| Тарифы | Импорт/экспорт по API доступны **только на платных тарифах**; на ознакомительном тарифе функционал закрыт (см. официальную статью). |
| Домен | Базовый шаблон: `https://{account_name}.getcourse.ru/...`. Если в аккаунте настроен редирект на свой домен, в запросах нужно использовать **фактический домен** вместо `*.getcourse.ru` (см. примечание в конце статьи GetCourse). |

---

## 2. Формат запроса (Import API)

Все параметры передаются в **теле POST** (не GET). Обычно **`application/x-www-form-urlencoded`** или эквивалент form-data.

| Поле | Описание |
| --- | --- |
| **`key`** | Секретный ключ API (генерируется в настройках аккаунта GetCourse). Передаётся **при каждом** запросе. Отдельной сессии/OAuth нет. |
| **`action`** | Имя действия (например `add` для импорта пользователя или сделки). |
| **`params`** | JSON с параметрами действия, закодированный в **Base64**. |

**Пример оболочки (логически):**

```http
POST https://{account}.getcourse.ru/pl/api/deals
Content-Type: application/x-www-form-urlencoded

action=add&key={secret_key}&params={base64(json)}
```

---

## 3. Формат ответа

Ответ — **JSON**. Типичная структура (по документации GetCourse):

```json
{
  "success": "true/false",
  "action": "вызванное действие",
  "result": {
    "deal_id": "уникальный id заказа",
    "deal_number": "номер заказа",
    "success": "true/false",
    "user_id": "id пользователя",
    "user_status": "статус пользователя",
    "payment_link": "ссылка на страницу оплаты",
    "error_message": "сообщение об ошибке",
    "error": "true/false"
  }
}
```

При реализации в Chatium различать **HTTP-успех** и **`result.success` / `result.error`**: GetCourse может вернуть HTTP 200 с вложенной ошибкой в `result`.

## 4. Импорт сделок (заказов)

**URL:** `https://{account_name}.getcourse.ru/pl/api/deals`

**Действие:** `action=add`

**Тело `params` (после Base64-декодирования)** — объект с вложенными блоками, среди прочего:

- **`user`** — данные пользователя-владельца сделки (как в импорте пользователя; часто минимум `email`).
- **`system`** — флаги, в т.ч.:
  - `refresh_if_exists` — обновлять ли существующего пользователя (0/1);
  - `multiple_offers` — несколько позиций в одном заказе (0/1);
  - **`return_payment_link`** — **1/0**, возвращать ли ссылку на оплату в ответе;
  - `return_deal_number` — возвращать номер заказа.
- **`session`** — UTM и аналоги (контекст создания заказа).
- **`deal`** — поля заказа: `deal_number`, `offer_code` / `offer_id`, `deal_cost`, `deal_status`, `deal_is_paid`, `payment_type`, `payment_status`, валюта, доп. поля, воронка и т.д.

**Минимум для нового заказа** (из статьи):

- уникальный **код предложения** или наименование **+** сумма **+** email пользователя  
  **или**
- **ID предложения** **+** email пользователя.

Для **обновления существующего заказа** дополнительно нужен **`deal_number`**.

Особенности **`offer_id`**, **`deal_cost`** и игнорирование `product_title` при найденном предложении — см. блок «Особенности параметра offer_id» в [официальной документации](https://getcourse.ru/help/api).

---

## 5. Изменение статуса заказа

Тот же endpoint: `pl/api/deals`, **`action=add`**, в `params` минимум:

```json
{
  "user": { "email": "email@example.com" },
  "deal": {
    "deal_number": "13667463",
    "deal_status": "new"
  }
}
```

**Коды `deal_status` (пример из документации):** `new`, `payed`, `cancelled`, `false`, `in_work`, `payment_waiting`, `part_payed`, `waiting_for_return`, `not_confirmed`, `pending`.

**Важно:** при смене статуса на «Завершён» (`payed`) в заказе может быть создан платёж (аналогично передаче `deal_is_paid`). Отмена доступа по покупке через смену статуса с «Завершён» на другой **не отменяет** активные покупки автоматически — для отзыва доступа GetCourse рекомендует сценарий с доп. полем и процессом (см. статью).

**Коды `payment_status` и `payment_type`:** длинные перечни в статье; для внешней оплаты через Lava конкретное значение `payment_type` нужно согласовать с настройками аккаунта GetCourse (в списке есть `OTHER` и множество провайдеров).

## 6. Импорт пользователей (кратко)

**URL:** `https://{account_name}.getcourse.ru/pl/api/users`  
**Действие:** `add` / `update` и т.д.

Нужен, когда при создании сделки в GetCourse ещё нет пользователя с данным email — импорт сделки может создать пользователя. Для потока «только обновить оплату по существующему заказу» достаточно блока `user` с `email` в запросе к `deals`.

---

## 7. Export API (справочно)

**База:** `https://{account_name}.getcourse.ru/pl/api/account/...`  
Выгрузки пользователей, групп, заказов, платежей — **асинхронно**, с `export_id` и повторным запросом статуса.

**Лимит:** до **100 запросов за 2 часа** на Export API (все методы экспорта суммарно). Не предназначен для потоковой доставки новых событий; для оперативных callback GetCourse в статье рекомендуют процессы с операцией **«Вызвать URL»**.

Для сценария «Lava оплатил → Chatium уведомил GetCourse» обычно достаточно **Import API** (`deals`), а не Export.

---

## 8. Лимиты и типовые ошибки

- На создание **новых** объектов через API действуют **месячные лимиты** (тип объекта: пользователи / заказы / платежи). На изменение уже существующих объекты эти лимиты **не распространяются** (см. раздел «Лимиты Import API» в статье).
- Частые ошибки: неверный `key` / имя аккаунта, ключ только на чтение, `params` не в Base64, пустой `action`, GET вместо POST, неверный `Content-Type`.

Полный список ошибок и рекомендаций — в разделе «Возможные ошибки и способы их решения» на [getcourse.ru/help/api](https://getcourse.ru/help/api).

## 9. Связь с архитектурой Lava + Chatium

| Роль | Как использовать PL API GetCourse |
| --- | --- |
| GetCourse → Chatium | Запрос ссылки на оплату реализуется **не** через PL API, а по **внутреннему** контракту приложения ([integration-http-contracts.md](../integration-http-contracts.md)). |
| Chatium → GetCourse | После webhook Lava — вызов **`pl/api/deals`** (или сценарий «Вызвать URL» из процесса GetCourse, если так проще заказчику): обновить статус заказа / платежа согласованным способом. Поля `deal_number` / идентификаторы должны совпадать с тем, что передаётся в `payment-link` из GetCourse. |
| Секреты | Ключ API GetCourse хранить в настройках Heap / workspace; **не** логировать. |

Точный набор полей в `deal` при подтверждении оплаты (статус, тип платежа, доп. поля) — **предмет согласования** с владельцем аккаунта GetCourse и фиксируется при реализации (см. [integration-readiness-decisions-open-questions.md](../integration-readiness-decisions-open-questions.md)).
