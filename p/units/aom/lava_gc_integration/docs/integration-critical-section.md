# Критическая секция и защита от гонки

## 9.1. Почему гонка реальна

Если два запроса одновременно работают с одним техническим оффером:

1. запрос A меняет цену на 4990;
2. запрос B меняет цену на 1990;
3. запрос A создаёт контракт и получает цену 1990;
4. запрос B создаёт свой контракт.

Такой сценарий недопустим. Защита от гонки — **обязательная** часть архитектуры.

## 9.2. Граница блокировки

Под одной блокировкой выполняются только:

1. загрузка конфигурации шаблонного оффера (при необходимости);
2. обновление цены оффера в Lava;
3. создание контракта в Lava;
4. сохранение результата создания контракта локально (Heap).

После сохранения локальной записи о контракте блокировку можно снимать.

## 9.3. Что не выполнять под блокировкой

1. ожидание оплаты пользователем;
2. полную обработку webhook (кроме минимального приёма, если платформа иначе не позволяет — предпочтительно вынести);
3. callback в GetCourse о результате оплаты;
4. побочные бизнес-операции;
5. длинные фоновые задачи.

## 9.4. Требования к блокировке

1. эксклюзивная;
2. атомарная по смыслу операции «цена + контракт + запись»;
3. ограниченная по времени;
4. устойчивая к падению процесса (платформенная семантика);
5. гарантированное освобождение при ошибке (try/finally или эквивалент).

## 9.5. Реализация в Chatium (обязательно для этого проекта)

В общем случае допустимы Redis lock, SQL-блокировка, распределённый lock. **Для приложения в Chatium** основной механизм — модуль **`@app/sync`**:

- `runWithExclusiveLock(ctx, lockId, options, fn)` — ожидание до таймаута, при таймауте `LockAcquisitionError`;
- `tryRunWithExclusiveLock` — без ожидания, мгновенный отказ при занятости.

Справочник: `inner/docs/028-sync.md`, контекст Heap: `inner/docs/008-heap.md` (раздел о race condition).

**Недопустимо:**

1. lock только в памяти одного процесса как единственная защита;
2. флаг в обычной переменной;
3. фронтенд-задержки вместо серверной блокировки;
4. чтение и запись цены/контракта без атомарной сериализации запросов к одному офферу.

## 9.6. Имя lock-ключа

Для одного технического оффера — один ключ, например:

```text
lava:template-offer:payment-lock
```

При нескольких шаблонах ключ строится по `templateId` или `offerId`.

## 9.7. TTL и таймауты

Рекомендуется удержание/таймаут порядка **15–30 секунд** (уточнять по латентности Lava). Должно покрывать:

1. один вызов обновления цены;
2. один вызов создания контракта;
3. локальное сохранение результата.

В `runWithExclusiveLock` задаются `timeoutMs` / `maxDurationMs` согласно документации `@app/sync`.

## 9.8. Поведение при занятой блокировке

**Вариант A — короткое ожидание:** запрос ждёт освобождения ограниченное время, затем повтор; при истечении — ошибка занятости (`PAYMENT_TEMPLATE_BUSY`).

**Вариант B — мгновенный отказ:** `tryRunWithExclusiveLock`, GetCourse инициирует повтор.

Для UX предпочтительнее короткое ожидание с ограниченным retry со стороны GetCourse.
