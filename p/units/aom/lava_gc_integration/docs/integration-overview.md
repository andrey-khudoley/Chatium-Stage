# Обзор интеграции: назначение и ответственность

## 1. Назначение

Описывается целевая логика интеграции, в которой **не используется** пул продуктов, подбор совпадений, резервирование слотов и проверка доступности на этапе открытия страницы оплаты.

Новая модель опирается на следующее свойство Lava.top:

1. Перед созданием контракта можно изменить стоимость оффера.
2. В момент создания контракта Lava фиксирует текущую цену.
3. После создания контракта дальнейшее изменение цены оффера не влияет на уже созданный контракт.

Упрощённая схема:

1. В Lava существует один заранее подготовленный **технический** продукт с одним техническим оффером.
2. При запросе ссылки на оплату Chatium временно блокирует выполнение критической секции.
3. Внутри критической секции Chatium обновляет стоимость оффера под сумму текущего заказа.
4. Сразу после этого Chatium создаёт контракт.
5. Контракт получает актуальную цену на момент создания.
6. Блокировка снимается.
7. Chatium возвращает ссылку на оплату в GetCourse.

Вся логика подбора, совпадений, недельных cooldown, поиска свободных продуктов и предварительной резервации **удаляется** из архитектуры.

## 2. Итоговая бизнес-идея

GetCourse продолжает хранить весь реальный каталог продуктов и предложений. Lava не хранит каталог продуктов GetCourse и не участвует в выборе товаров. Lava используется только как платёжный шлюз, в котором есть один технический оффер, цена которого каждый раз меняется под конкретный заказ перед созданием контракта.

**Следствие:** продукт и оффер в Lava не отражают конкретный курс, подписку или КП GetCourse. Они — **технический шаблон** для создания платежей.

## 3. Что убрано по сравнению с прежней логикой

Полностью исключаются:

1. запрос доступности Lava при открытии страницы оплаты;
2. проверка совпадающих продуктов;
3. поиск неиспользуемых продуктов;
4. резервирование слотов на 15 минут;
5. правило 7 суток;
6. перепривязка слотов;
7. хранение пула переиспользуемых продуктов.

Остаётся один критический сценарий:

1. получить запрос на оплату;
2. захватить блокировку;
3. обновить цену оффера;
4. создать контракт;
5. освободить блокировку;
6. дождаться webhook;
7. обновить статус заказа в GetCourse.

## 4. Границы ответственности систем

### 4.1. GetCourse

**Отвечает за:**

1. хранение каталога товаров;
2. сбор данных заказа и покупателя;
3. отображение страницы оплаты;
4. передачу запроса на создание ссылки оплаты в Chatium;
5. получение результата оплаты от Chatium;
6. выдачу доступа или запуск бизнес-логики после оплаты.

**Не отвечает за:**

1. работу с API Lava;
2. изменение стоимости оффера Lava;
3. синхронизацию цены перед созданием контракта;
4. обработку webhook Lava;
5. защиту от гонки между обновлением цены и созданием контракта.

### 4.2. Chatium

**Отвечает за:**

1. приём запроса на создание ссылки оплаты;
2. сериализацию критической секции;
3. обновление стоимости оффера Lava;
4. создание контракта в Lava;
5. сохранение связи между заказом GetCourse и контрактом Lava;
6. возврат ссылки на оплату в GetCourse;
7. приём и обработку webhook;
8. передачу статуса оплаты обратно в GetCourse;
9. логирование и мониторинг.

### 4.3. Lava.top

**Используется для:**

1. хранения технического продукта и оффера;
2. обновления стоимости оффера;
3. создания контракта;
4. предоставления ссылки на оплату;
5. отправки webhook по результату оплаты.
