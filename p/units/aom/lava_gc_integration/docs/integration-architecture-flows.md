# Схемы взаимодействия, допущения, модель в Lava

## 5. Целевая схема взаимодействия

Итоговая схема **без** предварительного шага проверки доступности.

### 5.1. Основной сценарий

1. Пользователь открывает страницу оплаты в GetCourse.
2. GetCourse показывает способ оплаты через Lava без запроса в Lava и без запроса на предварительную доступность.
3. Пользователь выбирает оплату через Lava.
4. GetCourse отправляет в Chatium запрос на создание ссылки оплаты.
5. Chatium захватывает блокировку на критическую секцию.
6. Chatium обновляет цену технического оффера в Lava под сумму заказа.
7. Chatium сразу создаёт контракт в Lava на этом оффере.
8. Lava создаёт контракт и фиксирует цену на момент создания.
9. Chatium сохраняет результат и освобождает блокировку.
10. Chatium возвращает ссылку на оплату в GetCourse.
11. GetCourse отправляет пользователя на страницу оплаты Lava.
12. После оплаты Lava отправляет webhook в Chatium.
13. Chatium нормализует событие и сообщает итоговый статус в GetCourse.
14. GetCourse завершает свой бизнес-сценарий.

### 5.2. Главное архитектурное правило

Операции «обновить цену оффера» и «создать контракт» должны выполняться внутри **одной** защищённой критической секции. Между ними нельзя допускать параллельное выполнение другого платежного запроса, который работает с тем же техническим оффером.

## 6. Базовые допущения

Схема корректна, если выполняется:

1. цена действительно фиксируется внутри контракта в момент его создания;
2. изменение цены оффера после создания контракта не меняет уже созданный контракт;
3. один и тот же технический оффер может многократно использоваться для создания разных контрактов;
4. допустимо, что пользователь видит технический продукт Lava только как внутреннюю сущность, а не как бизнес-товар;
5. нет необходимости хранить в Lava отдельные предложения под каждый товар GetCourse.

При нарушении любого пункта архитектуру нужно пересматривать.

## 7. Ограничения решения

### 7.1. Последовательное создание платежей

При одном техническом оффере запросы на создание контрактов по нему фактически **последовательны**, так как критическая секция эксклюзивна.

### 7.2. Потолок пропускной способности

Определяется временем двух последовательных вызовов:

1. обновление цены оффера;
2. создание контракта.

При высоком потоке заказов один шаблонный оффер может стать узким местом.

### 7.3. Масштабирование

Если одного шаблона мало — не возвращаться к пулу в старом смысле, а:

1. создать несколько независимых технических офферов / шаблонов;
2. распределить нагрузку между ними;
3. держать **отдельный lock на каждый шаблон** (см. также [integration-critical-section.md](./integration-critical-section.md)).

## 8. Техническая модель объекта в Lava

### 8.1. Технический продукт

В Lava заранее создаётся один технический продукт:

1. существует только для интеграции;
2. не отражает реальный товар GetCourse;
3. не требует ручного изменения оператором в обычном процессе;
4. изменения в runtime выполняет только Chatium.

### 8.2. Технический оффер

У продукта — один оффер-шаблон. Chatium меняет только необходимые для контракта поля; на текущем этапе главное — **цена**.

### 8.3. Рекомендации по именованию

Чтобы не путать с реальными товарами:

1. product name: `Chatium Payment Template`
2. offer name: `Default Dynamic Offer`

Необязательно, но упрощает сопровождение.
