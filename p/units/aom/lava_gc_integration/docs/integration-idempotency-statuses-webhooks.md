# Идемпотентность, статусы контракта, webhook

## 14. Идемпотентность

### 14.1. Запрос на создание ссылки

При повторной отправке по одному заказу Chatium должен определять повтор.

**Реализовано в `lava-payment.service`:**

1. если по сочетанию `gcOrderId` + `amount` + `currency` уже есть активный незавершённый контракт (`findActiveByGcOrderAmountAndCurrency`) — возвращается существующий `paymentUrl` (без нового вызова Lava);
2. если сумма или валюта изменились при том же `gcOrderId`, прочие активные контракты по этому заказу переводятся в `cancelled` (`deactivateActiveContractsForGcOrderId`), затем создаётся новый контракт;
3. поле `requestId` передаётся в `lava_lock_log` как корреляция, не как ключ идемпотентности ссылки.

**Рекомендации к спецификации (дополнительно):**

- для строгого контроля снаружи можно использовать `requestId` как идемпотентный ключ на стороне GetCourse;
- повторный запрос с теми же заказом, суммой и валютой не создаёт новый контракт без необходимости.

### 14.2. Webhook

События от Lava могут приходить повторно. Обработчик обязан:

1. сохранять сырой payload в Heap;
2. строить `dedupe_key`;
3. не применять одно и то же событие дважды к бизнес-состоянию и к GetCourse.

## 15. Локальные статусы контракта

| Статус | Значение |
| --- | --- |
| `created` | Контракт создан, ссылка получена |
| `redirected` | Пользователь направлен на оплату |
| `in_progress` | Ожидается webhook или финальный статус |
| `paid` | Оплата подтверждена |
| `failed` | Оплата неуспешна |
| `cancelled` | Платёж отменён, если доступно в Lava |
| `unknown` | Событие ещё не сопоставлено |

## 16. Обработка webhook Lava

Формат payload и требования к endpoint описаны в OpenAPI Lava: путь-пример **`POST /example-of-webhook-route-contract`**, схема **`PurchaseWebhookLog`**, схемы аутентификации **`BasicWebhookAuth`** и **`ApiKeyWebhookAuth`** — см. [reference/lavatop-openapi3.yaml](./reference/lavatop-openapi3.yaml) и [integration-lava-openapi-reference.md](./integration-lava-openapi-reference.md).

**Повторы от Lava:** при ответе **4xx/5xx** Lava выполняет до 19 повторных попыток с заданными интервалами (описание в том же YAML). Имеет смысл отвечать **2xx** после сохранения сырого события в Heap, а тяжёлую обработку не блокировать ответ (или укладываться в разумный SLA).

### 16.1. Маршрут

```http
POST /integrations/lava/webhook
```

Файл-роут в проекте: например `api/integrations/lava/webhook/`.

### 16.2. Обязательные шаги обработчика

1. проверить аутентификацию webhook;
2. сохранить сырой payload;
3. вычислить `dedupe_key`;
4. проверить отсутствие ранее успешной обработки того же события;
5. найти локальный контракт по `lavaContractId`;
6. обновить локальный статус;
7. отправить нормализованный статус в GetCourse;
8. пометить событие обработанным (или зафиксировать ошибку для повтора).

### 16.3. Минимальный маппинг событий

| Событие Lava | Действие |
| --- | --- |
| `payment.success` + `status=completed` | Статус контракта `paid`, уведомить GetCourse (перевод заказа в `in_work`) |
| `payment.failed` и прочие неуспешные/нерелевантные события | Не обрабатывать в бизнес-логике, в GetCourse не отправлять |

Подписки и иные типы событий — отдельное расширение спецификации.

### 16.4. Ошибки обработки

Если webhook сохранён, но бизнес-обработка упала, событие должно оставаться доступным для **повторной внутренней** обработки (флаг `processed`, поле `processing_error` — см. [integration-data-model.md](./integration-data-model.md)).
