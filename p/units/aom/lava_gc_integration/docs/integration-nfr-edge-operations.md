# НФТ, граничные случаи, операции

## 22. Нефункциональные требования

### 22.1. Производительность

1. создание ссылки — за время, приемлемое для клика пользователя;
2. блокировка удерживается минимально;
3. под блокировкой нет лишних операций.

### 22.2. Надёжность

1. lock гарантированно снимается при исключении;
2. webhook не теряется (запись в Heap до бизнес-логики);
3. повторы запросов и дубли webhook безопасны.

### 22.3. Согласованность

**Инвариант:** нельзя допускать ситуацию, в которой заказ A получает цену заказа B.

## 23. Граничные случаи

### 23.1. Многократное нажатие оплаты

Идемпотентность; по возможности возвращать уже созданную ссылку для того же заказа.

### 23.2. Два разных заказа одновременно

Сериализация через lock; параллельное изменение цены одного оффера без lock запрещено.

### 23.3. Цена обновилась, контракт не создался

Допустимо. Следующий запрос в новой секции снова выставит цену.

### 23.4. Webhook раньше финальной локальной записи контракта

Webhook сохранить; отложенная/повторная обработка до появления записи контракта.

### 23.5. Ручное изменение технического продукта в Lava

Зафиксировать инцидент, вернуть `CONFIG_ERROR` или `LAVA_UPDATE_ERROR`, не продолжать оплату до исправления конфигурации.

## 24. Операционная модель

### 24.1. Первичная настройка

1. создать технический продукт в Lava;
2. создать технический оффер;
3. сохранить `productId` и `offerId` в настройках Chatium;
4. проверить сценарий: смена цены + создание контракта.

### 24.2. Эксплуатация

Оператору не создавать продукты под каждый заказ GetCourse. Runtime — обновление цены шаблонного оффера и создание контракта.

### 24.3. Масштабирование

Несколько шаблонов офферов, распределение нагрузки, **отдельный lock на шаблон** — без возврата к старой логике пула.
