# Готовность, приёмка, решения, открытые вопросы

## 26. Минимальные критерии готовности

Система готова к запуску, если:

1. GetCourse отправляет запрос на создание ссылки в Chatium;
2. Chatium эксклюзивно блокирует критическую секцию (`@app/sync`);
3. Chatium меняет цену технического оффера в Lava;
4. Chatium создаёт контракт сразу после изменения цены;
5. Chatium сохраняет связь `gcOrderId` → `lavaContractId` в Heap;
6. Chatium принимает webhook Lava;
7. Chatium передаёт нормализованный статус в GetCourse;
8. есть метрики или измеримые логи по lock, обновлению цены и контрактам;
9. есть защита от дублей webhook;
10. есть базовая идемпотентность повторного создания ссылки.

## 27. Приёмочные сценарии

### 27.1. Успешная оплата

Цена обновлена → контракт создан → ссылка возвращена → оплата → webhook → GetCourse получил `payment.success`.

### 27.2. Одновременные запросы

Два запроса не в критической секции одновременно; цена заказа A не попадает в контракт B.

### 27.3. Ошибка обновления цены

Контракт не создан; lock снят; управляемая ошибка наружу.

### 27.4. Ошибка создания контракта

Lock снят; следующий запрос может повторить цикл; есть запись в логах.

### 27.5. Повторный webhook

Событие не применяется дважды; в логах отметка о дубле.

## 28. Зафиксированные решения

1. нет предварительной логики до клика оплаты;
2. в runtime нет подбора продуктов/офферов Lava под товар GC;
3. в Lava один технический продукт и один технический оффер;
4. цена оффера меняется непосредственно перед созданием контракта;
5. контракт фиксирует цену на момент создания;
6. защита от гонки — серверная блокировка критической секции;
7. блокировка покрывает только цикл: обновить цену → создать контракт → сохранить результат;
8. webhook обрабатывает Chatium;
9. GetCourse получает от Chatium нормализованный статус оплаты.

## 29. Открытые вопросы

Уточнить перед/в ходе разработки:

1. меняется ли только цена или ещё название/описание оффера;
2. нужен ли отдельный шаблон на каждую валюту;
3. нужен ли отдельный шаблон на тип оплаты / подписки;
4. сколько GetCourse ждёт при занятом lock;
5. должен ли повтор по тому же заказу автоматически возвращать существующую ссылку.

## 30. Следующие шаги в репозитории

Часть пунктов перенесена в модульные документы; при реализации в коде остаётся:

1. зафиксировать точные пути файлов роутов и подключение к дереву приложения;
2. описать индексы Heap и уникальные ключи в ADR или в [data.md](./data.md) после создания таблиц;
3. выбрать финальные `timeoutMs` / политику wait vs fail для lock;
4. согласовать с GetCourse формат retry и таймауты HTTP;
5. добавить интеграционные и бизнес-тесты (см. `inner/docs` по тестированию и существующую страницу тестов проекта).
