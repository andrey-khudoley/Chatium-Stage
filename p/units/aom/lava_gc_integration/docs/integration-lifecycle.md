# Жизненный цикл операций

## 12.1. Успешное создание ссылки

1. GetCourse отправляет запрос `payment-link`.
2. Chatium валидирует входные данные.
3. Chatium захватывает lock (`runWithExclusiveLock` или политика с `tryRunWithExclusiveLock`).
4. Chatium получает `productId` и `offerId` из конфигурации (настройки Heap / workspace).
5. Chatium отправляет в Lava запрос на обновление цены оффера.
6. После успешного ответа — немедленно запрос на создание контракта.
7. Chatium получает `contractId` и `paymentUrl`.
8. Chatium сохраняет локальную запись о контракте и связи с заказом GetCourse (Heap).
9. Chatium освобождает lock.
10. Chatium возвращает `paymentUrl` в GetCourse.
11. GetCourse открывает пользователю страницу оплаты Lava.

## 12.2. Ошибка обновления цены

1. GetCourse вызывает `payment-link`.
2. Chatium захватывает lock.
3. Обновление цены в Lava завершается ошибкой.
4. Контракт не создаётся.
5. Ошибка пишется в лог (`ctx.account.log` / проектный logger).
6. Lock освобождается.
7. GetCourse получает `LAVA_UPDATE_ERROR`.

## 12.3. Ошибка создания контракта

1. Цена уже обновлена в Lava.
2. Создание контракта завершилось ошибкой.
3. Логирование, освобождение lock.
4. Следующий платёжный запрос снова обновит цену в **новой** критической секции.

**Важно:** «зависшая» цена оффера после ошибки не считается фатальной — следующий запрос обязан выставить свою цену под lock.

## 12.4. Параллельные запросы

1. Запрос A захватывает lock.
2. Запрос B приходит одновременно.
3. B ждёт или получает отказ (политика из [integration-critical-section.md](./integration-critical-section.md)).
4. A завершает обновление цены и создание контракта.
5. A освобождает lock.
6. B входит в критическую секцию только после A.
