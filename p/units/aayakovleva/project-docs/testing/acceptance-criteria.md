# Бизнес-приёмочные критерии

Этот документ задаёт **критерии приёмки от заказчика** (Николай Яковлев / Ольга Шавковская): что должно работать на боевой странице оплаты школы, чтобы считать проект готовым к закрытию по договору. Технические сценарии прогона тестов разработки - в [testing-strategy](../../../../saas/gw/lifepay/docs/gateway/testing-strategy.md); бизнес-логика отображения - в [payment-scheme](../../sbp-client/docs/architecture/payment-scheme.md).

Принцип: каждый критерий формулируется в терминах, которые заказчик может проверить сам, без чтения кода и без понимания внутренней архитектуры. Технический исполнитель отчитывается по каждому пункту записью в журнал проекта со скриншотом или ссылкой на воспроизводимый сценарий.

## 1. Раскладка способов оплаты на странице школы

- [ ] **Сумма заказа меньше 50 000 ₽.** На странице оплаты видны все способы оплаты, в порядке: СБП через LifePay, ОТП Банк, оплата из-за границы (Lava Top), затем стандартные методы GetCourse (GetCourse Pay - Тинькофф, Сбер, прочие банки, рассрочки, карты РФ). Все способы технически выбираемы.
- [ ] **Сумма заказа от 50 000 ₽.** На первом экране остаются только СБП, ОТП и Lava Top. Остальные способы свёрнуты в раскрывающийся блок «Другие методы» (серый, ненавязчивый). Раскрытие блока работает кликом, родные сценарии методов GetCourse не сломаны.
- [ ] **Мобильная вёрстка.** Раскладка корректно отображается на Chrome Android, Safari iOS, в WebView Telegram (если школа гонит туда трафик). Кнопки кликабельны, переход в банковское приложение по СБП-deeplink работает.

## 2. Сценарий «Покупатель оплачивает курс через СБП»

- [ ] **Десктоп.** Покупатель открывает страницу оплаты, выбирает «СБП», на странице появляется QR-код. Покупатель сканирует код банковским приложением, подтверждает оплату. Через несколько секунд в личном кабинете школы заказ переходит в статус «оплачен», открывается доступ к курсу, покупатель получает письмо с инструкциями (родной триггер GetCourse).
- [ ] **Мобильное устройство.** Покупатель открывает страницу оплаты в мобильном браузере, выбирает «СБП», происходит автоматический переход в установленное банковское приложение через СБП-deeplink. После оплаты результат тот же: заказ оплачен, доступ открыт.
- [ ] **Фискальный чек.** Покупатель получает фискальный чек на email (через ОФД, автоматически из `my.life-pay.ru` при включённой автофискализации). Чек содержит наименование курса, сумму, ставку НДС в соответствии с настройками школы.

## 3. Сценарий «Покупатель оплачивает через ОТП-рассрочку»

- [ ] Способ «ОТП Банк» виден на первом экране страницы оплаты при любой сумме заказа в скоупе (правило раскладки §1).
- [ ] Клик по способу запускает родной флоу ОТП, реализованный школой ранее (виджет проекта не подменяет обработчик ОТП, только перестраивает порядок).
- [ ] После одобрения заявки покупатель возвращается на страницу подтверждения и получает доступ к курсу.

## 4. Сценарий «Покупатель оплачивает из-за границы»

- [ ] Способ «Оплата из-за границы» (Lava Top) виден на первом экране при любой сумме в скоупе.
- [ ] Клик запускает родной флоу Lava Top без вмешательства виджета проекта.
- [ ] После оплаты статус заказа обновляется штатными средствами Lava Top + школы (этот канал в скоуп текущего проекта не входит, проверяется только что виджет его не сломал).

## 5. Стандартные методы GetCourse (карты, рассрочки, банки)

- [ ] Все стандартные методы оплаты GetCourse (Тинькофф через GetCourse Pay, Сбер, прочие банки РФ, рассрочки, карты) остаются работоспособными после установки виджета. При сумме < 50 000 ₽ они выбираемы напрямую, при сумме ≥ 50 000 ₽ - через раскрытие блока «Другие методы».
- [ ] Виджет не подменяет ни одного нативного обработчика GetCourse, кроме обработчика СБП через LifePay.

## 6. Сценарий «Что-то пошло не так»

- [ ] **Покупатель не довёл оплату до конца.** Заказ остаётся в исходном статусе («ожидает оплаты»). Покупатель может вернуться на страницу позже и оплатить заново, статус корректно обновляется при успешной оплате.
- [ ] **Сбой LifePay (временный).** На странице появляется понятное сообщение: «Сервис временно недоступен, попробуйте через несколько минут». Заказ не теряется, повторная попытка работает.
- [ ] **Webhook задержался.** Если LifePay подтвердил оплату, но webhook пришёл с задержкой более минуты, при обновлении страницы заказа в школе статус всё равно отображается корректно (после получения webhook).

## 7. Операционная приёмка

- [ ] **Доступ к админке payments-gateway.** Андрей передаёт Ольге короткую инструкцию (1 страница): где смотреть статистику вызовов, что делать при сбое, к кому обращаться.
- [ ] **Журнал событий.** В админке payments-gateway видна история создания счетов с фильтром по дате, статусу, ID заказа.
- [ ] **Ежедневная сводка.** Ольга или администратор школы получает на email/в чат ежедневную сводку: «За сутки создано N счетов, оплачено M, неоплачено K, ошибок L».

## 8. Сроки и критерий закрытия проекта

- [ ] Запуск виджета на боевой странице школы выполнен до **15.05.2026** (дедлайн по договору; см. [корневой файл проекта](../brief.md)).
- [ ] В течение **7 дней** после запуска нет инцидентов уровня «оплата прошла, статус не обновился» в боевом потоке.
- [ ] Заказчик подтверждает приёмку письменным сообщением (email или сообщение в мессенджере), после чего проект переводится в статус `done`.
