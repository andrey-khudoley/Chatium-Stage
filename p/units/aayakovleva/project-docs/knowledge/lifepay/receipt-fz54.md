# LifePay — Фискализация (54-ФЗ)

У клиента подключена облачная касса LifePay. При любом приёме онлайн-оплаты чек обязателен. Конкретный способ фискализации зависит от выбранного API и настроек кабинета.

## Матрица сценариев

| Сценарий приёма                         | Кабинет            | Способ фискализации                              | Кто инициирует              |
| --------------------------------------- | ------------------ | ------------------------------------------------ | --------------------------- |
| СБП через `/v1/bill` (вариант B)        | `my.life-pay.ru`   | Автоматическая фискализация при настройке в ЛК   | LifePay сам после оплаты    |
| ECOM через `/v1/invoices/` (вариант A)  | `home.life-pay.ru` | Параметр `send_receipt_through` в invoice        | LifePay по флагу из запроса |
| Любой сценарий, чек со сложным составом | оба                | Cloud Print API отдельным запросом после webhook | Backend интегратора         |

Развилка вариантов A и B описана в [js-integration](js-integration.md), кабинеты в [cabinets](cabinets.md).

---

## Вариант 0. Автоматическая фискализация в `my.life-pay.ru` (для сценария B)

Этот путь идёт «бесплатно» в комплекте с облачной кассой и работает для счетов, созданных через `POST /v1/bill` с `method: "sbp"`. После успешной оплаты LifePay сам формирует чек и отправляет его в ОФД, без отдельного вызова Cloud Print API.

### Что включить в `my.life-pay.ru`

В разделе «Фискализация → Сотрудники» проставить галочку напротив логина владельца. Это ключевой флаг, без него автофискализация не запустится.

В разделе «Фискализация → Настройки» выбрать данные чека для автоматической отправки (https://support.life-pay.ru/ru/knowledge_base/article/154/category/5/#nastroiki). В разделе «Касса → Настройки» включить автоматическое закрытие смены (открытие можно не настраивать, смена откроется сама при первой операции).

### Ограничение

Автоматический чек поддерживает только одну товарную позицию. В случае сложного состава (несколько товаров, разные ставки НДС, агентские схемы) автофискализация неприменима, нужен ручной чек или Cloud Print API.

Для проекта Ольги этого ограничения, по всей видимости, нет: один курс = одна позиция в чеке. Сценарий рабочий.

## Вариант 1. Через параметр в invoice (ECOM API, для сценария A)

При создании invoice через ECOM API можно указать `send_receipt_through`, и LifePay сам сформирует и отправит чек через ОФД.

```json
{
  "order_id": "order_12345",
  "amount": 5000.00,
  "currency_code": "RUB",
  "service_id": "87875",
  "name": "Иван Иванов",
  "email": "client@example.com",
  "send_receipt_through": "email",
  ...
}
```

| Значение `send_receipt_through` | Канал                   |
| ------------------------------- | ----------------------- |
| `"email"`                       | Чек на email покупателя |
| `"phone"`                       | Чек на телефон (SMS)    |

> ⚠️ Конкретный формат передачи позиций чека (items) прямо в ECOM invoice — уточнять в документации:  
> 📄 https://docs.life-pos.ru/docs/api/ipsp_api_v1/create-invoice

## Вариант 2. Cloud Print API (отдельный запрос, универсальный)

Если автофискализация не подходит (несколько позиций, разные ставки НДС, агентская схема) или ECOM не поддерживает items напрямую, фискализация делается отдельным запросом сразу после подтверждения успешного webhook. Этот путь работает поверх любого сценария приёма (A или B).

**Endpoint:** `POST https://api.life-pay.ru/v1/cloud-print`

### Структура запроса

```json
{
  "apikey": "{apikey}",
  "login": "{login}",
  "type": "sale",
  "customer_phone": "79161234567",
  "customer_email": "client@example.com",
  "purchase": {
    "products": [
      {
        "name": "Курс «Имя продукта»",
        "price": 5000.0,
        "quantity": 1,
        "unit": "шт",
        "tax": "none"
      }
    ]
  }
}
```

### Поля объекта `products[]`

| Поле       | Тип       | Описание                                    |
| ---------- | --------- | ------------------------------------------- |
| `name`     | string    | Наименование товара/услуги                  |
| `price`    | float     | Цена единицы в рублях                       |
| `quantity` | int/float | Количество                                  |
| `unit`     | string    | Единица измерения: `"шт"`, `"услуга"` и др. |
| `tax`      | string    | Ставка НДС (см. таблицу ниже)               |
| `type`     | int       | Тип позиции (опционально)                   |

### Ставки НДС (`tax`)

| Значение | Ставка                 |
| -------- | ---------------------- |
| `none`   | Без НДС                |
| `vat0`   | НДС 0%                 |
| `vat10`  | НДС 10%                |
| `vat20`  | НДС 20%                |
| `vat110` | НДС 10/110 (расчётная) |
| `vat120` | НДС 20/120 (расчётная) |

### Тип чека (`type`)

| Значение   | Описание         |
| ---------- | ---------------- |
| `"sale"`   | Приход (продажа) |
| `"refund"` | Возврат прихода  |

### Параметр `tax_system`

Если бизнес работает на нескольких системах налогообложения — создавать отдельные чеки с разными `tax_system` для каждой группы товаров.

## Когда отправлять чек

Порядок зависит от выбранного варианта фискализации.

### При автофискализации (вариант 0 или 1)

```
1. Покупатель оплатил
       ↓
2. LifePay сам формирует чек и отправляет в ОФД
       ↓
3. LifePay отправил webhook на backend
       ↓
4. Backend верифицировал подпись, статус оплаты
       ↓
5. Backend → GetCourse CRM: deal_status = paid
```

В этом случае backend не вызывает Cloud Print API: чек уже ушёл.

### При Cloud Print (вариант 2)

```
1. Покупатель оплатил
       ↓
2. LifePay отправил webhook
       ↓
3. Backend верифицировал подпись, статус оплаты
       ↓
4. Backend → POST /v1/cloud-print с items
       ↓
5. ОФД получил чек через облачную кассу LifePay
       ↓
6. Backend → GetCourse CRM: deal_status = paid
```

> По законодательству чек должен быть отправлен покупателю до момента расчёта или сразу после. Передача данных в ОФД происходит автоматически через облачную кассу LifePay в обоих сценариях.
