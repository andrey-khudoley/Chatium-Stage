## Два сценария приёма СБП

LifePay даёт два разных способа выставлять СБП-оплату, и они идут через разные кабинеты и разные API.

### Сценарий А. СБП через ссылку из фискального кабинета (`my.life-pay.ru`)

Подключается «бесплатно» вместе с облачной кассой, комиссия 0.4% (Ольга так и описывала на созвоне 08.05). Ссылка формируется одним из способов:

вручную через раздел «Клиенты → Выставить счёт» (динамическая ссылка), статически через регулярную ссылку, программно через старый API `POST https://api.life-pay.ru/v1/bill` с авторизацией `apikey + login` в теле запроса.

Программный путь через `POST /v1/bill` с параметром `method: "sbp"` (значение по умолчанию) возвращает в ответе пару ссылок:

`paymentUrl` — ссылка на `qr.nspk.ru` (для отображения как QR или редиректа в банковское приложение).
`paymentUrlWeb` — ссылка на `web.qr.nspk.ru` (веб-страница СБП с выбором банка).

`callback_url` для уведомлений о статусе можно передавать прямо в теле запроса на создание счёта, или задать глобально в ЛК (Настройки → Разработчикам). Webhook идёт через единый раздел `/notification/`, формат и retry-схема описаны там же.

Автоматическая фискализация работает при условии, что в разделе «Фискализация → Сотрудники» проставлена галочка напротив логина. Ограничение: автоматический чек может содержать только одну товарную позицию. Если позиций несколько, чек придётся формировать вручную или через `Cloud Print API`.

### Сценарий Б. СБП через эквайринг (`home.life-pay.ru`)

Подключается отдельной заявкой, требует подписания договора эквайринга. Использует новый ECOM API `api-ecom.life-pay.ru/v1/`. Кроме СБП покрывает карты, Mir Pay, Yandex Pay. Условия по комиссиям и совместимость с действующей облачной фискализацией на 13.05.2026 уточняются у инженеров.

## Принципиальный вывод для проекта Ольги

Если по итогу подключение эквайринга затянется или окажется невыгодным по комиссии, есть рабочий fallback: программное выставление СБП-ссылок через старый API `POST /v1/bill` из `my.life-pay.ru`. Тогда вместо ECOM `POST /v1/invoices/` используется `POST /v1/bill`, webhook идёт на тот же сервер по схожему формату, фискализация остаётся через текущую облачную кассу.

Финальное решение принимается после ответа инженеров: сравнить комиссии и условия двух сценариев, выбрать более подходящий для оборота школы.
