# gw-client (Payments Client)

## Назначение

Универсальный клиентский слой к нескольким платёжным гейтвеям. Исходно — клиентская панель LifePay (implementation-plan §1.8), с 2026-05-28 расширен до многогейтвейной архитектуры. Одно Chatium-приложение совмещает: хранилище секретов магазинов (LifePay и Lava.Top), серверную прокладку `POST /api/lp/invoke` (исторический URL) между UI/JS и нужным payments-gateway с журналом запросов, приёмники входящих webhook от обоих гейтвеев.

## Поддерживаемые гейтвеи

- **LifePay** — серверный gateway `p/saas/gw/lifepay` (контур `bills_v1`). Заголовки авторизации: `X-Lp-Apikey`, `X-Lp-Login`. Операции: `createBill`, `getBillStatus`, `cancelBill`. Webhook: `/web/webhook` (multipart, токен в query).
- **Lava.Top** — серверный gateway `p/saas/gw/lavatop` (контур `invoices_v1`). Заголовок авторизации: `X-Lava-Apikey`. Операции: `createInvoice`, `getInvoiceStatus`, `listProducts`, `updateOfferPrice`. Webhook: `/web/webhook-lavatop` (JSON, секрет в `X-Api-Key` / Basic).
- **GetCourse (GC)** — серверный gateway `p/saas/gw/gc`. Заголовки авторизации: `X-Gc-School-Api-Key`, `X-Gc-School-Host`. Каталог операций динамический: список enabled-операций загружается с `GET /api/v1/operations` GC-гейтвея при SSR; изменение набора операций на стороне `p/saas/gw/gc` не требует правок клиента. Для операций GC в теле `POST /api/lp/invoke` обязателен `httpMethod` (`'GET'` или `'POST'`).

## Важно

- Платформа: Chatium. Серверная часть управляется платформой.
- Стек фиксирован платформой, новые зависимости не добавляем.
- Деплой происходит автоматически после пуша.

## Текущее состояние

- Heap-настройки LifePay: `lp_apikey`, `lp_login`, `lp_webhook_token`, `gateway_base_url` (с валидациями).
- Heap-настройки Lava.Top: `lava_test_apikey`, `lava_base_url` (default `https://gate.lava.top`), `lava_webhook_secret` (≥ 16 символов).
- Heap-настройки GetCourse: `gc_base_url`, `gc_test_school_api_key`, `gc_test_school_host`, `gc_enabled` (с валидациями; `gc_test_school_api_key` маскируется в exit-логах).
- Все настройки редактируются на `/web/admin`.
- Heap-таблицы: `request_log` (с опциональным полем `gatewayId`), `webhook_log` (с опциональными `gatewayId`, `gatewayExternalId`), `webhook_idempotency`, `panel_access`, `panel_invites`, `settings`, `logs`, `PaymentPageMethods` (`t__lifepay-sbp-client__ppmethod__k9Xp2mL4`, методы страницы оплаты GC), `GcDealCache` (кэш суммы сделки, TTL 60 с). Ключи таблиц сохранены исторически (`t__lifepay-sbp-client__*`) — переименование сломает данные; технический долг, не блокирующий.
- Серверная прокладка `POST /api/lp/invoke`: тело `{ gatewayId: 'lifepay' | 'lavatop' | 'gc', op, args, httpMethod? }`. **`gatewayId` обязательно**. Для GC дополнительно обязателен `httpMethod` (`'GET'` или `'POST'`). Диспатчер по `gatewayId` (`lib/gateway/invokeDispatcher.ts`) выбирает клиент: LifePay (`lib/gateway/invokeClient.ts`), Lava.Top (`lib/gateway/lavatopClient.ts`) или GC (`lib/gateway/gcClient.ts`). Один вызов через `@app/request` к соответствующему gateway, без ретраев; журнал в `request_log` с `gatewayId`.
- Приёмники webhook:
  - `POST /web/webhook?token=...` — LifePay; анонимный + токен в query; `req.formData()` + фоллбэки; дедупликация через `runWithExclusiveLock` по `number`; запись в `webhook_log` с `gatewayId='lifepay'`. **С 2026-06-02 (bugfix):** downstream createDeal использует стратегию persist-first: `lookupDealResolve` (`lib/gateway/dealResolveCache.ts`) читает `dealNumber`/`email`/`amount` из `createBill`-записи `request_log` без новых GC-вызовов; при промахе — fallback `resolveGcDeal`. В поле `deal_number` передаётся GC поле `number` (извлекается из `resolveGcDeal.dealNumber`), а не correlationId/id — исправлен INVOKE_GC_SEMANTIC_ERROR.
  - `POST /web/webhook-lavatop` — Lava.Top; анонимный + секрет в `X-Api-Key` / Basic Authorization (значение `lava_webhook_secret`); JSON-тело `PurchaseWebhookLog` (`contractId`, `eventType`, `status`); дедупликация по композитному ключу `lavatop:contractId:eventType:status`; запись в `webhook_log` с `gatewayId='lavatop'` и `gatewayExternalId=contractId`. GET на тот же URL возвращает статус доступности (без раскрытия секрета).

- **GC deal-резолвер** (`lib/gateway/gcDealResolver.ts`): два резолвера — `resolveGcDeal(ctx, dealId, correlationId?)` (полный, intent-поток и webhook-fallback: getDealFields → getUserFields; **с 2026-06-02** возвращает `dealNumber` = GC поле `number` сделки наряду с amount/currency/email/title/userId/positions; при пустом `number` — `dealNumber` отсутствует без hard-fail; записывает GC-вызовы в `request_log` через `recordRequestLog` с привязкой по correlationId) и `resolveGcDealAmount(ctx, dealId)` (лёгкий, только сумма: один вызов getDealFields, без email/валюты; оплаченная → ok:false; используется только в config-эндпоинте с кэшом). Email в логи не пишется (PII). `resolveGcDealAmount` логирует `entry` и `fields_extracted` на severity 7 (debug).
- **Кэш резолв-данных** (`lib/gateway/dealResolveCache.ts`): **новый с 2026-06-02**. `lookupDealResolve(ctx, correlationId)` — читает последнюю `createBill`-запись по correlationId из `request_log` (через `findLatestCreateBillByCorrelationId`), извлекает `dealNumber`, `email`, `amount`. Реализует persist-first стратегию в webhook шаге 5a: при промахе — fallback `resolveGcDeal`. Новая колонка `gcDealNumber` в `request_log` (`tables/requestLog.table.ts`) — Optional String, пишется в `createBill`-записях deal-потока через `recordRequestLog` в `api/widgets/intent-by-deal.ts`.
- **Кэш суммы GC** (`lib/gateway/gcDealCache.ts`, `tables/gcDealCache.table.ts`): **новый с 2026-06-01**. TTL=60с, upsert без лока (гонка безвредна). Поле `amount` = рублёвый эквивалент суммы заказа (non-RUB конвертируются через `convertToRub`). Устаревшие записи удаляются при cache miss (lazy eviction). Используется только в `POST /api/widgets/config`.
- **Конвертер валют** (`lib/rates/currencyConverter.ts`): `convertRubTo(amount, currency)` — RUB→RUB (тождество), RUB→USD/EUR. `convertToRub(amount, currency)` — обратная конвертация валюты в рубли. Приватный `getRubForOne` — единый источник курса: ручной курс из настроек (`widget_lavatop_manual_rate_usd`/`eur`) → курс ЦБ РФ → `RATE_UNAVAILABLE`.
- **Lava.Top deal-orchestrator** (`lib/gateway/lavatopDealIntent.ts`): `handleLavatopDealIntent` — фильтр суммы → конвертация → `runWithExclusiveLock('lavatop-offer:'+offerId)` → `updateOfferPrice` → `createInvoice`. Единый оффер (`widget_lavatop_offer_id` + `widget_lavatop_product_id`) для всех заказов. **С 2026-06-01:** пишет успешный `createInvoice` в `request_log` (прежде не писал) — является источником кэша идемпотентности для Lava.Top-ветки.
- **Идемпотентность `createBill`/`createInvoice`** в deal-потоке (`api/widgets/intent-by-deal.ts`, обе ветки — LifePay и Lava.Top): повторное нажатие кнопки на тот же заказ в окне 30 минут возвращает ранее созданную `paymentUrl` без нового вызова gateway. Реализовано через `lib/gateway/idempotentBillCache.ts` (кэш-lookup по `request_log`) + новый метод `repos/requestLog.repo.ts → findLatestOkForIdempotency`. Ключ идемпотентности — LifePay: `orderNumber` + `amount`; Lava.Top: `orderNumber(=correlationId=dealId)` + `currency` + `amount`. Гонка конкурентных кликов закрыта `runWithExclusiveLock`: LifePay — `'gw-client:bill-idempotency:lifepay:<orderNumber>'`; Lava.Top — существующий `'lavatop-offer:<offerId>'`. Константы локов вынесены в `lib/gateway/constants.ts` (BILL_LOCK_WAIT_MS/BILL_LOCK_MAX_DURATION_MS=20s, LAVATOP_LOCK_WAIT_MS/LAVATOP_LOCK_MAX_DURATION_MS=35s).
