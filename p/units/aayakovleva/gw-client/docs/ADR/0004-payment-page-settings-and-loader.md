# ADR-0004: Панель управления платёжной страницей GetCourse — настройки и централизованный loader

Дата: 2026-06-06

## Контекст

Кастомизация страницы оплаты GetCourse (dealPay) была реализована через 13 отдельных JS-скриптов, захардкоженных в конструкторе GC у каждой подключённой школы. Обновления требовали ручного прохода по каждой школе. Нужны: централизованное управление настройками отображения методов оплаты и единая точка раздачи скриптов, не требующая ручного обновления на стороне школ.

## Решение

### 1. Единый клиентский loader вместо 13 скриптов

Школа подключает один тег `<script src="pp-loader.js">` и прелоадер-оверлей с `id="pp-preloader"` (полноэкранный fixed со спиннером). Лоадер использует тот же id: `showPreloader()` не создаёт дубль, если оверлей уже вставлен в HTML (показывается мгновенно, перекрывая «сырую» страницу), `hidePreloader()` скрывает его после загрузки скриптов. В режиме редактирования конструктора GC (`?editMode=1` в URL) лоадер полностью отключён ранним `return` сразу после idempotency-guard: не запрашивает конфиг, не инжектит стили и скрипты, не показывает прелоадер — кастомизация не должна мешать редактору. Прелоадер, вставленный школой в HTML, в этом режиме прячет встроенный inline-`<script>` самого сниппета прелоадера (независимо от лоадера — на случай, если `pp-loader.js` не загрузится). Loader самостоятельно запрашивает конфиг с сервера и подгружает нужные скрипты и стили. Все обновления скриптов и стилей централизованы на стороне gw-client и применяются ко всем школам без изменений в конструкторе GC.

При ошибке получения конфига или таймауте loader показывает оверлей «Ошибка рендеринга», полностью перекрывающий содержимое страницы — это сознательное продуктовое требование: страница не отображается без валидного конфига.

### 2. Публичный эндпоинт конфига без авторизации

Эндпоинт `POST /api/payment-page/config` работает без авторизации и с заголовком `Access-Control-Allow-Origin: *`.

Обоснование: loader исполняется в браузере покупателя на стороннем домене GC-школы; домены у школ разные, поэтому один loader должен работать cross-origin. Тело ответа содержит исключительно визуальный конфиг (без accountId и секретов).

Формат ответа: `{ success: true, ... }` при успехе и `{ success: false }` с HTTP 200 при любой ошибке. Loader проверяет `json.success`, а не `response.ok`, что позволяет обрабатывать ошибки на уровне прикладной логики.

**Уточнение (2026-06-07): публичный конфиг-эндпоинт не сидирует методы.** `config.ts` вызывает `getPaymentPageMethods(ctx, { seed: false })` — только чтение из таблицы без reconcile. Сид выполняется исключительно в admin-путях: SSR главной панели (`index.tsx`) и `GET /api/payment-page/settings-get`. Причина: сид в публичном эндпоинте создавал риск конкурентного initialisation при одновременных запросах покупателей до первого admin-визита. Следствие: если admin ни разу не открывал вкладку «Страница оплаты», публичный конфиг вернёт пустой список методов — это ожидаемое, корректное поведение (страница GC не кастомизирована).

### 3. Настройки в Heap по слоям api → lib → repos

Настройки страницы оплаты хранятся в той же Heap key-value settings (по ADR-0002), ключи `payment_page_general` и `payment_page_methods`. Для каждого метода оплаты хранятся: `enabled`, границы суммы `min`/`max`, `imageUrl`, раздел (section), порядок (order), бело/чёрный список офферов, `label`.

Слои сохраняются без изменений: `HTTP → API → lib → repos → Heap`.

**Уточнение (2026-06-07): seed fast-path по countSystem.** `ensureSeeded` использует `countSystem()` (`countBy({ isSystem: true })`), а не `countAll()`, для определения необходимости досева. Это исключает сценарий, при котором наличие кастомных методов маскировало недосев системных (например, после частичного сбоя первоначального reconcile). Метод `countSystem()` добавлен в `repos/paymentPageMethods.repo.ts`. Аналогично, `repos/paymentPageMethods.repo.ts:list()` логирует обнаруженные дубликаты `methodKey` (severity 3).

### 4. Разделение ответственности: отображение и платёжный допуск

Ключевое архитектурное решение:

| Группа ключей                 | Назначение                                | Изменять                            |
| ----------------------------- | ----------------------------------------- | ----------------------------------- |
| `payment_page_*`              | Отображение методов оплаты на странице GC | только в контексте UI/UX страницы   |
| `widget_*` (lifepay, lavatop) | Платёжный допуск к intent-флоу            | только в контексте платёжной логики |

Группы не объединяются и не удаляются: слияние сломало бы платёжный допуск. Переиспользуются только типы и pure-функции: `offerListType`/`AllowedOffer`, CORS-обёртка, success-обёртка ответа.

Это ответ на запрос «убрать дублирующие настройки»: дублирование устранено для отображения (раньше параметры были захардкожены в 13 скриптах, теперь управляются через `payment_page_methods`), платёжный допуск не затронут.

### 5. Хостинг статики через userscripts/

Ассеты (pp-loader.js, pp-style.css, pp-script-NN.js) хранятся в `userscripts/` как плоские файлы и отдаются платформой cross-origin с заголовком `Content-Type: application/javascript` и `CORS: *` — это поведение подтверждено эмпирически на `userscripts/common.js`.

CSS загружается loader-ом как текст и инжектируется через `<style>` — это устраняет зависимость от `Content-Type: text/css` на стороне платформы.

Конфиг (список методов, параметры отображения) является динамическим и раздаётся через API, а не как статика.

### 6. Метод, выделенный по умолчанию (defaultMethod)

GetCourse штатно помечает один метод (`#sber-pay`) классом `.picked` при рендере формы dealPay. Наш CSS подсвечивает `.picked`, поэтому метод выглядел выбранным сразу при загрузке (кнопка оплаты активна), хотя покупатель его не выбирал. Решение: поле `defaultMethod` в `payment_page_general` (methodKey или `''` = ни один) управляет стартовым выделением из админки.

Снятие навязанного выделения и применение настройки сделано в `userscripts/pp-script-05.js` (`normalizeDefaultSelection()`), а не в CSS: только JS может одновременно снять `.picked` с навязанного GC метода и поставить его на настроенный, синхронизировав состояние кнопки оплаты (`pay-disabled`). Чисто-CSS-решение не смогло бы отличить «навязанный GC» выбор от настроенного и от ручного выбора покупателя.

После ручного выбора покупателя (флаг `userHasPicked`, ставится в обработчиках клика) нормализация становится no-op — иначе поздние ретраи `wire()` (1600/3600 мс) перетирали бы выбор покупателя. Метод проходит весь путь через `general` без изменений конфиг-эндпоинта и лоадера (general передаётся целиком).

## Последствия

**Плюсы:**

- обновления скриптов и стилей применяются ко всем школам без изменений на стороне GC;
- управление отображением методов оплаты не требует деплоя скриптов;
- единый источник истины для отображения: `payment_page_methods` вместо 13 захардкоженных наборов параметров;
- скрипты fallback-safe: при ошибке конфига loader блокирует страницу управляемым оверлеем, а не показывает некорректный UI.

**Минусы / риски:**

- оверлей «Ошибка рендеринга» полностью блокирует страницу при недоступности gw-client — by design, но требует высокой доступности сервиса;
- школы зависят от доступности gw-client для работы страницы оплаты;
- live e2e-проверка требует деплоя через sync (локально не проверяется);
- 13 скриптов хранятся в vault как источник и в `userscripts/` как served-копии — при изменении требуется синхронизация обоих мест.
