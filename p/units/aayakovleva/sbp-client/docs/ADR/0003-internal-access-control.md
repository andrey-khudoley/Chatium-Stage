# ADR 0003 — Внутренняя система прав доступа к панели `sbp-client`

## Статус

Принято 24-05-2026. Заменяет схему «всё под `requireAccountRole(ctx, 'Admin')`», описанную в [implementation-plan §1.8](../../../../../saas/gw/lifepay/docs/gateway/implementation-plan.md) и реально не реализованную в коде (auth-разрыв из аудита 24-05-2026: 6 эндпоинтов `api/lp/*` помечены «Admin-only» в комментариях, но проверка прав не вызывается).

> ⚠️ **Статус реализации: НЕ начата (на 24-05-2026).** Решение принято, но в коде реализованы только две Heap-таблицы (`panelAccess`, `panelInvites`) — и они пока нигде не используются. Утилита `requireInternalAccess`, функции управления инвайтами (`lib/access/*`), роуты `/web/access/invite`, `/web/forbidden`, `/api/access/*` и миграция эндпоинтов `api/lp/*` **ещё не написаны**. До завершения этой работы auth-разрыв остаётся открытым: `api/lp/*` (включая `/api/lp/invoke`) принимают запросы **без авторизации**, а комментарии «Admin-only» в коде не соответствуют действительности. Пошаговый план — в [implementation-plan §1.11](../../../../../saas/gw/lifepay/docs/gateway/implementation-plan.md) и в [docs/data.md](../data.md#план-реализации-внутренней-авторизации-adr-0003).

## Контекст

Панель `p/units/aayakovleva/sbp-client` несёт несколько ролей: серверная прокладка к payments-gateway (`/api/lp/invoke`), приёмник webhook от LifePay, журнал запросов и webhook, дашборд для оператора, форма ручного создания счёта. Доступ нужен сотрудникам школы (для мониторинга и ручных операций), а не только Андрею-разработчику.

Изначальная схема в спецификации - всё под `requireAccountRole(ctx, 'Admin')`. У этой схемы три практических проблемы:

1. **Грубая гранулярность.** Admin в Chatium - это владелец workspace; давать его сотруднику школы означает дать ему полный доступ ко всему воркспэйсу, включая чужие приложения и системные настройки. Это неприемлемо.
2. **Чатиум-роль «User» - слишком широкая.** Любой авторизованный пользователь Chatium-аккаунта (в т.ч. покупатель курса школы, который зарегистрировался для оплаты) пройдёт `requireAccountRole(ctx, 'User')`. Это даст ему доступ к панели.
3. **Аудит 24-05-2026.** Фактическая реализация в коде даже не делает заявленную проверку - все 6 эндпоинтов `api/lp/*` принимают любые HTTP-запросы без auth. До выкатки в боевую среду это безопасности проблема: `/api/lp/invoke` позволяет создавать счета в LifePay от имени школы.

## Решение

Двухуровневая модель:

1. **Базовая проверка - `requireRealUser(ctx)`** на всех защищённых страницах и API. Гарантирует, что вызывает реальный (не анонимный, не бот) авторизованный пользователь Chatium-аккаунта. Анонимные → редирект на форму входа (стандартный `/s/auth/signin?back=...`).
2. **Внутренняя проверка - `requireInternalAccess(ctx)`** (новая утилита панели). После `requireRealUser`:
   - Если `ctx.user.is('Admin')` - проход (Admin-владелец всегда имеет доступ).
   - Иначе ищем запись в Heap-таблице `panel_access` по `userId`. Запись есть → проход.
   - Записи нет → ошибка 403 / редирект на `/web/forbidden`.

Доступ для не-Admin сотрудников выдаётся через **одноразовые пригласительные ссылки**, генерируемые Admin'ом:

- Admin в интерфейсе панели жмёт «Создать ссылку доступа» → генерируется случайный токен (≥ 32 символа, `@app/nanoid`), создаётся запись в Heap `panel_invites`, в UI отображается полный URL `https://.../web/access/invite?token=<...>`.
- Admin копирует ссылку и передаёт сотруднику любым каналом (мессенджер, почта).
- Сотрудник переходит:
  - **Не авторизован** → редирект на `/s/auth/signin?back=/web/access/invite?token=<...>`. Токен **не расходуется**.
  - **Авторизован** → отрисовка страницы «Получить доступ к панели управления интеграцией LifePay» с информацией о текущем аккаунте и кнопкой «Подтвердить».
- На клик «Подтвердить» - `POST /api/access/consume-invite` под `runWithExclusiveLock` по ключу токена: атомарно проверяет, что инвайт активен (не использован, не отозван, не просрочен), создаёт запись в `panel_access` с `userId = ctx.user.id`, помечает инвайт `usedAt + usedByUserId`. На последующих заходах в панель сотрудник проходит `requireInternalAccess`.

## Альтернативы и почему не выбраны

| Альтернатива | Почему отклонена |
|---|---|
| Использовать роль `Staff` Chatium-аккаунта | Та же проблема, что у Admin: роль выдаётся на уровне воркспэйса, а не приложения. |
| Дополнительные поля у пользователя (`ProfilesTable.projectRole`) | Требует ручной редакции профилей Admin'ом в админке Chatium; нет UX-потока «дать ссылку и пусть подтвердят». |
| Только IP-allowlist | Сотрудники работают из дома / в разъездах; IP меняются. |
| Использовать `provideUser({minRole: 'Staff'})` middleware | Зеркало роли аккаунта, та же проблема гранулярности. |
| Открытый доступ + проверка по `confirmedEmail` из whitelist | Хрупко, требует ручного ведения whitelist Admin'ом без UI; то же, что `panel_access`, только без удобного потока выдачи. |

## Последствия

**Положительные:**
- Сотрудники школы получают доступ к панели, не получая Admin-прав воркспэйса.
- Гранулярность - per-user, не per-role. Admin видит, кому именно выдан доступ, и может отозвать.
- UX выдачи доступа - один клик у Admin + один клик у получателя. Без редактирования профилей и системных ролей.
- Закрывает auth-разрыв из аудита 24-05-2026 единым проходом (одна утилита `requireInternalAccess` применяется ко всем защищённым эндпоинтам).

**Отрицательные:**
- Две новых таблицы Heap (`panel_invites`, `panel_access`), новый код (`lib/access/`), новая страница (`/web/access/invite`), новая страница 403 (`/web/forbidden`), новый API (4 эндпоинта). Объём - ~день работы.
- Инвайт - **«ссылка-claim»**, а не персональный токен: первый авторизованный пользователь, который нажмёт «Подтвердить», получит доступ. Если ссылку перехватят, доступ заберёт перехвативший. Митигация - короткий TTL (7 дней) + Admin видит в списке грантов поле `usedByUserId` и может отозвать выданный доступ через `revoke-grant`.

**Семантика жизненного цикла инвайта (важно):**
- Просто переход по URL (GET страницы `/web/access/invite`) **не расходует** инвайт. Пользователь может открыть ссылку, закрыть вкладку, открыть снова, переслать другому - инвайт остаётся активным.
- Инвайт расходуется **только при успешном `POST /api/access/consume-invite`** (нажатие кнопки «Подтвердить» авторизованным пользователем).
- После успешного потребления инвайт навсегда переходит в состояние `used` - повторный переход по той же ссылке от того же или другого пользователя показывает страницу «Ссылка уже была использована».
- Admin может в любой момент отозвать неиспользованный инвайт через `revoke-invite` - тогда переход покажет «Ссылка недействительна».

## Открытые вопросы

- **Срок жизни инвайта.** По умолчанию - 7 дней с момента генерации (просроченный инвайт нельзя использовать, но Admin может вручную сгенерировать новый). Точное значение - в §1.11 implementation-plan.
- **Отзыв уже выданного доступа.** Поддерживается через `POST /api/access/revoke-grant` (Admin-only): удаляет запись из `panel_access`, при следующем заходе сотрудник видит 403.
- **Сценарий «Admin потерял доступ».** Admin аккаунта Chatium всегда имеет доступ - проверка `ctx.user.is('Admin')` в `requireInternalAccess`. Восстановление через ЛК Chatium-воркспэйса.
- **Защита публичной страницы инвайта от перебора токенов.** Длина токена ≥ 32 символа (`accountNanoid`) - 36^32 = ~10^49 комбинаций, перебор нереалистичен. Дополнительная rate-limit мера - открытый вопрос; если LifePay-нагрузка покажет атаки, добавить позже.
