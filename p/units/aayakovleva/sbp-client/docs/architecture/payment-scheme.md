# Раскладка способов оплаты на странице школы

Бизнес-логика клиентского виджета, встраиваемого в страницу оплаты школы GetCourse. Описывает, какие способы оплаты должен видеть покупатель, в каком порядке, при каких условиях, и как виджет ведёт себя при разной сумме заказа. Технический норматив реализации (контракты, ошибки, секреты) - в [operation-manual](../../../../../saas/gw/lifepay/docs/gateway/operation-manual.md); полная карта системы со всеми приложениями и каналами - в [data-flow](../../../project-docs/architecture/data-flow.md); приёмочные критерии заказчика - в [acceptance-criteria](../../../project-docs/testing/acceptance-criteria.md).

## 1. Целевая раскладка способов оплаты

### 1.1. Группы и приоритеты

| Группа                                         | Способ                       | Источник / провайдер      | Приоритет    |
| ---------------------------------------------- | ---------------------------- | ------------------------- | ------------ |
| **Первый экран (флагманы)**                    | СБП                          | LifePay                   | Обязательный |
| **Первый экран (флагманы)**                    | ОТП Банк (рассрочка)         | Подключение клиента       | Обязательный |
| **Первый экран (флагманы)**                    | Оплата из-за границы         | Lava Top                  | Обязательный |
| **Блок «Другие методы» (скрыт под раскрытие)** | Тинькофф через GetCourse Pay | GetCourse Pay             | Обязательный |
| **Блок «Другие методы»**                       | Сбер через GetCourse Pay     | GetCourse Pay             | Обязательный |
| **Блок «Другие методы»**                       | Прочие банки РФ              | GetCourse Pay / GetModule | Обязательный |
| **Блок «Другие методы»**                       | Рассрочки (split-сервисы)    | GetCourse Pay / GetModule | Обязательный |
| **Блок «Другие методы»**                       | Карты РФ                     | GetCourse Pay / GetModule | Обязательный |

### 1.2. Правило переключения по сумме заказа

| Сумма заказа                 | Видимая раскладка                                                                                                                                                                       |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **до 50 000 ₽**              | Все способы оплаты видимы, в порядке из §1.1 (флагманы сверху, остальные ниже). Блок «Другие методы» развёрнут по умолчанию.                                                            |
| **от 50 000 ₽ включительно** | На первом экране только флагманы (СБП, ОТП, Lava Top). Остальные способы свёрнуты в раскрывающийся блок «Другие методы», визуально приглушённый. Покупатель может раскрыть блок кликом. |

Порог 50 000 ₽ выбран из коммерческой логики клиента: при сумме выше этого порога приоритетной для школы становится СБП-оплата (комиссия 0.4% против 3% у GetCourse Pay), и UX направляет покупателя на флагманские способы.

### 1.3. Поведение виджета

- **На старте.** Виджет ждёт DOMReady, находит контейнер способов оплаты GetCourse, читает сумму заказа из контекста страницы (data-атрибут или глобальный JS-объект GetCourse).
- **Перестроение порядка.** Виджет тэгирует доступные методы по справочнику (`sbp-lifepay`, `otp`, `lava-top`, `gc-tinkoff`, `gc-sber`, `gc-others`, `splits`, `cards-rf`) и перестраивает их в нужном порядке через DOM-операции.
- **Скрытие через CSS + JS.** Способы, которые должны быть в свёрнутом блоке, скрываются CSS-классом виджета; визуально подменяются на `<details>`/`<summary>`-обёртку «Другие методы». Технически методы остаются на странице - нативный обработчик GetCourse при клике срабатывает корректно.
- **Подмена обработчика только для СБП.** Виджет-бандл подменяет click-handler только для способа «СБП через LifePay»: клик инициирует `fetch` на эндпоинт `POST /api/lp/invoke` клиентской панели `p/units/aayakovleva/sbp-client` (см. [operation-manual §12.1](../../../../../saas/gw/lifepay/docs/gateway/operation-manual.md), [implementation-plan §1.8 / §2.7](../../../../../saas/gw/lifepay/docs/gateway/implementation-plan.md), [data-flow §2](../../../project-docs/architecture/data-flow.md)). Панель сама делает прямой HTTP-запрос к payments-gateway и пишет результат в свой журнал `request_log`. Все остальные способы (ОТП, Lava Top, методы GetCourse Pay) работают через свои нативные обработчики без вмешательства виджета.
- **Mobile.** При выборе СБП на мобильном устройстве виджет делает `window.location.href = paymentUrl` (deeplink на `qr.nspk.ru`, который банковское приложение перехватывает). На десктопе - QR-код прямо на странице плюс кнопка «Оплатить в браузере» с `href = paymentUrlWeb`.
- **Деградация.** Если виджет не нашёл контейнер способов оплаты (GetCourse изменил вёрстку), он логирует ошибку в `console.warn` и оставляет страницу как есть, не ломая нативный флоу GetCourse. `MutationObserver` с таймаутом 5 секунд страхует случай асинхронного рендеринга методов.

## 2. Риски и открытые решения

### 2.1. Риски

- **GetCourse меняет разметку страницы оплаты.** Виджет должен быть устойчивым к изменению порядка элементов и текстов; селекторы вынесены в `config.ts` виджета и подбираются по нескольким признакам (data-атрибут, class, text content) с фолбэком.
- **Асинхронный рендеринг методов.** Если способы оплаты подгружаются через AJAX после загрузки страницы, виджет должен дождаться появления элементов через `MutationObserver`.
- **Скрытие способа через CSS.** Не должно ломать его технически: нативный обработчик при программном клике (или клике на родительский элемент) должен срабатывать. Проверяется в приёмке.
- **Lava Top.** Иностранные оплаты могут потребовать отдельного сценария обновления статуса заказа на стороне школы. В скоупе текущего проекта этот канал не модифицируется, виджет только сохраняет работоспособность.

### 2.2. Открытые решения

- **Селекторы DOM на странице школы.** Точные `class` / `data-*` подбираются после получения доступа к боевой странице школы. Зафиксируется в `notes/2026-MM-DD--gc-page-anatomy.md` приёмника при подключении виджета.
- **Fallback для отсутствующего метода.** Если в боевой школе один из ожидаемых способов оплаты не подключён (например ОТП временно недоступен), виджет должен корректно пропустить его в раскладке без падения. Реализация - проверка через `querySelector` перед перемещением; отсутствующий способ просто не появляется в финальной раскладке.
