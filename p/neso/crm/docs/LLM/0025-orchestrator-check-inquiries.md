# 0025 — Реализация раздела «Обращения» и проверка оркестратором

## Пользователь

Запрос на реализацию нового раздела интерфейса «Обращения» для переписки с конечными пользователями/клиентами. На данном этапе требовалась только страница и компоновка блоков без реализации логики переписки, отправки сообщений и интеграций.

**Цель:** добавить в приложение новый раздел/страницу «Обращения» с базовой вёрсткой и структурой в стиле проекта, разместить ссылку в меню слева.

**Объём работ:**
- Создать новый маршрут/страницу в рамках существующего роутинга.
- Добавить пункт навигации в меню/сайдбар.
- Сверстать страницу: общий стиль проекта, адаптивная разметка, переиспользуемые компоненты (Layout, PageHeader, Card, Tabs, EmptyState, SearchInput, Button, Badge и т.п.).

**Требуемая компоновка:**
1. Верхняя панель: заголовок «Обращения», справа место под действия (кнопка «Фильтры»), под заголовком — поиск по обращениям (только UI).
2. Левая колонка: список обращений (Card/Panel), поиск, табы «Все»/«Открытые»/«Закрытые», список элементов (имя клиента, превью, дата, статус badge) или Empty state.
3. Правая колонка (область переписки): шапка треда (имя, статус, атрибуты), область сообщений (плейсхолдер), внизу textarea, кнопка «Отправить» (без логики, допустимо disabled), опционально «Прикрепить файл».
4. Справа третья колонка — детальная информация о клиенте (Ф.И.О., email, телефон, спойлер с техническими данными). На широких экранах открыта, на узких свёрнута и открывается по кнопке, заменяя область переписки; после открытия — кнопка свернуть.
5. Адаптивность: на мобильных по умолчанию список; при выборе — скрывать правую колонку, корректная вёрстка без полноценного состояния.

Данные и логика: не подключать API, не реализовывать реальную переписку; допускается статический массив мок-данных или skeleton/empty state.

**Итог:** раздел «Обращения» доступен по маршруту, отображается трёхколоночный UI-скелет с мок-списком обращений; логика переписки и API не реализованы.
