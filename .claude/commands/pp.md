# /pp — Parallel Pipeline

Сценарий разработки в режиме **параллельного исполнения**: формализация → квалификация → декомпозиция → параллельный запуск воркеров в фоне → проверка → отчёт.

В отличие от `/pipeline` (последовательный 12-шаговый конвейер), `/pp` декомпозирует задачу на независимые куски и запускает воркеров одновременно через `Agent(run_in_background: true)`.

## Когда использовать

- Задача декомпозируется на ≥ 2 независимых куска (разные слои, разные модули, разные подсистемы).
- Куски не зависят друг от друга по результатам (никто не ждёт чужого вывода).
- Каждый кусок имеет свою область файлов (минимизация конфликтов правок).

Если задача монолитная — используй `/pipeline`.

## Инвариант: блок «Вопросы» перед ожиданием

Если субагент задал уточняющие вопросы, **обязательно** перед остановкой выведи отдельный блок:

```
**Вопросы (повтор перед ожиданием ответа):**
1. <вопрос дословно>
2. <вопрос дословно>

**Ожидаю ваш ответ** — до следующего сообщения действий не выполняю.
```

Этот блок — последнее, что видит пользователь. Без него остановка считается неоформленной.

## Конвейер /pp

### Шаг 1. Техническая переформулировка

Возьми $ARGUMENTS (или последнюю формулировку из чата) и:
1. Перепиши технически грамотно: цель, контекст, ожидаемый артефакт.
2. Дополни очевидным контекстом (рабочие папки, релевантные модули, известные ограничения проекта).
3. **Не выдумывай требования.** Всё неясное — открытый вопрос.
4. Покажи переформулировку пользователю **только** если она существенно меняет смысл; иначе передавай её сразу в task-formalizer.

### Шаг 2. Квалификация постановки (Agent → `task-formalizer`)

Вызови **Agent** с `subagent_type: task-formalizer` и `run_in_background: true`. В промпт передай:
- сырой текст пользователя;
- твою техническую переформулировку (шаг 1);
- 10 критериев приёмки постановки (см. ниже);
- явное указание: «Если хотя бы один критерий не выполнен — задай 2–5 уточняющих вопросов. **Не додумывать.** Если все критерии выполнены — выдай финальную формализацию + предложение по декомпозиции.»

### Критерии приёмки постановки (для /pp)

Постановка считается полной только если выполнены **все 10 критериев:**

1. **Цель** — одно проверяемое предложение.
2. **Артефакт** — тип результата.
3. **Область и контекст** — конкретный проект/папка/модуль.
4. **Границы** — что НЕ входит.
5. **Входные данные** — источники, файлы, API, события.
6. **Критерии приёмки задачи** — измеримый чеклист (≥ 3 пункта).
7. **Ошибки и edge cases** — поведение при ошибках, пустом входе, отсутствии прав.
8. **Платформенные ограничения Chatium** — file-based роутинг, Heap-only-on-server, встроенные модули.
9. **Декомпозируемость** — задача делится на N независимых кусков, либо явно «монолит — обоснование».
10. **Зависимости** — внешние API, секреты, чужие проекты, последовательные предусловия.

### Шаг 3. Возврат вопросов пользователю

Если `task-formalizer` вернул вопросы:

1. Кратко покажи контекст (1–2 строки).
2. Перешли пользователю **полный список вопросов в исходном виде** (не сокращай).
3. Покажи, **какие критерии** на текущий момент не выполнены.
4. **Дублирование перед ожиданием:** отдельный блок «**Вопросы (повтор перед ожиданием ответа):**» — снова с дословным списком.
5. Завершающая строка: «**Ожидаю ваш ответ** — до следующего сообщения действий не выполняю.»
6. **Остановись.**

### Шаг 4. Итерация после ответа пользователя

После ответа пользователя:
1. Обнови переформулировку.
2. Снова вызови Agent → `task-formalizer` с обновлённой постановкой и историей итерации.
3. Лимит итераций уточнений — **5**.

### Шаг 5. Декомпозиция

Когда постановка пройдена:
1. Возьми из формализации список независимых подзадач.
2. Раздели на **максимально возможное количество** кусков, для которых одновременно:
   - не зависят друг от друга по результатам;
   - имеют собственные критерии приёмки;
   - имеют собственную область файлов.
3. Если кусок неделим — пометь как «монолит» с обоснованием.
4. Покажи пользователю **краткий план декомпозиции** (1–2 строки на воркера) **до запуска**.

### Шаг 6. Параллельный запуск воркеров

1. Все Agent-вызовы воркеров — в **одном сообщении** (один блок tool calls).
2. Каждому воркеру: `run_in_background: true`.
3. `subagent_type` подбирай по характеру:
   - `general-purpose` — общие задачи, незакрепленные за одной ролью;
   - `docs-keeper`, `planner`, `code-reviewer` и т.д. — где это именно их зона.
4. В промпт каждому воркеру передавай:
   - общую цель и формализацию;
   - **только его кусок** с собственными критериями приёмки;
   - явный список файлов/папок, которые он может править;
   - запрет на правки за границами куска;
   - формат финального ответа.
5. Зафиксируй **timestamp начала** через `date "+%s"`.

### Шаг 7. Сбор результатов

1. Дождись уведомлений о завершении **всех** background-воркеров.
2. Не делай реактивный poll — Claude Code пришлёт notification по каждому.
3. По завершении воркера сразу сверь его выход с критериями приёмки его куска.
4. Зафиксируй **timestamp завершения**.

### Шаг 8. Верификация

1. По каждому воркеру: пройдены ли его критерии приёмки?
2. По задаче целиком: пройден ли общий чеклист?
3. При невыполненных критериях:
   - сформулируй **конкретные** замечания;
   - запусти воркера повторно через Agent с задачей «доработать так-то»;
   - параллельно стартуй других воркеров с замечаниями.
4. Лимит циклов доработки — **3**. После — фиксируй остаточные риски.

### Шаг 9. Финальный отчёт

```
## /pp — итог

**Статус:** готово / закрыто частично / провалено
**Воркеров задействовано:** N (успех с первого раза: K, доработок: M)
**Время:** Δ секунд (start: <ts>, end: <ts>)
**Итераций квалификации постановки:** Q

**Что сделано:**
- кусок 1 (`<воркер>`): краткое описание + ключевые файлы
- кусок 2 (`<воркер>`): ...

**Критерии приёмки:**
- [x] критерий 1
- [x] критерий 2
- [ ] критерий N — если незакрыт: причина

**Открытые риски / TODO:** если есть
```

## Запрещено

- ❌ Реализовывать задачу до прохождения квалификации постановки.
- ❌ Переходить в режим ожидания **без** обязательного блока «Вопросы (повтор перед ожиданием ответа):» и явной строки «Ожидаю ваш ответ».
- ❌ Отвечать пользователю за пользователя на уточняющие вопросы.
- ❌ Запускать воркеров последовательно вместо параллельных Agent-вызовов в одном сообщении.
- ❌ Объявлять задачу «готовой», если хотя бы один применимый критерий приёмки не выполнен — без явного упоминания этого в финальном отчёте.
- ❌ Финальный отчёт без секций «Воркеров задействовано / Время / Итераций».
