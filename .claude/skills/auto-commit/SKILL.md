# Автокоммит изменений

Цель: превратить произвольный набор незакоммиченных изменений в **несколько
осмысленных коммитов**, по одному на логическую единицу, с сообщениями на русском
в формате `type(scope): описание` + список изменений, затем закоммитить и запушить.

Работай только с git-репозиторием текущего workspace (`s.chtm.khudoley.pro`).
Все git-команды выполняй через **Bash tool**.

---

## Шаг 0. Предусловия

1. Убедись, что находишься в git-репозитории и нет незавершённого merge/rebase:
   ```bash
   git rev-parse --is-inside-work-tree
   git status --porcelain=v1
   ```
2. Если есть **unmerged** записи (`UU`, `AA`, `DD` и т.п.) — **остановись**, сообщи
   о конфликте, не коммить.
3. Если изменений нет вообще — сообщи «нечего коммитить» и заверши.
4. Аргументы команды (`$ARGUMENTS`), если переданы:
   - `--no-push` — выполнить только коммиты, без пуша.
   - свободный текст — подсказка по группировке/формулировкам, учти её.

## Шаг 1. Собрать и понять изменения

```bash
git -c core.quotepath=false status --porcelain=v1   # читаемые кириллические пути
git rev-parse --abbrev-ref HEAD                      # текущая ветка
git --no-pager diff --stat                           # изменения в трекнутых файлах
git --no-pager diff --cached --stat                  # уже застейдженные
```

- Для понимания сути правок смотри `git --no-pager diff -- <путь>` по нужным файлам.
- Untracked-файлы (`??`) и новые каталоги — открой ключевые из них (Read), чтобы
  понять назначение. Каталог в статусе (`?? path/`) разворачивай: `git status --porcelain -- path/` или `ls`.
- **Чистый старт стейджинга** (безопасно, рабочее дерево не трогает): если что-то
  уже застейджено, сними со стейджа, чтобы управлять группировкой самому —
  `git reset` (без флагов). Это не теряет изменения.

## Шаг 2. Сгруппировать в коммиты

Один коммит = одна логическая единица. **Основной ключ группировки — корень
проекта / scope**, выводимый из пути:

| Путь | scope |
|------|-------|
| `p/units/<unit>/<project>/...` | `<project>` (напр. `sbp-client`) |
| `p/saas/<...>/<project>/...` | `<project>` (напр. `lifepay`) |
| `k/assistant/...` | `assistant` |
| `tg/<project>/...` | `<project>` |
| `inner/...` | `inner` или опустить scope |
| `.claude/...`, `.cursor/...`, корневые конфиги | без scope, тип `chore` |

Правила:
- Файлы **разных проектов** — всегда **разные коммиты**.
- Внутри одного проекта обычно один коммит. Раздели на несколько, только если
  правки явно про **разные задачи** (например, новая фича в `api/` + не связанная
  правка документации) — тогда отдели по назначению (код / тесты / docs).
- Не дроби чрезмерно: связанные код+тесты+docs одной фичи — это **один** коммит.
- Инфраструктурные/служебные файлы (`.claude/`, `.cursor/`, бинарные сэмплы вроде
  `inner/samples/*.zip`) — отдельный `chore`-коммит, не подмешивай к коду проекта.

Если в группе непонятно намерение (не ясно, фича это или фикс) — посмотри дифф
внимательнее перед формулировкой.

## Шаг 3. Сообщение коммита (`type(scope): описание` + список изменений)

Формат сообщения — **заголовок `type(scope): короткое описание`, пустая строка,
маркированный список** конкретных изменений группы:

```
type(scope): короткое описание в формате «что сделано»

- первое изменение
- второе изменение
- третье изменение
```

Типы (`type`):
- `feat` — новая функциональность.
- `fix` — исправление бага.
- `refactor` — изменение кода без изменения поведения.
- `docs` — только документация (README, docs/, .md).
- `test` или `feat(tests)` — тесты.
- `chore` — инфраструктура, конфиги, служебное, бинарные сэмплы.

Правила **заголовка** (`type(scope): описание`):
- `scope` — корень проекта/область из Шага 2 (`sbp-client`, `lifepay`, `claude`…);
  опционален, но предпочтителен.
- Описание — на русском, кратко «что сделано»: глагол в прошедшем времени
  (*добавил, обновил, исправил, удалил, переработал*) либо отглагольное
  существительное (*добавление, обновление…*). С маленькой буквы после `:`,
  без точки в конце.
- Технические термины и имена не переводить (`webhook`, `GetCourse`, `API`).
- Ориентир ≤ 72 символа на первой строке.

Правила **списка изменений**:
- Между заголовком и списком — **пустая строка** (обязательно).
- Каждый пункт начинается с `- `, по одному значимому изменению на строку.
- Формулировки на русском, кратко и по сути.
- Ориентир 1–7 пунктов. Если в группе единственное тривиальное изменение —
  список можно опустить и оставить только заголовок.

Пример:
```
feat(sbp-client): добавил внутренний контроль доступа и приглашения

- гард requireInternalAccess на эндпоинтах api/lp
- эндпоинты генерации и погашения приглашений в api/access
- страницы web/access/invite и web/forbidden
- обновил docs (architecture, api, data, ADR 0003) и тесты
```

## Шаг 4. Закоммитить каждую группу

Для каждой группы по очереди:

1. Застейдж **только файлы этой группы** явными путями (не `git add -A`!).
   Стейджинг через `--` и кавычки для путей с пробелами/кириллицей:
   ```bash
   git add -- "p/units/aayakovleva/sbp-client/api/lp/invoke.ts" "p/units/.../другой файл.ts"
   ```
   `git add -- <путь>` корректно стейджит и изменения, и удаления, и новые файлы.
   Новый каталог можно застейджить целиком: `git add -- "path/to/dir/"`.
2. Проверь, что в стейдже именно то, что нужно: `git --no-pager diff --cached --name-only`.
3. Закоммить многострочным сообщением (заголовок + список) через here-doc —
   это надёжнее всего сохраняет переводы строк. `EOF` должен стоять в начале
   строки без отступа:
   ```bash
   git commit -F - <<'EOF'
   feat(sbp-client): добавил внутренний контроль доступа и приглашения

   - гард requireInternalAccess на эндпоинтах api/lp
   - эндпоинты генерации и погашения приглашений в api/access
   - страницы web/access/invite и web/forbidden
   EOF
   ```
   Альтернатива — несколько `-m`: первый аргумент = заголовок, второй =
   блок списка (с реальными переводами строк внутри одной строки-аргумента).
   Каждый `-m` становится отдельным абзацем, разделённым пустой строкой.

**Атрибуция:** footer `Co-Authored-By` **не добавляй** — в истории репозитория
его нет, цель — единый стиль с прошлыми коммитами.

## Шаг 5. Запушить

Если передан `--no-push` — пропусти этот шаг.

```bash
git rev-parse --abbrev-ref --symbolic-full-name '@{u}'   # есть ли upstream
```
- Если upstream есть: `git push`.
- Если upstream нет: `git push -u origin <текущая-ветка>` (ветку из шага 1).
- Если push отклонён (non-fast-forward) — **не делай** `--force`. Сообщи
  пользователю и предложи `git pull --rebase` вручную.

## Шаг 6. Отчёт

Кратко (на русском):
- Список созданных коммитов: `<hash короткий> — <заголовок>` (тело-список не дублировать).
- Что в каждый попало (по проектам/файлам — без листинга диффов).
- Результат пуша (ветка, upstream) или отметку «пуш пропущен (--no-push)».

## Edge cases

- **Нечего коммитить** → сообщить и выйти.
- **Конфликты/незавершённый merge** → не коммитить, сообщить.
- **Удалённые файлы** → `git add -- <путь>` стейджит удаление; либо `git rm`.
- **Переименования** → стейдж обоих путей (старый+новый) в одной группе.
- **Бинарные/сэмплы** (`.zip`, `.docx`) → отдельный `chore`-коммит, описание по сути
  («обновление сэмпла…», «обновление договорных документов…»).
- **Пути с кириллицей/пробелами** → всегда в двойных кавычках; для чтения статуса
  использовать `-c core.quotepath=false`.

## Чего НЕ делать

- Не запускать `/check` сам (CLAUDE.md §5) — если нужно, пользователь сделает это до `/commit`.
- Не трогать workspace `p.chtm.khudoley.pro` (это только через `/toprod`).
- Не использовать `git add -A` / `git commit -am` — это ломает группировку.
- Не делать `git push --force` и не переписывать историю.
