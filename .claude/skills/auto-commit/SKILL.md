# Автокоммит изменений (async, Haiku-субагенты)

Цель: превратить произвольный набор незакоммиченных изменений в **несколько
осмысленных коммитов**, по одному на логическую единицу, с сообщениями на русском
в формате `type(scope): описание` + список изменений, затем закоммитить и запушить.

**Архитектура (важно):**

- **Основной поток** (исполнитель этого навыка) делает только две вещи: определяет
  **границы задач** (группы + точный список файлов в каждой) и в конце делает
  **один `git push`**.
- На каждую группу основной поток порождает **отдельного субагента** через **Agent
  tool**, жёстко на модели **Haiku** (`model: 'haiku'`), и запускает их
  **асинхронно** — все вызовы Agent **одним сообщением** (несколько tool-use
  подряд), чтобы они работали параллельно.
- Каждый субагент: читает дифф своих файлов, формулирует сообщение, и **сам
  коммитит свою группу** — но реальный `git commit` сериализуется через
  lock-файл (см. §Протокол), поэтому гонки за git-индексом нет. Субагенты
  **не пушат** — пуш делает основной поток один раз в конце.

Работай только с git-репозиторием текущего workspace (`s.chtm.khudoley.pro`).
Все git-команды выполняй через **Bash tool**.

---

## Шаг 0. Предусловия (основной поток)

1. Убедись, что находишься в git-репозитории и нет незавершённого merge/rebase:
   ```bash
   git rev-parse --is-inside-work-tree
   git status --porcelain=v1
   ```
2. Если есть **unmerged** записи (`UU`, `AA`, `DD` и т.п.) — **остановись**, сообщи
   о конфликте, не коммить.
3. Если изменений нет вообще — сообщи «нечего коммитить» и заверши.
4. Аргументы команды (`$ARGUMENTS`), если переданы:
   - `--no-push` — выполнить только коммиты, без финального пуша.
   - свободный текст — подсказка по группировке/формулировкам, учти её при
     определении границ и передай субагентам в их промпте.

## Шаг 1. Собрать и понять изменения (основной поток)

```bash
git -c core.quotepath=false status --porcelain=v1   # читаемые кириллические пути
git rev-parse --abbrev-ref HEAD                      # текущая ветка
git --no-pager diff --stat                           # изменения в трекнутых файлах
git --no-pager diff --cached --stat                  # уже застейдженные
```

- Достаточно понять, **к каким группам относятся файлы** — детальное чтение
  диффов для формулировок делают субагенты. Но если по `--stat` неочевидно,
  фича это или фикс / к какому проекту относится, — посмотри
  `git --no-pager diff -- <путь>` и/или открой untracked-файл (Read).
- Каталог в статусе (`?? path/`) разворачивай: `git status --porcelain -- path/` или `ls`.
- **Чистый старт стейджинга** (безопасно, рабочее дерево не трогает): если что-то
  уже застейджено, сними со стейджа — `git reset` (без флагов). Это не теряет
  изменения и нужно, чтобы субагенты сами управляли стейджингом своих групп.

## Шаг 2. Определить границы групп (основной поток)

Это **главная задача основного потока**. Один коммит = одна логическая единица.
**Основной ключ группировки — корень проекта / scope**, выводимый из пути:

| Путь                                           | scope                            |
| ---------------------------------------------- | -------------------------------- |
| `p/units/<unit>/<project>/...`                 | `<project>` (напр. `sbp-client`) |
| `p/saas/<...>/<project>/...`                   | `<project>` (напр. `lifepay`)    |
| `k/assistant/...`                              | `assistant`                      |
| `tg/<project>/...`                             | `<project>`                      |
| `inner/...`                                    | `inner` или опустить scope       |
| `.claude/...`, `.cursor/...`, корневые конфиги | без scope, тип `chore`           |

Правила:

- Файлы **разных проектов** — всегда **разные группы**.
- Внутри одного проекта обычно одна группа. Раздели на несколько, только если
  правки явно про **разные задачи** (например, новая фича в `api/` + не связанная
  правка документации) — тогда отдели по назначению (код / тесты / docs).
- Не дроби чрезмерно: связанные код+тесты+docs одной фичи — это **одна** группа.
- Инфраструктурные/служебные файлы (`.claude/`, `.cursor/`, бинарные сэмплы вроде
  `inner/samples/*.zip`) — отдельная `chore`-группа, не подмешивай к коду проекта.

Результат шага — список групп, где для каждой определены:

- `scope` и предполагаемый `type` (как подсказка субагенту);
- **точный список файлов** (явные пути, как они в `git status`).

Файл может входить ровно в **одну** группу (без пересечений между группами).

## Шаг 3. Запустить субагентов параллельно (основной поток)

Для **каждой** группы вызови **Agent tool** с `model: 'haiku'`. **Все вызовы — в
одном сообщении** (асинхронный fan-out). В промпте каждого субагента передай:

- scope и предполагаемый type группы (подсказка, субагент может уточнить по диффу);
- **точный список файлов** этой группы;
- **абсолютный** путь к lock-файлу: `LOCK` = `<git-dir>/.commit-serial.lock`,
  где `<git-dir>` = вывод `git rev-parse --absolute-git-dir` (вычисли один раз
  в основном потоке; абсолютный — чтобы субагент не зависел от своего cwd);
- подсказку пользователя из `$ARGUMENTS`, если была;
- **полный текст промпта-инструкции субагента** из §Промпт субагента ниже.

Если групп всего одна — всё равно используй одного субагента (единый путь кода).

После запуска **дождись завершения всех** субагентов. Собери их результаты
(hash + заголовок каждого коммита, либо текст ошибки).

## Шаг 4. Финальный пуш (основной поток)

Если передан `--no-push` — пропусти.

Делай пуш **один раз**, после того как все субагенты закоммитили:

```bash
git rev-parse --abbrev-ref --symbolic-full-name '@{u}'   # есть ли upstream
```

- Если upstream есть: `git push`.
- Если upstream нет: `git push -u origin <текущая-ветка>` (ветка из Шага 1).
- Если push отклонён (non-fast-forward) — **не делай** `--force`. Сообщи
  пользователю и предложи `git pull --rebase` вручную.

## Шаг 5. Отчёт (основной поток)

Кратко (на русском):

- Список созданных коммитов: `<hash короткий> — <заголовок>` (тело-список не дублировать).
- Что в каждый попало (по проектам/файлам — без листинга диффов).
- Если какой-то субагент упал — отметь, что его группа **не закоммичена**.
- Результат пуша (ветка, upstream) или отметку «пуш пропущен (--no-push)».

## Протокол сериализации коммитов (lock-файл)

Субагенты работают параллельно, но git-индекс и рабочее дерево — общий ресурс.
Поэтому **реальный `git commit` каждый субагент выполняет под взаимоисключающим
lock-файлом**. Атомарность даёт `mkdir` (создание каталога атомарно и падает,
если каталог уже есть):

```bash
acquire() {                                  # ожидание + захват
  while ! mkdir "$LOCK" 2>/dev/null; do
    # stale-guard: если lock «висит» дольше 2 минут — считаем его осиротевшим
    if find "$LOCK" -maxdepth 0 -mmin +2 2>/dev/null | grep -q .; then
      rmdir "$LOCK" 2>/dev/null || true
      continue
    fi
    sleep 0.2
  done
}
release() { rmdir "$LOCK" 2>/dev/null || true; }
```

- **Только** стейджинг + коммит идут под локом. Чтение диффа и формулировку
  сообщения делай **до** захвата (это самая долгая, LLM-часть — она и идёт
  параллельно).
- Лок держи минимально — захватил, застейджил свои файлы, закоммитил, отпустил.
- `release` обязателен и в случае ошибки коммита (чтобы не заблокировать других).

## Промпт субагента (Haiku) — шаблон

Основной поток подставляет `{SCOPE}`, `{TYPE_HINT}`, `{FILES}` (список путей),
`{LOCK}`, `{HINT}` и передаёт это как `prompt` в Agent tool.

> Ты коммитишь **одну** логическую группу изменений в git-репозитории Chatium
> workspace `s.chtm.khudoley.pro`. Все git-команды — через Bash. Твоя группа:
>
> - предполагаемый scope: `{SCOPE}`, предполагаемый type: `{TYPE_HINT}`;
> - **файлы группы** (только эти, явными путями): `{FILES}`;
> - lock-файл для сериализации коммита: `{LOCK}`;
> - подсказка пользователя (если есть): `{HINT}`.
>
> **Шаг A. Понять изменения (без лока, параллельно с другими).**
> Прочитай дифф только своих файлов:
> `git --no-pager diff -- <твои пути>` (для untracked — открой файл через Read).
> Уточни type по сути правок, если подсказка не подходит.
>
> **Шаг B. Составить сообщение** в формате `type(scope): описание` + пустая
> строка + маркированный список (см. правила ниже).
>
> **Шаг C. Закоммитить под локом.** Захвати лок (mkdir-spinlock), застейдж
> **только свои файлы** явными путями, проверь стейдж, закоммить here-doc'ом,
> отпусти лок. Не пушь — пуш делает основной поток.
>
> ```bash
> LOCK="{LOCK}"
> while ! mkdir "$LOCK" 2>/dev/null; do
>   if find "$LOCK" -maxdepth 0 -mmin +2 2>/dev/null | grep -q .; then
>     rmdir "$LOCK" 2>/dev/null || true; continue
>   fi
>   sleep 0.2
> done
> git add -- "путь1" "путь2"                       # только свои файлы, не git add -A
> git --no-pager diff --cached --name-only         # убедись, что в стейдже только твоё
> git commit -F - <<'EOF'
> type(scope): короткое описание
>
> - изменение 1
> - изменение 2
> EOF
> rc=$?
> rmdir "$LOCK" 2>/dev/null || true                # отпусти лок ВСЕГДА
> exit $rc
> ```
>
> **Верни** ровно: короткий hash и заголовок созданного коммита
> (`git --no-pager log -1 --format='%h %s'`), либо текст ошибки, если коммит не
> удался. Кратко, без листинга диффа.

### Правила сообщения (для субагента)

Типы (`type`): `feat` (новая функциональность), `fix` (багфикс), `refactor`
(без изменения поведения), `docs` (только документация), `test` / `feat(tests)`
(тесты), `chore` (инфраструктура, конфиги, бинарные сэмплы).

Заголовок `type(scope): описание`:

- `scope` — корень проекта/область (`sbp-client`, `lifepay`, `claude`…);
  опционален, но предпочтителен.
- Описание — на русском, кратко «что сделано»: глагол в прошедшем времени
  (_добавил, обновил, исправил, удалил, переработал_) либо отглагольное
  существительное. С маленькой буквы после `:`, без точки в конце.
- Технические термины и имена не переводить (`webhook`, `GetCourse`, `API`).
- Ориентир ≤ 72 символа на первой строке.

Список изменений:

- Между заголовком и списком — **пустая строка** (обязательно).
- Каждый пункт с `- `, по одному значимому изменению на строку, на русском.
- Ориентир 1–7 пунктов. Если изменение единственное тривиальное — список
  можно опустить и оставить только заголовок.

Пример:

```
feat(sbp-client): добавил внутренний контроль доступа и приглашения

- гард requireInternalAccess на эндпоинтах api/lp
- эндпоинты генерации и погашения приглашений в api/access
- страницы web/access/invite и web/forbidden
- обновил docs (architecture, api, data, ADR 0003) и тесты
```

**Атрибуция:** footer `Co-Authored-By` **не добавляй** — в истории репозитория
его нет, цель — единый стиль с прошлыми коммитами.

## Edge cases

- **Нечего коммитить** → основной поток сообщает и выходит, субагентов не порождает.
- **Конфликты/незавершённый merge** → не коммитить, сообщить.
- **Одна группа** → один субагент (единый путь).
- **Субагент упал** (ошибка коммита) → лок он обязан отпустить (`release` всегда);
  основной поток отмечает группу как незакоммиченную в отчёте, остальные коммиты
  валидны.
- **Удалённые файлы** → `git add -- <путь>` стейджит удаление; либо `git rm`.
- **Переименования** → оба пути (старый+новый) в одной группе, одному субагенту.
- **Бинарные/сэмплы** (`.zip`, `.docx`) → отдельная `chore`-группа.
- **Пути с кириллицей/пробелами** → всегда в двойных кавычках; для чтения статуса
  использовать `-c core.quotepath=false`.

## Чего НЕ делать

- Основной поток **не коммитит сам** — только границы групп и финальный пуш.
- Субагенты **не пушат** — пуш один раз в конце основным потоком.
- Не запускать `/check` сам (CLAUDE.md §5) — если нужно, пользователь сделает это до `/commit`.
- Не трогать workspace `p.chtm.khudoley.pro` (это только через `/toprod`).
- Не использовать `git add -A` / `git commit -am` — это ломает группировку.
- Не делать `git push --force` и не переписывать историю.
- Не делать реальный `git commit` без захвата lock-файла — будет гонка за индексом.
