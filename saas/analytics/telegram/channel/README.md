### Projects

Таблица для хранения проектов.

| Поле | Тип | Описание |
|------|-----|----------|
| name | String | Название проекта |
| description | String (Optional) | Описание проекта |
| members | Array | Массив участников проекта. Каждый элемент: `{ userId: string, role: 'owner' | 'member' }` |

Проекты позволяют организовать совместную работу нескольких пользователей над ботами и каналами. Каждый проект имеет владельцев (owners) и участников (members). Администраторы имеют полный доступ ко всем проектам.

### BotTokens

Таблица для хранения токенов Telegram ботов.

| Поле | Тип | Описание |
|------|-----|----------|
| token | String | Токен бота, введённый пользователем |
| botName | String | Название бота, полученное через API Telegram |
| botUsername | String | Имя бота (username с суффиксом "_bot") |
| projectId | String | ID проекта, к которому привязан бот |

Каждая запись привязана к проекту через поле `projectId`. При создании записи необходимо указать `projectId` проекта, к которому принадлежит бот.

### TelegramWebhooks

Таблица для логирования всех входящих вебхуков от Telegram.

| Поле | Тип | Описание |
|------|-----|----------|
| botId | String | ID бота, которому принадлежит вебхук |
| updateId | Number (Optional) | Уникальный ID обновления от Telegram (для предотвращения дубликатов) |
| rawData | Any | Сырые данные вебхука в формате JSON |
| receivedAt | DateTime | Время получения вебхука |

Каждая запись привязана к боту через поле `botId`. Вебхуки сохраняются автоматически при получении от Telegram через endpoint `POST /api/webhook/:id`. Вебхуки отправляются всем участникам проекта, к которому принадлежит бот (определяется через `bot.projectId`).

### TelegramChats

Таблица для хранения уникальных телеграм-чатов и каналов, от которых хотя бы раз приходили вебхуки.

| Поле | Тип | Описание |
|------|-----|----------|
| chatId | String | Уникальный ID чата/канала из Telegram (используется как ключ для createOrUpdateBy) |
| botId | String | ID бота, от которого пришёл вебхук с этим чатом |
| projectId | String | ID проекта, к которому принадлежит бот |
| chatType | String (Optional) | Тип чата (private, group, supergroup, channel) |
| chatTitle | String (Optional) | Название чата/канала (если доступно) |
| chatUsername | String (Optional) | Username чата/канала (если доступно) |
| firstSeenAt | DateTime | Время первого появления чата в вебхуках |
| lastSeenAt | DateTime | Время последнего появления чата в вебхуках |

Каждая запись привязана к боту через поле `botId` и к проекту через поле `projectId`. Чаты сохраняются автоматически при обработке вебхуков через endpoint `POST /api/webhook/:id` с использованием метода `createOrUpdateBy` для предотвращения дубликатов.

**Важно:**
- Если вебхук содержит информацию о чате (поле `chat`), чат автоматически сохраняется или обновляется в таблице
- Если вебхук не содержит информацию о чате, запись в таблицу не создаётся
- Для каждого уникального `chatId` создаётся одна запись, которая обновляется при последующих вебхуках от этого чата
- Используется эксклюзивная блокировка (`runWithExclusiveLock`) для предотвращения race condition при параллельных запросах с одним `chatId`
- При обновлении существующего чата сохраняется оригинальное значение `firstSeenAt` и обновляется `lastSeenAt`

**Обрабатываемые типы обновлений:**
Чаты извлекаются из следующих типов обновлений Telegram Bot API:
- `message.chat` - обычные сообщения
- `channel_post.chat` - сообщения в каналах
- `edited_message.chat` - отредактированные сообщения
- `edited_channel_post.chat` - отредактированные сообщения в каналах
- `callback_query.message.chat` - callback queries с сообщениями
- `my_chat_member.chat` - изменения статуса бота в чате
- `chat_member.chat` - изменения статуса участника чата
- `chat_join_request.chat` - запросы на вступление в чат
- `chat_boost.chat` - события boost чата (Bot API 9.0+)
- `removed_chat_boost.chat` - события удаления boost (Bot API 9.0+)

**НЕ обрабатываются** (не содержат Chat объект):
- `inline_query` - содержит только `from` (User), `query`, `location`, `chat_type`
- `chosen_inline_result` - содержит только `from` (User), `result_id`, `query`
- `shipping_query` - содержит только `from` (User), `invoice_payload`, `shipping_address`
- `pre_checkout_query` - содержит только `from` (User), `currency`, `total_amount`, `invoice_payload`
- `poll` - не содержит chat
- `poll_answer` - содержит только `poll_id`, `user` (User), `option_ids`

## Работа с токенами ботов

### Добавление токена бота

Процесс добавления нового токена Telegram бота состоит из следующих этапов:

1. **Валидация токена** (`POST /api/bots/validate-token`):
   - Пользователь вводит токен в модальном окне на странице управления ботами
   - Токен отправляется на сервер для валидации через Telegram Bot API
   - Сервер делает запрос к `https://api.telegram.org/bot{token}/getMe`
   - Проверяется корректность токена и получение информации о боте (id, username, first_name)
   - При успешной валидации возвращается информация о боте

2. **Сохранение токена** (`POST /api/bots/add`):
   - Проверка авторизации пользователя (`requireRealUser`)
   - Валидация наличия токена и `projectId` в запросе
   - Проверка прав доступа к проекту (только участники или админ могут добавлять ботов)
   - Проверка на дубликаты: поиск существующего токена в проекте
   - Если токен уже существует в проекте, возвращается ошибка "Этот токен уже добавлен в этот проект"
   - Создание новой записи в таблице `BotTokens` с полями:
     - `token` - токен бота
     - `botName` - название бота (из Telegram API)
     - `botUsername` - имя бота (из Telegram API)
     - `projectId` - ID проекта
   - Возврат информации о созданном боте

3. **Обновление интерфейса**:
   - После успешного добавления список ботов автоматически обновляется
   - Модальное окно закрывается
   - Новый бот отображается в списке

**Пример использования:**

```typescript
// 1. Валидация токена
const validationResult = await apiValidateTokenRoute.run(ctx, {
  token: '1234567890:ABCdefGHIjklMNOpqrsTUVwxyz'
})

if (!validationResult.success) {
  // Обработка ошибки валидации
  return
}

// 2. Сохранение токена
const saveResult = await apiAddBotRoute.run(ctx, {
  token: '1234567890:ABCdefGHIjklMNOpqrsTUVwxyz',
  botName: validationResult.botInfo.name,
  botUsername: validationResult.botInfo.username
})

if (!saveResult.success) {
  // Обработка ошибки сохранения
  return
}

// 3. Обновление списка
await loadBots()
```

### Удаление токена бота

Процесс удаления токена Telegram бота состоит из следующих этапов:

1. **Инициация удаления**:
   - Пользователь нажимает кнопку удаления (иконка корзины) на карточке бота
   - Вызывается функция `deleteToken(botId)` с ID бота

2. **Удаление через API** (`POST /api/bots/delete`):
   - Проверка авторизации пользователя (`requireRealUser`)
   - Валидация наличия `botId` в запросе
   - Поиск бота в таблице по ID
   - Получение проекта по `bot.projectId`
   - Проверка прав доступа: пользователь должен быть участником проекта (или админом)
   - Если бот не найден, возвращается ошибка "Бот не найден"
   - Если пользователь не имеет доступа к проекту, возвращается ошибка "Нет доступа к этому проекту"
   - Удаление записи из таблицы `BotTokens` через `BotTokens.delete(ctx, botId)`
   - Возврат успешного результата

3. **Обновление интерфейса**:
   - После успешного удаления список ботов автоматически обновляется
   - Удалённый бот исчезает из списка

**Пример использования:**

```typescript
// Удаление бота
const deleteResult = await apiDeleteBotRoute.run(ctx, {
  botId: 'bot_id_12345'
})

if (!deleteResult.success) {
  // Обработка ошибки удаления
  console.error('Ошибка при удалении:', deleteResult.error)
  return
}

// Обновление списка
await loadBots()
```

### Получение списка ботов

Для получения списка ботов используется endpoint `GET /api/bots/list`:

**Формат ответа:**
```typescript
{
  success: true,
  bots: [
    {
      id: string,              // ID бота
      token: string,           // Токен бота
      botName: string | null,  // Название бота
      botUsername: string | null, // Username бота
      projectId: string,       // ID проекта
      channelsCount: number    // Количество каналов, для которых поступил хотя бы один вебхук
    },
    // ... остальные боты
  ]
}
```

**Особенности:**
- Возвращает список всех ботов проекта (требует query параметр `projectId`)
- Проверяет права доступа к проекту (только участники или админ могут видеть ботов)
- Для каждого бота автоматически подсчитывается количество каналов из таблицы `TelegramChats` с `chatType === 'channel'`
- Если для бота нет каналов, `channelsCount` будет равен `0`
- В случае ошибки подсчёта каналов для конкретного бота, значение `channelsCount` устанавливается в `0`, но сам бот всё равно возвращается в списке

**Пример использования:**
```typescript
const result = await apiGetBotsListRoute.run(ctx)

if (result.success && result.bots) {
  for (const bot of result.bots) {
    console.log(`Бот ${bot.botName}: ${bot.channelsCount} каналов`)
  }
}
```

### Безопасность

При работе с токенами ботов реализованы следующие меры безопасности:

1. **Авторизация**: Все операции требуют авторизованного пользователя (`requireRealUser`)
2. **Проверка прав доступа**: Пользователь должен быть участником проекта (или админом) для работы с ботами
3. **Валидация данных**: Проверка наличия обязательных полей (токен, ID бота, projectId)
4. **Проверка дубликатов**: Предотвращение добавления одинаковых токенов в проект
5. **Логирование**: Все операции логируются для аудита и отладки

### Обработка ошибок

Возможные ошибки при работе с токенами:

**При добавлении:**
- `Токен обязателен` - не передан токен в запросе
- `Токен недействителен` - токен не прошёл валидацию через Telegram API
- `Этот токен уже добавлен` - токен уже существует у пользователя

**При удалении:**
- `ID бота обязателен` - не передан ID бота в запросе
- `Бот не найден` - бот с указанным ID не существует
- `Проект бота не найден` - проект, к которому привязан бот, не найден
- `Нет доступа к этому проекту` - попытка удалить бота из проекта, участником которого пользователь не является

Все ошибки логируются через систему `Debug` с указанием контекста и stack trace.

## Работа с вебхуками

Проект поддерживает автоматическую обработку входящих вебхуков от Telegram для всех зарегистрированных ботов. Вебхуки сохраняются в таблицу `TelegramWebhooks` и отправляются владельцу бота через WebSocket в реальном времени.

### Регистрация webhook при добавлении бота

При добавлении нового бота через `POST /api/bots/add` автоматически выполняется:

1. **Создание записи бота** в таблице `BotTokens`
2. **Регистрация webhook в Telegram**:
   - Генерация URL webhook через `apiWebhookRoute({ id: botId }).url()`
   - Отправка запроса к `https://api.telegram.org/bot{token}/setWebhook` с параметром `url`
   - Логирование результата регистрации

### Обработка входящих вебхуков

Вебхуки от Telegram обрабатываются через endpoint `POST /api/webhook/:id`:

1. **Получение вебхука**:
   - Telegram отправляет POST запрос на `/api/webhook/:id`, где `:id` - это ID бота
   - Endpoint **не требует авторизации**, так как Telegram отправляет запросы напрямую

2. **Валидация бота**:
   - Поиск бота в таблице `BotTokens` по ID из URL параметра
   - Проверка наличия `projectId` у бота
   - Получение проекта по `bot.projectId` для определения участников

3. **Сохранение вебхука**:
   - Извлечение `update_id` из данных вебхука (если присутствует)
   - Создание записи в таблице `TelegramWebhooks` с полями:
     - `botId` - ID бота
     - `updateId` - ID обновления (опционально)
     - `rawData` - сырые данные вебхука
     - `receivedAt` - время получения

4. **Сохранение информации о чате**:
   - Извлечение информации о чате из вебхука через функцию `extractChatFromUpdate()`
   - Если вебхук содержит информацию о чате (поле `chat`), чат сохраняется или обновляется в таблице `TelegramChats`
   - Используется эксклюзивная блокировка (`runWithExclusiveLock`) для предотвращения race condition при параллельных запросах
   - Используется метод `createOrUpdateBy` для предотвращения дубликатов
   - Сохраняются поля: `chatId`, `botId`, `projectId`, `chatType`, `chatTitle`, `chatUsername`, `firstSeenAt`, `lastSeenAt`
   - Если вебхук не содержит информацию о чате, запись в таблицу не создаётся

5. **Отправка через WebSocket**:
   - Получение списка участников проекта из `project.members`
   - Отправка вебхука каждому участнику проекта через уникальный `socketId`: `webhooks-${member.userId}`
   - Отправка данных через `sendDataToSocket()` с типом события `webhook`
   - Данные включают: `timestamp`, `botId`, `botName`, `botUsername`, `raw` (сырые данные)

6. **Ответ Telegram**:
   - Всегда возвращается `200 OK`, даже при ошибках
   - Это предотвращает повторные запросы от Telegram

### Просмотр вебхуков

Вебхуки можно просматривать на странице `/webhooks`:

- **Фильтрация по боту**: опциональный query параметр `?botId=...`
- **Real-time обновления**: новые вебхуки отображаются автоматически через WebSocket
- **API для получения списка**: `GET /api/webhook/list?limit=30&botId=...`

### Проверка webhook

Для проверки текущего состояния webhook в Telegram используется endpoint `GET /api/webhook/check/:id`:

- Запрос к `https://api.telegram.org/bot{token}/getWebhookInfo`
- Сравнение текущего URL в Telegram с ожидаемым URL
- Возврат информации о webhook и флага `isCorrect`

### Безопасность

При работе с вебхуками реализованы следующие меры безопасности:

1. **Изоляция по проектам**: Вебхуки отправляются всем участникам проекта, к которому принадлежит бот, через уникальный `socketId` для каждого участника
2. **Валидация бота**: Проверка существования бота и наличия `projectId` перед обработкой
3. **Валидация проекта**: Проверка существования проекта и получение списка участников перед отправкой вебхуков
4. **Логирование**: Все операции логируются для аудита и отладки
5. **Обработка ошибок**: Все ошибки обрабатываются без прерывания работы, всегда возвращается `200 OK` для Telegram

## Логирование

Проект использует централизованную систему логирования с настраиваемыми уровнями детализации. Все логи структурированы и содержат контекстную информацию для упрощения отладки.

### Архитектура

Система логирования состоит из следующих компонентов:

- **`shared/debug.ts`** - класс `Debug` для структурированного логирования
- **`lib/logging.ts`** - управление уровнем логирования и кэшированием
- **Таблица настроек** - хранение выбранного уровня логирования в `TgChannelAnalyticsSettings` с ключом `log_level`

### Уровни логирования

Поддерживаются три уровня логирования (от наиболее детального к наименее):

| Уровень | Описание | Когда использовать |
|---------|----------|-------------------|
| **info** | Информационные сообщения | Успешные операции, начало обработки запросов, количество найденных записей, обновление кэша |
| **warn** | Предупреждения | Отсутствие данных, некорректные параметры, проблемы с кэшем, использование значений по умолчанию |
| **error** | Ошибки выполнения | Ошибки базы данных, ошибки валидации, исключения с stack trace |

Дополнительно используется уровень **throw** для критических ошибок, которые прерывают выполнение программы.

### Настройка уровня логирования

Уровень логирования настраивается через страницу настроек (`/settings`):

1. Перейдите на страницу настроек
2. В разделе "Log Level" выберите нужный уровень:
   - **Info** - максимальная детализация (все логи)
   - **Warn** - только предупреждения и ошибки
   - **Error** - только ошибки (по умолчанию)
3. Нажмите "Сохранить"

Уровень логирования сохраняется в таблице `TgChannelAnalyticsSettings` с ключом `log_level` и кэшируется на 60 секунд для оптимизации производительности.

### Использование в коде

#### Серверная часть (API роуты, страницы)

```typescript
import { Debug } from '../shared/debug'
import { applyDebugLevel } from '../lib/logging'

export const apiGetBotsListRoute = app.get('/list', async (ctx, req) => {
  // 1. Применяем уровень логирования из настроек (обязательно в начале)
  await applyDebugLevel(ctx, 'api/bots/list')
  
  // 2. Логируем начало операции
  Debug.info(ctx, '[api/bots/list] Начало обработки запроса')
  
  try {
    // 3. Логируем успешные операции
    const bots = await BotTokens.findAll(ctx, { ... })
    Debug.info(ctx, `[api/bots/list] Найдено ботов: ${bots.length}`)
    
    // 4. Логируем предупреждения
    if (bots.length === 0) {
      Debug.warn(ctx, `[api/bots/list] У пользователя нет ботов`)
    }
    
    return { success: true, bots }
  } catch (error: any) {
    // 5. Логируем ошибки с stack trace
    Debug.error(ctx, `[api/bots/list] Ошибка: ${error.message}`, 'E_GET_BOTS_LIST')
    Debug.error(ctx, `[api/bots/list] Stack trace: ${error.stack || 'N/A'}`)
    return { success: false, error: error.message }
  }
})
```

#### Клиентская часть (Vue компоненты)

В Vue компонентах используется стандартный `console` для логирования:

```typescript
const loadBots = async () => {
  console.log('[BotsPage] Начало загрузки списка ботов')
  
  try {
    const result = await apiGetBotsListRoute.run(ctx)

    } else {
      console.warn('[BotsPage] Ошибка в ответе API:', result.error)
    }
  } catch (e: any) {
    console.error('[BotsPage] Исключение при загрузке:', e)
    console.error('[BotsPage] Stack trace:', e.stack)
  }
}
```

### Структура логов

Все логи имеют единообразную структуру:

```
[контекст] Сообщение: детали
```

**Примеры:**
- `[api/bots/list] Начало обработки запроса на получение списка ботов`
- `[api/bots/list] Пользователь авторизован: userId=12345`
- `[api/bots/list] Найдено ботов: 3`
- `[logging] Кэш уровня логирования обновлён: error -> info`
- `[settings] Инициализация настроек по умолчанию`

### Правила логирования

1. **Всегда применяйте уровень логирования в начале обработчика:**
   ```typescript
   await applyDebugLevel(ctx, 'контекст')
   ```

2. **Используйте префиксы контекста:**
   - `[api/bots/list]` - для API роутов
   - `[logging]` - для модуля логирования
   - `[settings]` - для работы с настройками
   - `[BotsPage]` - для Vue компонентов

3. **Логируйте ключевые этапы:**
   - Начало обработки запроса
   - Проверка авторизации
   - Успешные операции с результатами
   - Предупреждения
   - Ошибки с stack trace

4. **Для ошибок всегда указывайте:**
   - Код ошибки (если применимо)
   - Stack trace
   - Контекстную информацию

5. **Избегайте логирования чувствительных данных:**
   - Не логируйте пароли, токены, персональные данные
   - Используйте маскирование для токенов

| `pages/SettingsPage.vue` | UI для выбора уровня логирования |

### Кэширование уровня логирования

Уровень логирования кэшируется на 60 секунд (TTL) для оптимизации производительности. При первом обращении к `applyDebugLevel()` уровень загружается из базы данных и сохраняется в памяти. Последующие вызовы в течение 60 секунд используют кэшированное значение.

При изменении уровня логирования через UI кэш обновляется немедленно.

### Логирование в тестах

Все тесты используют систему логирования проекта:
- Применение уровня логирования через `applyDebugLevel()`
- Использование `Debug.info()`, `Debug.warn()`, `Debug.error()` для структурированного логирования
- Логи отправляются через WebSocket в реальном времени

### Очистка тестовых данных

Тесты, которые создают данные в базе (например, тестовых ботов), автоматически очищают их после выполнения:

```typescript
try {
  // Создание тестового бота
  const testBot = await BotTokens.create(ctx, { ... })
  
  // Выполнение теста
  // ...
} finally {
  // Гарантированная очистка
  if (testBot && testBot.id) {
    await BotTokens.delete(ctx, testBot.id)
  }
}
```

### Безопасность

- Все тесты требуют авторизации (`requireAnyUser`)
- Тесты изолированы по пользователям (используется `ctx.user.id`)
- Тестовые данные создаются только для текущего пользователя
- Чувствительные данные (токены) маскируются в логах

### Best Practices

1. **Тестируйте реальный код** - тесты должны вызывать существующие API routes и таблицы, а не создавать новую логику
2. **Используйте единый источник истины** - все тесты определяются в `test-definitions.ts`
3. **Очищайте тестовые данные** - всегда удаляйте созданные в тестах данные
4. **Логируйте ключевые этапы** - используйте `Debug.info()` для отслеживания выполнения
5. **Обрабатывайте ошибки** - используйте `Debug.error()` и `Debug.throw()` для критических ошибок
6. **Проверяйте авторизацию** - тесты, требующие пользователя, должны проверять `ctx.user`

### Пример добавления нового теста

1. Добавьте тест в `tests/shared/test-definitions.ts`:
```typescript
{
  name: 'api',
  title: 'Тесты API endpoints',
  tests: [
    // ... существующие тесты
    { name: 'new_test', description: 'Описание нового теста' }
  ]
}
```

2. Добавьте обработку в `tests/api/run-tests.ts`:
```typescript
async function runApiTest(ctx: any, testName: string) {
  switch (testName) {
    // ... существующие тесты
    case 'new_test':
      // Логика теста
      const result = await apiSomeRoute.run(ctx, { ... })
      if (!result.success) {
        Debug.throw(ctx, 'Тест провален', 'E_TEST_FAILED')
      }
      return { success: true, message: 'Тест пройден' }
  }
}
```

3. Тест автоматически появится на странице `/tests` и в `/tests/ai`
