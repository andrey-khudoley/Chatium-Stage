## 1. Платформенные инварианты Chatium

Эти правила нарушать нельзя — продакшен сломается или код не пройдёт ревью.

**Глобальные объекты:**

- `ctx` и `app` — глобальные, не импортировать.
- Логирование — только `ctx.account.log()`. **Не** `console.log`. Если в проекте есть библиотека-обёртка — использовать её.

**File-based роутинг:**

- Один файл = один роут.
- Предпочтительный путь — `'/'` (без тильды в URL).
- Ссылки на роуты — через `withProjectRoot(route.url())` или `withProjectRootAndSubroute(base, '/sub')`. **Никакого хардкода URL.**
- На сервере для редиректов — относительные пути `./`, `../`.

**Vue / клиент:**

- Heap и таблицы — **только на сервере**. На клиенте `.run()` допустим лишь у роутов с `// @shared-route`.
- Данные в Vue — через SSR-пропсы или `fetch`, **никогда не импортировать** `tables/`, `repos/`, `lib/` в `.vue`.
- Импорт в Vue только из `shared/*` с `// @shared`.

**Зависимости:**

- npm недоступен. Использовать встроенные модули: `@app/*` (`@app/auth`, `@app/request`, `@app/sync`, `@app/feed`, `@app/heap`), `@pay/sdk`, `@user-notifier/sdk`, `@sender`, `@ai-agents/sdk`, `@start/sdk`. Полный перечень — `inner/docs/025-app-modules.md`.

**Heap (БД):**

- Подсчёт — `countBy`, **не** `findAll().length`.
- Фильтры — через `where`, **не** `filter`.
- Order — `[{ field: 'asc' | 'desc' }]`, **не** `[{ field, direction }]`.
- Money — только методы (`.add()`, `.subtract()`, `.multiply()`), **не** обычная арифметика.
- Race conditions — `runWithExclusiveLock`.

**Авторизация:**

- Защищённые эндпоинты — `requireRealUser(ctx)` или `requireAccountRole(ctx, 'admin')` первой строкой обработчика.
- Никогда не отдавать данные другого `accountId` без явной проверки.

**TypeScript:**

- `// @ts-ignore` допустим **только** для системных модулей Chatium (`@app/*`, `@start/sdk`, `@pay/sdk`, `@sender`, `@user-notifier/sdk`), которые не имеют локально доступных типов. **Не** ставить на собственный код.

**Документация платформы:** `inner/docs/`. Навигатор — `000-summ.md`. Темы 001–046 + E-серия (E03–E06).

## 2. Дата и время — только через shell

Если в любом тексте (отчёте, changelog, LLM-логе, имени файла) нужны текущая дата или время — **получай через shell**, не из внутренних часов модели:

```bash
date "+%d-%m-%Y %H:%M:%S %Z"   # дата + время + таймзона
date "+%Y-%m-%d"                # для changelog
date "+%y%m%d"                  # для имён файлов YYMMDD
```

Это правило применяется к плейсхолдерам `[дата]`, «текущая дата», `YYMMDD` и аналогам.

## 3. Документация после изменений кода

При изменениях в коде **папки проекта-приложения** (каталог с `index.tsx` / `index.ts` и собственным `docs/`):

1. **README.md** — обновить «Текущее состояние», «Навигацию по docs», «TODO», «Changelog» (запись с датой через shell).
2. **.CHATIUM-LLM.md** — словарь для LLM: что за проект, где README, где docs/. Обновлять только при изменении структуры указателей.
3. **docs/architecture.md, docs/api.md, docs/data.md** — по затронутым областям.

**Когда документация НЕ нужна:** изменения в общих библиотеках, `shared/`, инфраструктуре, `.cursor/`, `.claude/`, gateway. Документация для общих модулей ведётся в проектах, которые их используют.

Делегируется субагенту `docs-keeper` после крупных изменений.

## 4. LLM-логи в `docs/LLM/`

При значимых изменениях кода ищи в проекте каталог `./docs/LLM/` и логируй туда текущий запрос и ответ (без листинга кода — только описание что сделано).

**Правила нумерации (обязательно):**

- Перед созданием — `ls ./docs/LLM/`.
- Имя нового файла начинается с **следующего порядкового номера** после максимального существующего. Формат: `NNNN_DD-MM-YYYY_HH-MM.md` (4 цифры с ведущими нулями).
- Если текущая переписка продолжает ту же задачу → **дописать** в существующий файл.
- Если новая тема → создать новый файл со следующим номером.
- Дату и время в имени и тексте брать через shell (`date "+%d-%m-%Y %H:%M"`).
- Если каталога нет — создать.

В каждой записи указывай дату и точное время отдельной строкой.

## 5. Готовность к коммиту

Если пользователь явно или по контексту готовится к коммиту («готово?», «можно коммитить?», «всё ок?», «финальный ревью», «посмотри перед коммитом») — **предложи** запустить `/check` перед коммитом, не дожидаясь явного запроса.

## 6. Архитектура агентов и команд

В `.claude/agents/` живут **12 субагентов**:

| Агент                          | Назначение                          | Когда использовать                    |
| ------------------------------ | ----------------------------------- | ------------------------------------- |
| `task-formalizer`              | Формализация задачи                 | Шаг 1 любого конвейера                |
| `planner`                      | План реализации                     | После формализации                    |
| `plan-reviewer`                | Ревью плана                         | После плана                           |
| `completeness-reviewer`        | Полнота ревью (плана/кода)          | После plan-reviewer / code-reviewer   |
| `code-reviewer`                | Ревью кода по 10 областям           | После реализации                      |
| `standards-checker`            | Стандарты 001-standards.md          | Подвызов от verification-runner       |
| `file-based-routing-checker`   | Проверка роутинга                   | Подвызов от verification-runner       |
| `runtime-architecture-checker` | Рантайм-баги и архитектура          | Подвызов от verification-runner       |
| `verification-runner`          | Все технические проверки            | После code-reviewer; через `/check`   |
| `docs-keeper`                  | Обновление docs/README/.CHATIUM-LLM | После реализации, в шаге 12 конвейера |
| `discussion-architect`         | Обсуждение задачи до реализации     | Перед `/pipeline`                     |
| `pipeline-orchestrator`        | Полный 12-шаговый конвейер          | Активируется через `/pipeline`        |

| Команда     | Что делает                                                         |
| ----------- | ------------------------------------------------------------------ |
| `/pipeline` | Запускает полный 12-шаговый конвейер через `pipeline-orchestrator` |
| `/pp`       | Параллельный конвейер с декомпозицией и фоновыми воркерами         |
| `/check`    | Технические проверки через `verification-runner`                   |
| `/toprod`   | Деплой ассистента из dev в prod workspace                          |

## 7. Правила для оркестратора (pipeline-orchestrator)

Если ты `pipeline-orchestrator`:

- Шаги 2–5, 8–9, 10 конвейера — **только** через Agent tool. Запрещено выполнять их работу в своём чате.
- Все 12 шагов обязательны. Финальный отчёт — обязательная часть.
- Цикл правок ≤ 3 итераций. После — фиксируй остаточные риски, не «допиливай любой ценой».

См. полные инструкции в `.claude/agents/pipeline-orchestrator.md`.

## 8. Стиль ответа Claude

- **Язык:** русский. Технические термины — без перевода в скобках.
- **Тон:** деловой, без вступлений «Конечно!», «С удовольствием!», без эмодзи в обычных сообщениях.
- **Краткость:** короткий ответ предпочтителен длинному; не повторяй вопрос пользователя своими словами.
- **Никогда не пиши код**, если задача — формализация / планирование / ревью / обсуждение.
- **Структура отчётов** — единая по агентам (см. их файлы).

## 9. Workspace-специфика

- Workspace `s.chtm.khudoley.pro` — **dev**. Изменения здесь.
- Workspace `p.chtm.khudoley.pro` — **prod**. Изменения только через команду `/toprod` (для каталога `k/assistant`).
- В workspace может быть несколько проектов под `p/units/`, `p/saas/`, `tg/` и т.д. Корень проекта — каталог с `index.tsx` / `index.ts`. **Не** путай его с корнем workspace.
- SDK и client-проекты в `p/units/chatiumclub/` живут в одном dev-workspace вместе с gateway. План «отдельный workspace» применяется только к проду.
