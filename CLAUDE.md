## 1. Платформенные инварианты Chatium

Эти правила нарушать нельзя — продакшен сломается или код не пройдёт ревью.

**Глобальные объекты:**

- `ctx` и `app` — глобальные, не импортировать.
- Логирование — только `ctx.account.log()`. **Не** `console.log`. Если в проекте есть библиотека-обёртка — использовать её.

**File-based роутинг:**

- Один файл = один роут.
- Предпочтительный путь — `'/'` (без тильды в URL).
- Ссылки на роуты — через `withProjectRoot(route.url())` или `withProjectRootAndSubroute(base, '/sub')`. **Никакого хардкода URL.**
- На сервере для редиректов — относительные пути `./`, `../`.

**Vue / клиент:**

- Heap и таблицы — **только на сервере**. На клиенте `.run()` допустим лишь у роутов с `// @shared-route`.
- Данные в Vue — через SSR-пропсы или `fetch`, **никогда не импортировать** `tables/`, `repos/`, `lib/` в `.vue`.
- Импорт в Vue только из `shared/*` с `// @shared`.

**Зависимости:**

- npm недоступен. Использовать встроенные модули: `@app/*` (`@app/auth`, `@app/request`, `@app/sync`, `@app/feed`, `@app/heap`), `@pay/sdk`, `@user-notifier/sdk`, `@sender`, `@ai-agents/sdk`, `@start/sdk`. Полный перечень — `inner/docs/025-app-modules.md`.

**Heap (БД):**

- Подсчёт — `countBy`, **не** `findAll().length`.
- Фильтры — через `where`, **не** `filter`.
- Order — `[{ field: 'asc' | 'desc' }]`, **не** `[{ field, direction }]`.
- Money — только методы (`.add()`, `.subtract()`, `.multiply()`), **не** обычная арифметика.
- Race conditions — `runWithExclusiveLock`.

**Авторизация:**

- Защищённые эндпоинты — `requireRealUser(ctx)` или `requireAccountRole(ctx, 'admin')` первой строкой обработчика.
- Никогда не отдавать данные другого `accountId` без явной проверки.

**TypeScript:**

- `// @ts-ignore` допустим **только** для системных модулей Chatium (`@app/*`, `@start/sdk`, `@pay/sdk`, `@sender`, `@user-notifier/sdk`), которые не имеют локально доступных типов. **Не** ставить на собственный код.

**Документация платформы:** `inner/docs/`. Навигатор — `000-summ.md`. Темы 001–046 + E-серия (E03–E06).

## 2. Дата и время — только через shell

Если в любом тексте (отчёте, changelog, LLM-логе, имени файла) нужны текущая дата или время — **получай через shell**, не из внутренних часов модели:

```bash
date "+%d-%m-%Y %H:%M:%S %Z"   # дата + время + таймзона
date "+%Y-%m-%d"                # для changelog
date "+%y%m%d"                  # для имён файлов YYMMDD
```

Это правило применяется к плейсхолдерам `[дата]`, «текущая дата», `YYMMDD` и аналогам.

При изменениях в коде **папки проекта-приложения** (каталог с `index.tsx` / `index.ts` и собственным `docs/`):

## 4. LLM-логи в `docs/LLM/`

При значимых изменениях кода ищи в проекте каталог `./docs/LLM/` и логируй туда текущий запрос и ответ (без листинга кода — только описание что сделано).

**Правила нумерации (обязательно):**

- Перед созданием — `ls ./docs/LLM/`.
- Имя нового файла начинается с **следующего порядкового номера** после максимального существующего. Формат: `NNNN_DD-MM-YYYY_HH-MM.md` (4 цифры с ведущими нулями).
- Если текущая переписка продолжает ту же задачу → **дописать** в существующий файл.
- Если новая тема → создать новый файл со следующим номером.
- Дату и время в имени и тексте брать через shell (`date "+%d-%m-%Y %H:%M"`).
- Если каталога нет — создать.

В каждой записи указывай дату и точное время отдельной строкой.

### 6a. Маршрутизация моделей

Модель агента в конвейере определяется **уровнем** `/ppN` (см. §6b и матрицу в `pp-orchestrator.md`), а не фиксированным frontmatter. Базовые роли моделей:

- **Haiku** (механика, дёшево): `standards-checker`, `file-based-routing-checker`, `completeness-reviewer`; на низких уровнях — и формализация.
- **Sonnet** (генерация и ревью по умолчанию): `implementer`, `task-formalizer`, `docs-keeper`, `planner`/`plan-reviewer`/`code-reviewer`/`chatium-platform-checker` на средних уровнях.
- **Opus** (тяжёлое рассуждение, верхние уровни): `planner`, `plan-reviewer`, `code-reviewer`, `runtime-architecture-checker`, `chatium-platform-checker`, `discussion-architect` — на pp7+.

Принцип 2026: **минимально достаточная модель** + **один спавн на роль на фазу** + **диф-скоуп файлов каждому агенту**. Opus — не дефолт, а инструмент верхних уровней. Модель переопределяется параметром `model` в Agent tool на каждом вызове по матрице уровня (frontmatter агентов не редактируем).

### 6b. Уровни конвейера `/pp1`…`/pp10`

Один регулятор 1–10 крутит глубину и стоимость. Полная матрица (Form/Plan/PlanRev/Impl/CodeRev/Gate/Cmpl/Fix/Docs/Report и модели) — в `.claude/agents/pp-orchestrator.md`. Кратко:

- **pp1–2** — без агентов: правка основным чатом + Bash типы/стиль. Опечатки, однострочники.
- **pp3–4** — `implementer` (Sonnet) + одно консолидированное `code-reviewer` (Sonnet, покрывает стандарты/роутинг/рантайм/платформу одним проходом). Локальная фича.
- **pp5–6** — + формализация, план с ревью, отдельные checker'ы (Haiku) и платформенный гейт (Sonnet). Фича со связями.
- **pp7–8** — план/ревью/гейты на Opus, полный fan-out верификации, completeness (pp8). Сложная фича, платформенные тонкости.
- **pp9–10** — максимум: adversarial-проход, loop-until-clean, мульти-кандидатный план (pp10), бюджет до 1.5–2 ч. Крупные/рискованные задачи.

Выбирай уровень под сложность задачи — это и есть оптимизация по токенам и времени. Не гоняй pp8+ на тривиальном.

## 7. Правила для оркестратора (`pp-orchestrator`)

Когда ведёшь конвейер по `/ppN` (роль оркестратора исполняет основной чат, не субагент):

- **Уровень `N` — единственный регулятор.** Бери профиль из матрицы `pp-orchestrator.md` (модели, шаги, параллелизм, лимит Fix). Не добавляй и не выкидывай фазы вне профиля уровня.
- Делегирование ролей — **только** через Agent tool. Сам в чате — оркестрация, типы/стиль/тесты через Bash, сборка отчёта.
- **Код пишет `implementer`** (на pp1–2 — основной чат сам, т.к. спавн дороже правки).
- **Один спавн на роль на фазу.** Не дублируй вызов одного агента (кроме явного multi-candidate плана на pp10). Параллельный fan-out = разные роли в одном сообщении.
- **Каждому агенту — точный список файлов** (из `git diff`/плана), не «весь проект».
- Верификацию веди по плейбуку `verification-runner`; через Agent его НЕ запускай.
- Метрики цикла — на уровнях с пометкой «+метрики» (pp5+) **обязательная пара блоков**: «## Метрики цикла» (`scripts/pipeline-usage.mjs`, токены) И «## Лимиты подписки» (`scripts/usage-limits.mjs`, сколько % лимита ушло/осталось). На START — `usage-limits.mjs --track-start <START>` вместе с таймстемпом (фиксирует начальный % + детач-таймер на сброс 5ч-окна), в фазе G — `--track-end <START>` (Δ за прогон + остаток окон 5ч/7дн). Если track-start пропущен — фоллбэк `usage-limits.mjs` (снапшот без Δ). **Отчёт без блока «## Лимиты подписки» на pp5+ — недоделанный.**
- Цикл правок ≤ Fix из матрицы. После — фиксируй остаточные риски, не «допиливай любой ценой».

См. полный плейбук и матрицу уровней в `.claude/agents/pp-orchestrator.md`.
