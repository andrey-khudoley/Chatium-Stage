## Обязательный формат

### Базовая структура

Каждый коммит должен следовать формату:

**`<type>(<scope>): <summary>`**

Где:
- **type** — тип изменения (строго из фиксированного списка)
- **scope** — область изменений (модуль/подсистема/папка/интеграция)
- **summary** — краткое описание того, что сделано (1 действие, без воды)

### Примеры

```
feat(auth): добавлена silent-accept оферты при оплате
fix(gc): исправлен дубль UTM при повторной отправке формы
perf(heap): добавлен индекс на events(ts, dataset)
```

---

## Типы коммитов (type)

### Фиксированный словарь типов

Используйте **только** эти значения:

| Тип | Описание | Когда использовать |
|-----|----------|-------------------|
| **feat** | Новая функциональность | Добавлена новая возможность для пользователя |
| **fix** | Исправление бага | Исправлена ошибка в существующем функционале |
| **perf** | Оптимизация производительности | Ускорение работы, оптимизация запросов, кэширование |
| **refactor** | Рефакторинг кода | Переработка без изменения поведения |
| **docs** | Документация | Изменения в документации, комментариях, README |
| **test** | Тесты | Добавление или изменение тестов |
| **build** | Сборка и зависимости | Изменения в package.json, версиях, зависимостях |
| **ci** | CI/CD пайплайны | Изменения в GitHub Actions, деплой-скриптах |
| **chore** | Технические изменения | Скрипты, конфиги, мелочи без функциональных изменений |
| **style** | Форматирование | Изменения форматирования без изменения логики |
| **revert** | Откат коммита | Откат предыдущего коммита |

### Правило выбора между feat и fix

Если сомневаетесь между `feat` и `fix`, задайте вопрос:

**"Пользователь получил новую возможность?"**

- ✅ **Да** → `feat` (даже если это исправление недоработки)
- ❌ **Нет** → `fix` (исправление существующего функционала)

### Примеры выбора типа

```
✅ feat(auth): добавлена двухфакторная аутентификация
✅ fix(auth): исправлена проверка токена при истечении срока
✅ perf(heap): оптимизирован запрос findAll с фильтрами
✅ refactor(api): вынесена валидация в отдельный модуль
✅ docs(readme): обновлена инструкция по установке
✅ test(analytics): добавлены unit-тесты для writeWorkspaceEvent
```

## Область изменений (scope)

### Правила выбора scope

**scope** должен быть:
- Коротким (1–15 символов)
- Стабильным (не меняется часто)
- Без пробелов
- Предпочтительно нижний регистр и дефисы

### Рекомендуемые категории scope

#### По подсистеме

- `api` — API endpoints, роутинг
- `ui` — пользовательский интерфейс, компоненты
- `auth` — авторизация, аутентификация
- `billing` — платежи, подписки
- `analytics` — аналитика, события
- `sync` — синхронизация данных
- `webhook` — обработка webhook'ов

#### По интеграции

- `gc` — GetCourse
- `amo` — AmoCRM
- `tg` — Telegram
- `chatium` — Chatium core
- `heap` — база данных Heap
- `pg` — PostgreSQL
- `mongo` — MongoDB

#### По инфраструктуре

- `infra` — инфраструктура
- `nft` — NFT модуль
- `ci` — CI/CD
- `deploy` — деплой
- `docker` — Docker конфигурация
- `wsl` — WSL специфичные изменения

#### По домену/пакету

- `core` — ядро системы
- `admin` — админ-панель
- `worker` — воркеры, джобы
- `sdk` — SDK для внешних разработчиков

### Что делать, если изменили несколько зон?

**Вариант 1:** Выберите главный scope (где основное изменение)

```
✅ refactor(api): вынесена валидация в shared модуль
   (затронуты api, shared, но главное изменение в api)
```

**Вариант 2:** Разбейте на несколько коммитов

```
✅ refactor(api): вынесена валидация в shared модуль
✅ refactor(heap): использована новая валидация в таблицах
```

## Описание изменений (summary)

### Правила написания summary

**summary** пишется:
- **Прошедшим временем в страдательном залоге** (добавлена, создана, исправлена, убрана, вынесена) — **предпочтительно**
- **Инфинитивом** (добавить, исправить, убрать, вынести) — допустимо, но только если не звучит как команда/побуждение
- **Без точки в конце**
- **До 72 символов** (желательно 50–60)
- **Одно действие** = один коммит
- **Описывает то, что УЖЕ было сделано**, а не то, что нужно сделать

### Критически важно

**summary должно звучать как описание выполненной работы, а не как команда или побуждение к действию.**

❌ **Неправильно (звучит как команда):**
```
feat(ui): создать новую страницу
feat(api): добавить endpoint для пользователей
fix(heap): исправить баг в запросе
```

✅ **Правильно (описывает выполненную работу):**
```
feat(ui): создана новая страница
feat(api): добавлен endpoint для пользователей
fix(heap): исправлен баг в запросе
```

### Что обязательно отражать

✅ **Конечный эффект** — что изменилось для пользователя/системы
✅ **Ключевой контекст** — для чего/где (если неочевидно)
✅ **Факт выполнения** — что было сделано (прошедшее время)

### Что запрещено

❌ "исправления", "правки", "обновил", "разное", "мелочи"
❌ "WIP", "temp", "test", "fix", "update" без уточнения *чего*
❌ Слишком общие формулировки без конкретики
❌ **Повелительное наклонение** — "создать", "добавить", "исправить" (когда звучит как команда)
❌ **Будущее время** — "будет добавлено", "создастся"

### Примеры хороших и плохих summary

#### ✅ Хорошо (прошедшее время, страдательный залог)

```
feat(ui): создана новая страница профиля
feat(api): добавлен endpoint для получения списка пользователей
fix(heap): исправлена утечка памяти в findAll
perf(pg): добавлен индекс на events(ts, dataset)
refactor(analytics): вынесена нормализация utm в отдельный модуль
fix(heap): убран undefined в конфиге сегментов
feat(auth): добавлена silent-accept оферты при оплате
```

#### ✅ Хорошо (инфинитив, когда не звучит как команда)

```
refactor(analytics): вынести нормализацию utm в отдельный модуль
fix(gc): исправить дубль UTM при повторной отправке формы
perf(heap): оптимизировать запрос findAll с фильтрами
```

#### ❌ Плохо (звучит как команда/побуждение)

```
feat(ui): создать новую страницу
feat(api): добавить endpoint
fix(heap): исправить баг
refactor: правки
fix: исправления
update: обновил код
WIP: работа над фичей
test: тесты
```

## Тело коммита (body)

### Когда добавлять body

Добавляйте body, если:
- **Неочевидно "почему"** изменение было сделано
- Есть **нюансы миграции** или совместимости
- Требуется **дополнительный контекст** для понимания

### Формат body

После пустой строки после summary:

- 1–5 строк, списком
- Без лишней прозы
- Структура:
  - **Причина** — почему это изменение
  - **Суть подхода** — как решено
  - **Риски/совместимость** — что может сломаться
  - **Как проверить** — как убедиться, что работает

### Пример body

```
fix(tg): корректно определять chat.id для linked-channel сообщений

- Причина: from может быть фейковым пользователем в non-channel чатах
- Решение: брать chat.id из message.chat во всех релевантных апдейтах
- Проверка: update.message и update.channel_post на тестовом боте
```

## Подвал коммита (footer)

### Ссылки на задачи и тикеты

Внизу (после пустой строки) добавляйте ссылки:

- `Refs: GC-142` — ссылка на задачу
- `Closes: AMO-57` — закрывает задачу
- `Fixes: #123` — исправляет issue

### Breaking changes

Если изменение ломающее, добавьте:

**`BREAKING CHANGE: ...`**

Кратко опишите:
- Что сломалось
- Что нужно сделать для миграции

### Пример footer

```
feat(api): изменить схему events payload

BREAKING CHANGE: поле payload.utm теперь объект, а не строка; обновить парсер в аналитике
Refs: AN-88
```

## Шаблоны для частых случаев

### Быстрый шаблон (90% случаев)

```
<type>(<scope>): <глагол в прошедшем времени> + <объект> + <уточнение>
```

**Примеры:**
```
feat(auth): добавлена двухфакторная аутентификация
fix(heap): исправлена утечка памяти в findAll
perf(api): ускорен запрос списка пользователей
feat(ui): создана новая страница профиля
```

### Шаблон для бага

```
fix(<scope>): исправлен <симптом> при <условии>
```

**Примеры:**
```
fix(gc): исправлен дубль UTM при повторной отправке формы
fix(analytics): исправлено падение при отсутствии utm_source
fix(heap): исправлена race condition при параллельных обновлениях
```

### Шаблон для рефакторинга

```
refactor(<scope>): упрощена/вынесена/переименована <что>
```

**Примеры:**
```
refactor(api): вынесена валидация в отдельный модуль
refactor(analytics): упрощена нормализация UTM меток
refactor(heap): переименован метод findById в findOne
```

### Шаблон для производительности

```
perf(<scope>): ускорен <что> за счёт <как>
```

**Примеры:**
```
perf(heap): ускорен findAll за счёт добавления индекса
perf(api): ускорен рендеринг за счёт кэширования шаблонов
perf(pg): ускорены запросы за счёт оптимизации JOIN
```

### Шаблон для новой функциональности

```
feat(<scope>): добавлен <что> для <зачем>
```

**Примеры:**
```
feat(auth): добавлена silent-accept оферты при оплате
feat(analytics): добавлена фильтрация по категориям событий
feat(gc): добавлена синхронизация сделок в реальном времени
```

## Примеры хороших коммитов

### Простые коммиты (без body)

```
feat(chatium): добавлен endpoint /spin вместо /logs
fix(heap): убран undefined в конфиге сегментов
perf(pg): добавлен индекс на events(ts, dataset)
ci(deploy): запускаются миграции перед рестартом сервиса
build(core): обновлены зависимости node-fetch -> undici
docs(api): описан контракт webhook для GetCourse
```

### Коммиты с body

```
fix(tg): корректно определяется chat.id для linked-channel сообщений

- Причина: from может быть фейковым пользователем в non-channel чатах
- Решение: брать chat.id из message.chat во всех релевантных апдейтах
- Проверка: update.message и update.channel_post на тестовом боте
```

```

- Причина: дублирование логики в 3 местах

- Риски: нужно обновить все места использования
- Проверка: unit-тесты для нормализатора
```

### Коммиты с breaking changes

```
feat(api): изменена схема events payload

BREAKING CHANGE: поле payload.utm теперь объект, а не строка; обновить парсер в аналитике
Refs: AN-88
```

```
refactor(heap): переименован метод findById в findOne

BREAKING CHANGE: метод findById удалён, использовать findOne
Refs: HEAP-42
```

### Коммиты с задачами

```
feat(auth): добавлена двухфакторная аутентификация

Refs: AUTH-15
Closes: #234
```

```
fix(gc): исправлен дубль UTM при повторной отправке формы

Fixes: GC-142
```

## Примеры коммитов для разных ситуаций

Ниже приведены примеры правильных коммитов для типичных ситуаций разработки. Все примеры используют прошедшее время в страдательном залоге, что делает их понятными и описывающими выполненную работу.

### Создание новой страницы/компонента

❌ **Неправильно:**
```
feat(ui): создать новую страницу профиля
feat(ui): добавить компонент формы
```

✅ **Правильно:**
```
feat(ui): создана новая страница профиля
feat(ui): добавлен компонент формы регистрации
feat(admin): создана страница управления пользователями
```

### Добавление нового API endpoint

❌ **Неправильно:**
```
feat(api): добавить endpoint для получения списка пользователей
feat(api): создать роут для загрузки файлов
```

✅ **Правильно:**
```
feat(api): добавлен endpoint для получения списка пользователей
feat(api): создан роут для загрузки файлов
feat(api): добавлен POST /api/users для создания пользователя
```

### Исправление багов

❌ **Неправильно:**
```
fix(heap): исправить баг в запросе
fix(ui): исправить ошибку отображения
```

✅ **Правильно:**
```
fix(heap): исправлен баг в запросе findAll с фильтрами
fix(ui): исправлена ошибка отображения модального окна
fix(auth): исправлена проверка токена при истечении срока
fix(gc): исправлен дубль UTM при повторной отправке формы
```

### Рефакторинг кода

❌ **Неправильно:**
```
refactor: правки в коде
refactor(api): упростить код
```

✅ **Правильно:**
```
refactor(api): вынесена валидация в отдельный модуль
refactor(analytics): упрощена нормализация UTM меток
refactor(heap): переименован метод findById в findOne
refactor(ui): вынесены общие стили в shared компонент
```

### Оптимизация производительности

❌ **Неправильно:**
```
perf(heap): ускорить запросы
perf(api): оптимизировать код
```

✅ **Правильно:**
```
perf(heap): ускорен findAll за счёт добавления индекса
perf(api): ускорен рендеринг за счёт кэширования шаблонов
perf(pg): добавлен индекс на events(ts, dataset)
perf(ui): оптимизирован рендеринг списка за счёт виртуализации
```

### Добавление новой функциональности

❌ **Неправильно:**
```
feat(auth): добавить двухфакторную аутентификацию
feat(analytics): добавить фильтрацию
```

✅ **Правильно:**
```
feat(auth): добавлена двухфакторная аутентификация
feat(auth): добавлена silent-accept оферты при оплате
feat(analytics): добавлена фильтрация по категориям событий
feat(gc): добавлена синхронизация сделок в реальном времени
feat(ui): добавлена тёмная тема оформления
```

### Изменение документации

❌ **Неправильно:**
```
docs: обновить документацию
docs(readme): добавить информацию
```

✅ **Правильно:**
```
docs(readme): обновлена инструкция по установке
docs(api): описан контракт webhook для GetCourse
docs(standards): добавлен раздел про именование коммитов
```

### Добавление тестов

❌ **Неправильно:**
```
test: добавить тесты
test(api): написать тесты
```

✅ **Правильно:**
```
test(analytics): добавлены unit-тесты для writeWorkspaceEvent
test(api): добавлены интеграционные тесты для /api/users
test(auth): добавлены тесты для проверки токена
```

### Изменения в зависимостях и сборке

❌ **Неправильно:**
```
build: обновить зависимости
ci: настроить деплой
```

✅ **Правильно:**
```
build(core): обновлены зависимости node-fetch -> undici
build(deps): обновлён React до версии 18.2.0
ci(deploy): запускаются миграции перед рестартом сервиса
ci(github): добавлен workflow для автоматических тестов
```

### Технические изменения (chore)

❌ **Неправильно:**
```
chore: разные правки
chore: обновить конфиги
```

✅ **Правильно:**
```
chore(infra): обновлена конфигурация Docker
chore(scripts): добавлен скрипт для очистки кэша
chore(config): обновлены настройки линтера
```

### Стилистические изменения

❌ **Неправильно:**
```
style: форматирование
style(ui): правки стилей
```

✅ **Правильно:**
```
style(ui): исправлены отступы в компонентах
style(code): применено автоматическое форматирование
style(api): исправлено форматирование в роутерах
```

### Откат изменений

```
revert(api): откат изменений в схеме events payload

Откат коммита abc123 из-за проблем с совместимостью
```

## Чеклист перед коммитом

Перед каждым коммитом проверьте:

- [ ] `type` верный и из фиксированного списка?
- [ ] `scope` понятный и стабильный?
- [ ] `summary` отвечает "что сделал" за 1 взгляд?
- [ ] `summary` до 72 символов?
- [ ] `summary` начинается с глагола в прошедшем времени (предпочтительно) или инфинитиве (если не звучит как команда)?
- [ ] `summary` звучит как описание выполненной работы, а не как команда?
- [ ] Если есть риск/неочевидность — добавлен body?
- [ ] Если связано с задачей — добавлен `Refs/Closes`?
- [ ] Если breaking change — добавлен `BREAKING CHANGE`?
- [ ] Коммит содержит одно логическое изменение?

### Быстрая проверка формата

```bash
# Проверка формата коммита (пример регулярного выражения)
^[a-z]+\([a-z-]+\): .{1,72}$
```

## Дополнительные рекомендации

### Один коммит — одно изменение

✅ **Хорошо:**
```
feat(auth): добавлена двухфакторная аутентификация
fix(heap): исправлена утечка памяти
```

❌ **Плохо:**
```
feat: добавить 2FA и исправить баг в heap
feat(ui): создать новую страницу
```

### Группировка связанных изменений

Если изменения логически связаны, можно объединить:

✅ **Хорошо:**
```
refactor(api): вынесены валидация и роутинг в отдельные модули
```

❌ **Плохо:**
```
refactor: разные правки
refactor(api): вынести валидацию и роутинг
```

### Использование русского языка

Все коммиты пишутся **на русском языке**:
- Типы и scope остаются на английском (feat, fix, auth, api)
- Summary и body — на русском

## Часто задаваемые вопросы

### Как выбрать между feat и fix?

**Вопрос:** Добавил проверку, которой не было — это feat или fix?

**Ответ:** Если пользователь получил новую возможность (даже если это исправление недоработки) — `feat`. Если исправлен баг в существующем функционале — `fix`.

**Пример:**
```
✅ feat(auth): добавлена проверка истечения токена
   (раньше проверки не было вообще)

✅ fix(auth): исправлена проверка истечения токена
   (проверка была, но работала неправильно)
```

### Можно ли использовать несколько scope?

**Вопрос:** Изменил и api, и heap — как быть?

**Ответ:** Выберите главный scope или разбейте на несколько коммитов.

**Пример:**
```
✅ refactor(api): вынесена валидация в shared модуль
✅ refactor(heap): использована новая валидация в таблицах
```

### Нужен ли body для простых изменений?

**Вопрос:** Всегда ли нужен body?

**Ответ:** Нет, только если неочевидно "почему" или есть нюансы миграции.

**Пример:**
```
✅ fix(heap): исправлена опечатка в названии метода
   (body не нужен — всё очевидно)

✅ refactor(analytics): изменён алгоритм нормализации UTM
   (body нужен — объяснить почему и как проверить)
```

## Резюме

Правильный коммит:
1. ✅ Следует формату `<type>(<scope>): <summary>`
2. ✅ Использует фиксированный список типов
3. ✅ Имеет понятный и стабильный scope
4. ✅ Содержит конкретное описание в summary (до 72 символов)
5. ✅ При необходимости содержит body с объяснением
6. ✅ Ссылается на задачи через footer
7. ✅ Помечает breaking changes явно

**Помните:** Хороший коммит — это сообщение будущему себе и коллегам о том, что и почему было изменено.
