# assistant

## Назначение
Проект `k/assistant` для Chatium. Базовая структура: главная страница (карточки «Мой журнал», «Мои задачи», «Диалоги», «Инструменты», «PARA», «Финансы», «Сервисы», «Библиотека»), страница «Мой журнал» (`/web/journal` — упрощённый блокнот/инбокс с текстом и архивом; вкладка «Задачи» с задачами «В работе» из Heap, порядок дня и сброс в очередь; вкладка «День» с 4 полями дневной фиксации: Ночь/Утро/День/Вечер, фиксация/разблокировка по кнопке и хранение по ключу дня с границей в 05:00; вкладка «Неделя» с двумя уровнями планирования: отдельное поле `Неделя #N` для общего плана недели и дневные поля Пн–Вс с датами, навигацией на прошлую/следующую неделю и фиксацией/разблокировкой), страница «Управление задачами» (`/web/tasks` — три колонки на десктопе: дерево клиентов/проектов, таблица задач, чат с AI на фиде; на мобильном чат из FAB в шторке; клиенты → проекты → задачи, приоритеты, статусы, быстрый клик по статусу открывает меню смены статуса, режимы «Активные/Архив», а задачи со статусами «Завершено» и «Отменено» автоматически уходят в архив; плюс привязка задачи к Pomodoro), страница «Инструменты» (`/web/tools`) и страница Pomodoro (`/web/timers`), админка, профиль, логин и тестовая страница.

Подробнее о задачах и журнале: `docs/architecture.md`, `docs/api.md`, `docs/data.md`.

## Важно
- Платформа: Chatium. Серверная часть управляется платформой.
- Стек фиксирован платформой, новые зависимости не добавляем.
- Деплой происходит автоматически после пуша.

- Главная, админка, профиль и логин существуют как минимальные страницы.
- На главной 8 карточек быстрых разделов: «Мой журнал», «Мои задачи» и «Инструменты» открывают разделы, «Диалоги», «PARA», «Финансы», «Сервисы», «Библиотека» работают как action-кнопки с глобальным glitch-эффектом без перехода.
- Раздел `Инструменты` содержит одну карточку `Таймеры`, ведущую на страницу `Pomodoro`: разметка вкладок и инструментов собрана в `components/pomodoro/PomodoroToolsWorkspace.vue` (общий CRT-стиль). Для `Таймера` и `Секундомера` доступны: опциональная привязка задачи, модалка настроек, чекбокс `Фокус` (при включении сегменты пишутся в журнал рабочего времени, при выключении работают как обычные инструменты). Селектор задачи унифицирован между вкладками, поддерживает поиск по первым символам и фильтрацию по названию задачи/проекта/клиента, а в списке показывает иерархию `клиент / проект`. Нижняя строка из трёх счётчиков (`Сессий`, `Работа`, `Всего`) **одинакова на всех вкладках** (агрегаты Pomodoro + таймер + секундомер); часть данных таймера/секундомера хранится в `localStorage` по режимам и дневному ключу 05:00.
- Для Pomodoro ведётся отдельный журнал сегментов запусков (`tables/pomodoro-launches.table.ts`): фиксируются старт/продолжение, фаза, задача, длительность сегмента и причина завершения (пауза, стоп, автопереход фазы, смена задачи и т.д.) для последующей аналитики периодов работы.
- Реализован пользовательский Pomodoro в CRT-стиле проекта: отдельная страница с полным CRT-оформлением (виньетка, сканлайны, бэзел, сетка, crt-power-on анимация), пиксельными углами (`clip-path`) на всех элементах, SVG-циферблат с tick-марками и glow-эффектами, phase bar с индикатором цикла, панель действий под таймером (сетки `--n1` / `--n2`: «Старт»; «Пауза»+«Пропустить»; «Продолжить»+«Сбросить» через API `reset`); модальное окно настроек и селектор задач также в CRT-стиле; звук при смене этапа — пять пресетов (Web Audio API, настройка и прослушивание в модалке, хранение `phaseChangeSound` в Heap); браузерные Notifications, вибрация, обновление `document.title` с таймером.
- В `Header.vue` для страницы `/web/timers` добавлен интерактивный виджет в `header-clock`: первый клик по часам открывает выбор инструмента (`Помидор`, `Таймер`, `Секундомер`), после выбора доступны кнопки `Запуск / Пауза / Сброс`; для `Помидора` управление идёт через текущий API `control`, для `Таймера` длительность старта берётся из актуальных настроек вкладки `Таймер` (localStorage-ключ `assistant:focus-clock-settings:timer`), которые меняются только на странице таймеров. При клике `В помидор` в журнале (`JournalDayPane`) задача теперь отправляется в выбранный инструмент шапки через событие `assistant:focus-task-selected`: если выбран `Помидор/Таймер/Секундомер` и инструмент не запущен, он стартует (или продолжается после паузы), если уже запущен — только переключается задача; если в шапке выбран режим обычных часов, блок `header-clock` трижды мигает красным.
- Реализованы: API настроек (list, get, save), Heap-таблица settings, репозиторий, lib (бизнес-логика).
- Серверные логи: Heap-таблица logs (message, payload, severity, level, timestamp), repos/logs.repo (create, findAll, findById, findBeforeTimestamp, countBySeverityAfter, countErrorsAfter, countWarningsAfter), lib/logger.lib (проверка уровня по настройке log_level, запись в ctx.log — только сообщение, в ctx.account.log — сообщение и payload, Heap, WebSocket с хэшем для уникальности канала, вебхук log_webhook { enable, url } по умолчанию url: ""). API POST /api/logger/log (AnyUser), body: { severity, level, message, payload?, timestamp? }; GET /api/admin/logs/recent (Admin) — последние N логов; GET /api/admin/logs/before (Admin) — N логов старше указанного timestamp для пагинации. Админка получает encodedLogsSocketId, подписывается на new-log для отображения в дашборде, загружает историю логов через recent при монтировании, может догружать старые логи через before (кнопка «Загрузить ещё 50»); кнопка «Очистить логи» очищает вывод и сдвигает таймштамп на текущий — повторное нажатие «Загрузить ещё 50» восстанавливает последние логи.
- Дашборд админки: счётчики ошибок и предупреждений. Настройка `dashboard_reset_at` (таймштамп сброса в ms); lib/admin/dashboard.lib (getDashboardCounts, resetDashboard), GET /api/admin/dashboard/counts и POST /api/admin/dashboard/reset (Admin). При монтировании загружаются счётчики; кнопка «Сбросить» записывает текущее время в настройки; при новых логах (sink/WebSocket) инкремент только если timestamp >= dashboardResetAt.
- API тестов: каталог api/tests/ (GET /api/tests/list — список категорий и тестов, AnyUser); категория api/tests/endpoints-check/ с эндпоинтами для проверки слоёв: health, ping, config, settings-lib, settings-repo, logger-lib, logs-repo, dashboard-lib. Каждый возвращает массив results (по одной проверке на функцию). Порядок операций: создание до чтения (create/upsert перед read). Страница тестов: шесть блоков — «Проверка эндпоинтов» (маршруты /, /web/admin, …), «Библиотека настроек», «Репозиторий настроек», «Библиотека логов», «Репозиторий логов», «Библиотека админки»; дашборд метрик и «Запустить все тесты».
- При серверной загрузке главной, админки и профиля уровень логирования читается из настроек и передаётся на клиент в `window.__BOOT__.logLevel` (shared/logLevel.ts). В браузере доступен shared/logger по стандарту syslog (RFC 5424): уровни -1 (логи выключены, LOG_LEVEL_OFF), 0–7 (Emergency…Debug), функции `logEmergency`, `logAlert`, `logCritical`, `logError`, `logWarning`, `logNotice`, `logInfo`, `logDebug`; `createComponentLogger(name)` для логов с префиксом; `setLogSink`, `LogEntry` для дашборда. Вывод только если severity не строже настроенного порога.
- Клиентская часть полностью покрыта логами: страницы и компоненты используют `createComponentLogger`; AdminPage регистрирует sink и подписку на WebSocket для отображения логов в дашборде в реальном времени.
